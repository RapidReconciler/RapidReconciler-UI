Sales

Owns the prospect-to-contract motion. Triggers every customer-facing handoff up to but not including the install itself.

Responsibilities
  • Prospect identification — first-touch outreach and qualification (Sales Step 1).
  • PoC decision & agreement — raises a PoC only when necessary; sends the preliminary PoC agreement covering scope, billable GSI time, and duration (Sales Steps 2, 2A).
  • Provisioning document — sends the customer-facing technical-requirements pack so Customer I/T, security, and JDE teams can review (Sales Step 3).
  • Purchase agreement & contract — runs the commercial decision and generates/sends the signed contract (Sales Steps 4, 5).
  • Contract as source of truth — the signed contract is the authoritative source for the values the tech team executes against in VALC: number of companies purchased (drives Maximum Companies Allowed), exact 5-digit JDE company numbers to license, and license start/end dates. Renewals also run through Sales before tech updates the License End Date.
  • Discovery call — arranges the joint kickoff between Sales, Customer I/T, and RR DBA (Sales Step 6).
  • Installation prep handoff — sends the install-prep document to the customer after the discovery call (Sales Step 7).
  • Sets customer expectation on install meeting — the technical install is a 2-hour live web meeting with the customer's I/T contact available the full duration (not just the start). Plus a separate VALC working session for the agent install. Surfacing this during the commercial conversation prevents scheduling friction at handoff.
  • Collateral usage — runs the self-guided tour, overview, and fact sheet through the prospect funnel.
Required skills
  • RapidReconciler product knowledge — the elevator pitch, key features, business outcomes.
  • Working JD Edwards literacy — inventory reconciliation, in-transit, RNV, cardex, period-end close concepts.
  • Commercial & contract handling — PoC scoping, purchase agreements, signed-contract workflow.
  • Customer communication — running discovery calls and managing technical handoffs without doing the technical work.
  • Familiarity with the customer-facing docs (Provisioning, Install Prep, PoC, Discovery Call) so they can answer first-line questions.

RR DBA

The technical lead for the database and SSIS install end-to-end — from intake through the completion email that triggers VALC.

Responsibilities
  • Discovery-call attendance — joins the kickoff as the technical participant from GSI (Sales Step 6).
  • Intake — checks the rrsupport@getgsi.com inbox for the customer's pre-install submission email; if a PoC was previously run, rebuilds the production install from the PoC environment rather than starting clean (Tech Step 1 / DB Install Step 1).
  • Submission gate-check — before scheduling, verifies the submission is complete: 30 of 30 checklist items, topology selected, server names & IPs populated, JDE connection details with the trailing period on the table qualifier. Replies asking for re-submission rather than scheduling on a partial.
  • Schedules the 2-hour install meeting — pulls the contact from the SUBMITTED BY block, sends a web-meeting invite with the standard 9-item agenda, and confirms the customer's I/T contact is available for the full 2 hours (DB Install Step 2).
  • Pre-meeting architecture review — matches the customer's selected topology (Configuration 1, 2, or 3) against their actual environment, verifies hardware/RAM/disk-space minimums, and confirms the correct OLE DB provider is staged on the SSIS host in the right architecture (32-bit and/or 64-bit per JDE source). Driver mismatch is the most common cause of SSIS deployment failures (DB Install Step 4).
  • On-screen install zip download — at the start of the install meeting, downloads a fresh copy of the install zip directly to the customer's database server via screen-share and unzips it. Eliminates the most common variant of "missing files" tickets (DB Install Step 5).
  • Create SSISDB catalog — sets up the Integration Services Catalog and the RapidReconciler folder on the target SQL Server (DB Install Step 6).
  • Install RapidReconciler database — runs the four installation scripts to create the database, schema objects, the rruser SQL login, and the RapidReconciler_Prod SQL Agent job (DB Install Step 7).
  • Configure & deploy SSIS package — creates the Visual Studio solution, configures the JDE source and RR destination connection managers, sets the SSIS variables (aaStartDateGr, dbowner, DecExtCost, DecUnitCost, DecQTY, DecQtyCX) using values from the submission, deploys to the SSIS catalog, and points the SQL Agent job at the deployed package (DB Install Step 8).
  • Initial data load — enables the SQL Agent job's schedule against an off-peak run time agreed with Customer I/T (DB Install Step 9).
  • Configure security — verifies rruser permissions (db_datareader, db_datawriter, SQLAgentOperatorRole on msdb), SQL Agent service-account access, and SSIS catalog roles (ssis_admin, ssis_logreader, dc_operator) (DB Install Step 10).
  • Wait on customer load verification — does not proceed to Step 12 until Customer I/T emails confirming the first scheduled run completed successfully. The VALC tech needs the database online and populated, so notifying early creates re-work (DB Install Step 11 gate).
  • Completion email to RR Support — sends the database-ready notification with subject "RapidReconciler Database Ready — {Customer Name} — VALC Install Can Begin". Body includes customer name, SQL port (only if non-default 1433), rruser password (only if non-default), confirmation the load was verified, and a topology-filtered pre-flight checklist. Triggers RR Network Tech to start the VALC install (DB Install Step 12 / Tech Step 4).
  • PoC preliminary install (when applicable) — does the install work on the customer's servers during the PoC engagement, jointly with the RR Network Tech, against billable GSI time (Sales Step 2B).
Required skills
  • SQL Server (2017+, Standard edition or higher) — SSMS, SSISDB catalog creation, database/object install scripts, SQL Agent job configuration, login & user mapping.
  • SSIS — Visual Studio Community with the Integration Services extension; package configuration, connection managers, variables, deployment to the SSIS catalog, wiring SQL Agent jobs to deployed packages.
  • JD Edwards source connectivity — Oracle, AS/400 / IBM i, or SQL Server source platforms; OLE DB and iSeries Access (Client Access) drivers in both 32-bit and 64-bit where required.
  • JDE data dictionary fundamentals — ECST, UNCS, PQOH, TRQT decimals, proddta. table qualifier (with trailing period — the most common SSIS misconfiguration).
  • Mixed-mode SQL authentication and the rruser service-account pattern; sysadmin or ssis_admin privileges for catalog deployment.
  • SQL Server Agent — job creation, step configuration, schedule management, proxy account configuration where required.
  • Submission email format — ability to read the pre-install submission, gate-check it for completeness, and map each section (TOPOLOGY, SERVER NAMES & IPs, JDE connection details, decimal handling) to the corresponding install step.
  • Verification view fluency — able to direct the customer to the diagnostic view post-load and triage failed loads if the customer reports back errors.
  • Working knowledge of the Installing the Production Database procedure end-to-end and the install-prep checklist Customer I/T submits.
  • Customer-facing meeting skills — running a 2-hour web meeting with screen-share, narrating SSMS/Visual Studio actions for an I/T audience, knowing when to pause and reconcile if the submitted topology doesn't match the actual environment.

RR Network Tech

Picks up after the database is configured. Owns the entire VALC install (Steps 2–13), the GSI network/Cloudflare work that supports it, and most live-customer change requests post-go-live. Also participates in PoC preliminary installs alongside the RR DBA when the optional PoC path is taken (Sales Step 2B).

Responsibilities
  • PoC preliminary install (when applicable) — participates jointly with the RR DBA in the install work on the customer's servers during the PoC engagement, against billable GSI time. Only happens when the optional PoC path is taken; the default path skips this (Sales Step 2B).
  • Receives database-ready notification — the second trigger email from RR DBA at rrsupport@getgsi.com with subject "RapidReconciler Database Ready — {Customer Name} — VALC Install Can Begin". Pauses and checks back with the DBA if it hasn't arrived (VALC Step 1).
  • Create VALC client record — on the Clients page, sets Client Name, License Start/End dates, Version, JDE Version, HTTPS = True, Maximum Companies Allowed (from sales contract), Agent Protocol = SSL (VALC Step 2).
  • Create initial GSI Admin user — on the User Accounts tab, creates a placeholder user (Full Name "GSI Admin", email gsiadmin@{customerdomain}, temporary password) used during agent install and later handed off to the customer's RR administrator (VALC Step 3).
  • Configure modules — on Client Details, fills Agent External/Internal IPs, Domain URL (rrprod-{customername}.getgsi.com), and enables Inventory and Admin tabs to start (VALC Step 4).
  • DNS A-record & certificate request — before submitting, confirms both the domain URL and internal IP from the Client Details page in VALC. Submits the A-record addition to GSI I/T (typically Daren) for the customer's domain pointing to the agent internal IP, ensuring the application UI uses the wildcard *.getgsi.com certificate. Allows DNS propagation time (a few minutes to several hours; escalate to GSI I/T if not resolved within 24 hours) before agent install (VALC Step 5).
  • Joint VALC agent install — on a working session with Customer I/T, walks the customer through downloading the 64-bit agent from https://rapidreconciler.getgsi.com, running the installer with local administrator credentials on the application server, and confirming Installation Complete in the browser (VALC Step 8).
  • Validate SQL Server connection — provides the agent with the database server's internal IP, port (typically 1433), and rruser credentials when the connection prompt appears mid-install. Confirms the row in VALC's Clients grid updates to Connected with Services and Agent versions populated (VALC Step 9).
  • Register database in VALC — on the Databases tab, creates one record per registered database (Address, Port, Username, Password, Name, Category, Job Name RapidReconciler_Prod, Max Memory). Repeats for each Test/Dev/Train database, which the DBA has prepared via backup-restore from Prod (VALC Step 10).
  • Verify databases Online — confirms every registered database shows Online (green) before proceeding. Investigates Unknown or Stopped states using agent log files at C:\Program Files\Rapid Reconciler\logs\ (VALC Step 11).
  • License company numbers — on the Companies tab, checks the boxes for each 5-digit JDE company number per the sales contract. Repeats for each registered database (VALC Step 12).
  • Customer handoff email — sends the customer's I/T contact and designated RapidReconciler administrator a handoff email containing login URL, initial GSI Admin credentials (with first-login password reset), pointer to user management on the User Accounts tab, password-policy status, and the support contact (VALC Step 13).
  • Confirm first customer login — verifies the customer has logged in at least once before considering the install closed.
  • License renewals — updates License End Date in VALC after Sales confirms the renewal contract.
  • Live-customer changes — adds additional company licenses, enables additional module tabs, updates database connection details (IP, port, credentials), pushes Services/Agent updates via Manage Deploys, deactivates clients on cancellation (Client Active toggle).
  • Password policy toggle — enables or disables the per-customer complex password policy on the Client Details tab in response to Customer I/T's email request to rrsupport@getgsi.com.
  • Adding a Test/Dev/Train database — coordinates with the customer's DBA to back up RapidReconciler_Prod and restore it under the new name on their SQL Server, then registers the new database in VALC (does not re-run the creation scripts — they're hard-coded for Prod).
  • Posting Message Users broadcasts — uses the Message Users page to post system-wide notices for maintenance windows, deploys, or version changes. Messages are seen by all customers, so wording stays customer-neutral.
  • SSL & DNS management — wildcard *.getgsi.com certificate handling, DNS A-record changes (Cloudflare), and Steps 2–3 of the certificate renewal procedure: agent push via VALC Manage Deploys after backend is updated, then per-customer browser verification confirming the renewed cert is valid, non-expired, with the correct issuer and expiry date. Step 1 (backend cert config) belongs to RR UI Developer.
  • Cert renewal lead time — start coordination with the RR UI Developer at least 2–3 weeks before cert expiry. Renewal is sequenced per customer (the verification step has to happen for each), not a single batch. An expired cert affects every customer simultaneously.
  • IP-change handling — when a customer's internal IP changes, update both the VALC Client Details page AND submit a DNS update to GSI I/T. Updating only one creates a broken state where the UI loads but the agent can't connect.
  • Cert / DNS troubleshooting — four diagnostic patterns: browser shows "Not Secure" or expired cert (redeploy via Manage Deploys); domain URL doesn't resolve (verify DNS A-record with GSI I/T); UI loads but agent disconnected (IP-change mismatch between VALC and DNS); propagation taking too long (>24h escalates to GSI I/T).
  • Server migration (existing customers) — moves clients to new infrastructure across dedicated and separate-server topologies. Includes backup/restore, SSIS package re-deployment, SQL Agent job re-creation, agent re-install, VALC update, Cloudflare DNS cutover, post-migration verification.
  • First-line troubleshooting — uses agent log files (rr-valc-agent_YYYYMMDD.out and .err) to diagnose Disconnected clients, Unknown/Stopped database rows, agent installs hanging at Validating Data, Setup Step not advancing, post-Cloudflare data-not-flowing issues. Escalates to RR DBA when failures point at the database or SSIS side.
Required skills
  • VALC fluency end-to-end — client record creation, user accounts, module/tab configuration, database registration, company licensing, status interpretation (Connected/Disconnected, Online/Unknown/Stopped, Setup Step values).
  • Cloudflare — DNS A-record management, propagation behavior, cutover sequencing during migrations.
  • SSL/TLS — the wildcard *.getgsi.com certificate model, browser warnings, the three-step renewal procedure (knowing Step 1 is RR UI Developer's, Steps 2–3 are yours), the four diagnostic patterns for cert/DNS issues, lead-time discipline for sequenced multi-customer rollouts.
  • Agent installation & validation — running the installer on the customer's application server with local admin credentials, providing SQL connection details to the agent, confirming Connected status in VALC, manage-deploys for ongoing updates.
  • Network access requirements — outbound 443 to rapidreconciler.getgsi.com, rr-spa.cloudapp.net, rr-valc-spa.cloudapp.net; internal port range 32145–49152; ability to verify connectivity by browsing to the login page from the application server.
  • Agent log file diagnostics — reading .out and .err files at C:\Program Files\Rapid Reconciler\logs\; recognising connectivity errors and stack traces; asking the customer to send today's log when triaging remotely.
  • Migration mechanics — backup/restore, SSIS package and SQL Agent re-scheduling on new infrastructure, Setup Step reset back to Created before re-running agent install.
  • Password policy administration — the rules (8-char minimum, 3-of-4 categories, 10-password history, 90-day expiry), the activation request workflow, and the per-customer toggle.
  • Familiarity with the Troubleshooting reference (the six diagnostic patterns + agent log files) and Common Requests (ownership matrix for live-customer asks).
  • Comfort directing customer requests back to the customer's RR administrator when they're user-management requests rather than GSI tasks.
  • Discretion when posting Message Users broadcasts — messages are global; wording must be customer-neutral and free of internal jargon.

RR DB Developer

RapidReconciler product developer. Owns schema and code changes pushed across the customer base via the VALC SQL Scripts page. Not involved in individual customer installs.

Responsibilities
  • Authors update scripts — SQL update scripts (schema changes, data fixes, patches) intended to be applied across customer databases.
  • Maintains the install zip — updates the database object script (versioned, e.g. ...178.sql), packaged installation files, and the SSIS package distributed via files.getgsi.com. RR DBAs always download a fresh copy at the start of each install.
  • Applies update scripts via VALC SQL Scripts page — selects target databases (filtered by Category and Revision Number), reviews script contents in the editor, executes against checked databases (5 at a time recommended), and reviews pass/fail results.
  • Test before Production — runs scripts against Test databases first when available, then promotes to Production after verification.
  • Failure investigation — if a script fails on one or more databases, investigates root cause before re-running. Re-executing a partially-applied script can compound the problem.
  • Supports DBAs and Network Techs — escalation path when failed installs or migrations point at a schema or product-code issue rather than a configuration one.
Required skills
  • RapidReconciler database schema (current and prior versions; revision-number versioning).
  • SQL scripting against multiple SQL Server versions (2017+ baseline).
  • SSIS package authoring, including the RapidReconciler-Prod.dtsx connection managers and variables (aaStartDateGr, dbowner, decimal places).
  • VALC SQL Scripts page mechanics — database list filtering, batch execution discipline (5 at a time), reading the Results panel.
  • Database categorisation in VALC (Production, Test, Archive, Change Control, Inactive) and how Category drives the filter.
  • Change-control hygiene — reviewing script contents before execution, verifying source even when trusted, rollback strategy for partial failures.
  • Coordination with RR DBA and RR Network Tech when a deploy fails on customer environments.

RR UI Developer

Owns the backend configuration for the RapidReconciler web interface. Surfaces in the workflow during certificate renewal: Step 1 of the three-step renewal procedure (backend update) belongs to this role.

Responsibilities
  • Backend cert configuration update — Step 1 of the three-step certificate renewal procedure. After GSI procures the renewed wildcard *.getgsi.com cert, updates the backend configuration so the RapidReconciler web interface serves the renewed cert.
  • Confirms web interface serves the renewed cert correctly — before signaling RR Network Tech to begin the agent rollout (Step 2 of renewal). This is the gate between Step 1 and Step 2: agent push shouldn't start until the backend is verified.
  • Coordinates lead time — works with RR Network Tech 2–3 weeks before cert expiry to sequence Step 1 (backend) ahead of Steps 2–3 (per-customer rollout). Cert renewal that runs short of time is one of the few failure modes where every customer is affected simultaneously.
Required skills
  • Backend / web-server configuration for the RapidReconciler UI — updating cert binding, restart procedures, post-deploy verification.
  • SSL/TLS certificate handling — importing renewed certs, configuring the wildcard *.getgsi.com binding, validating chain of trust.
  • Coordination with RR Network Tech — clear handoff signal once Step 1 is verified, so the agent rollout in Step 2 starts against a known-good backend.
  • Awareness of the cascade effect — an expired wildcard cert affects every customer at once; the role’s lead time discipline is what prevents this.
Where they appear

Cert renewal Step 1 Outside the install flow

Hand-offs in
  • From GSI: the renewed wildcard *.getgsi.com certificate, ahead of expiry.
  • From RR Network Tech: coordination on lead time — renewal kickoff 2–3 weeks before cert expiry.
Hand-offs out
  • To RR Network Tech: confirmation that the backend is updated and the web interface serves the renewed cert correctly. This signal triggers Steps 2–3 (per-customer agent rollout via Manage Deploys, then browser verification).
Documents owned

Helpdesk Tech

Monitors the rrsupport@getgsi.com mailbox and performs Tier 1 support. Triages and resolves first-line issues, and routes inbound work to the right tech-team member at the right time.

Responsibilities
  • Mailbox monitoring — watches the rrsupport@getgsi.com inbox throughout the day so nothing sits unattended between Customer I/T sending and the right tech picking up.
  • Tier 1 support — first-line response on inbound support requests: triages the issue, gathers details from the customer, resolves what can be resolved at Tier 1, and escalates to the appropriate tech (DBA, Network Tech, or Developer) when the issue is out of scope.
  • Intake routing — receives the install-prep email from Customer I/T (Sales Step 8) and forwards it to RR DBA to kick off the technical install (Tech Step 1).
  • Completion routing — receives the DB/SSIS completion email from RR DBA (Tech Step 4) and routes the work to RR Network Tech for VALC.
  • Documentation feedback — collects corrections, new procedures, and field notes from real installs across both the sales and tech docs.
Required skills
  • Tier 1 troubleshooting — reproducing customer-reported issues, gathering logs and screenshots, working through standard fixes before escalating.
  • Triage & routing — recognising whether an inbound email is an intake, a completion, a Tier 1-resolvable issue, or an escalation, and getting it to the right destination.
  • Working knowledge of where each team sits in the workflow (Sales Step 8, Tech Step 1, Tech Step 4) so handoffs land with the right person.
  • Customer communication — clear, prompt replies that set expectations on what's been resolved vs. escalated.
  • Ticketing / mailbox hygiene — ensuring nothing in the queue gets dropped or duplicated.
