Side-by-side reference for what changed, what was preserved, and what gaps remain between the legacy v359 Services jar and the green-field rebuild that V8 talks to today.

Overview

This doc answers the four questions the team keeps asking when the conversation turns to the green-field data-services agent:

  1. What changed at the Agent level vs the Data Services level?
  2. Were any existing endpoints changed in a breaking way?
  3. Which v359 endpoints are absorbed by the new agent today, which are net-new, and which are still missing?
  4. Will the new agent support existing v359 customers?

Pair this doc with:

  • Deploy architecture — the three-component install model + the Actuator contract a new Services jar plugs into. (RapidReconciler-Agent/docs/deploy-architecture.md)
  • API.md — full v359 controller catalog (the “before” picture). (RapidReconciler-Agent/docs/API.md)
  • Gotchas — field-name traps + the two ValidationLight sources that have the same shape and different semantics. (RapidReconciler-Agent/docs/gotchas.md)
TL;DR
  • Agent layer is unchanged. The new code is a from-scratch rebuild of the Services jar only — the per-DB HTTP process that V8 talks to. The rr-valc-agent.jar JMS broker client and the central VALC SPA stay where they are.
  • No existing endpoint has been changed in a breaking way. Three endpoints have shipped additive changes (/inventory/as-of, /inventory/audit-detail, /poll); each is backwards compatible with v359 callers.
  • V8’s entire boot path is absorbed. The dev box can shut v359 entirely. The legacy AngularJS SPA still hits v359 on its own host because it exercises endpoints (In Transit, PO Receipts, Roll Forward, Admin) that the new agent doesn’t expose yet.
  • Customer compatibility is gated on closing that gap. Today the new agent is V8-shaped, not v359-feature-complete. Production-blessed release + legacy-SPA smoke test are both still ahead.

1. Agent level vs Data Services level

Every RapidReconciler install has three independently-versioned components (see deploy-architecture.md):

ComponentWhat it isWhere it runsChanged?
Valc Central SPA + server. staging-valcspa.cloudapp.net / rr-valc-spa.cloudapp.net No. RapidReconciler-Valc is a parallel dev-side mini-broker + dashboard, not a replacement.
Agent rr-valc-agent.jar — the JMS message broker client. Owns file streaming, SHA1 verification, Services instance start/stop, self-update, Derby DB at db\. Each customer host, under the WinSW wrapper. No. Source isn’t in this repo; deploy machinery is preserved.
Services Per-DB HTTP jar at files\<n>\ — the one V8 calls. Spawned by Agent, one process per DB. Yes. This is the entire green-field rewrite. New package coral.rapidreconciler.client.services.* in the agent repo’s src/.

The new Services jar slots into the existing Agent the same way v359’s did: expose the three Actuator endpoints (/health, /info, /shutdown) per specs/actuator-deploy-integration.md; the Agent’s HealthService polls them after a JMS DEPLOY message lands and reports success/failure back through the queue.

The deploy flow at the Agent level (per deploy-architecture.md § Deploy flow) runs unchanged. A customer flipping Services version from 1.6.22 to the new build in VALC’s Manage Deploy panel produces the same JMS DEPLOY → file stream → start instance → verify /health + /info sequence; the Agent doesn’t care that the new jar comes from a different source repo.

2. Endpoint changes — additive only

The migration policy (feedback_v8_test_agent_default) is: preserve v359’s wire contract so V8 can flip a routing-table entry per area without changing client code. Concretely:

  • Request DTO field names match v359’s Jackson bindings (camelCase first-letter-lowercase; type not docType on transactions/details; daily not period on as-of).
  • Response field names match v359’s view-aliased column shapes.
  • The reconciliationFilter two-shape split (Item-wrapped on /reconciliation-filtered, bare-string arrays everywhere else) is preserved per API.md § Reconciliation filter shapes.

Three additive changes have shipped on existing endpoints, all backwards-compatible:

EndpointPRChangeCompatibility
POST /inventory/as-of Agent#23 New optional columns: List<String> body field for wire-level column projection. Omit it → full row. Old clients ignore the field; Spring binds it as null.
POST /inventory/audit-detail Agent#31 Response gains five additional arrays (accountSummary, accountDescriptions, unpostedBatchesUi, unpostedCardexUi, manualJournalEntries) alongside the existing reconcilingItems + perpetual. Jackson skips unknown response fields on the client; old clients see no change.
GET /poll Agent#32 Optional query params ?updating=&recalculating=&timeoutMs= enable v359-style long-poll. No params → classic short-poll (unchanged). Mini-VALC’s /health probe and any other no-param caller hit the unchanged short-poll path.

No existing endpoint has been renamed, removed, or had its DTO shape narrowed. The endpoint name typo on POST /inventory/rollIItem (double-I) is preserved verbatim — matches the bytecode.

4. Will the new agent support existing v359 customers?

Mechanically, yes. The deploy story is unchanged: a customer can flip the Services version in VALC’s Manage Deploy panel, the existing Agent jar pulls the new jar via JMS, starts it, and verifies it via the Actuator endpoints. No customer-side configuration change required.

Functionally, not yet. Three caveats block a real customer cutover today:

  1. Endpoint coverage is V8-shaped. The table above lists ~15 endpoint groups the new agent doesn’t expose. A customer whose deployed AngularJS SPA exercises In Transit, PO Receipts, Roll Forward, the daily-probe GET, PDF downloads, or any Admin module screen would see those screens break. The legacy SPA hasn’t been smoke-tested against the new agent — it currently runs against v359 on its own host (staging-rr-spa.azurewebsites.net / rapidreconciler.getgsi.com).
  2. No production release has been cut. The new code currently runs only on the dev box (localhost:34537 / rrtest-rrsqltest.getgsi.com:34537). The repo’s main branch hasn’t been tagged v360 or otherwise blessed for deploy. The artifact in RapidReconciler-Agent/artifacts/v359/ is still the production reference.
  3. SQL compatibility floor still applies. Customers may sit at SQL Server compat level 100 (per feedback_sql_compat_floor and RRV8/WORKFLOW.md § SQL targets the lowest customer compat level). Every SQL the new agent issues must clear that bar. Current code does; future changes have to keep clearing it. The dev DB sits at compat 100 to catch regressions early.

4a. Cutover preconditions

In rough order, a customer-ready cutover would need:

  1. Close the legacy-SPA coverage gap above (or sunset the legacy SPA in favor of V8 for the affected modules).
  2. Tag a release on main and pin the corresponding artifacts/v360/ directory.
  3. Validate the actuator + JMS plumbing against a real Agent install (the actuator-deploy-integration.md spec, exercised end-to-end).
  4. Pilot on one internal install / one consenting customer with a rollback plan to v359.

None of these are blocked today — they’re just queued.

5. Agent updates and clean installs

No — agent-layer changes don’t require a clean install. In-place updates via JMS are the whole point of the architecture.

The mechanism is SelfUpdateMessage in rr-common:

FieldTypePurpose
lastAgentVersionStringThe target version (e.g. 1.6.21) — lands in client_deploys.target_version_string and surfaces in the dashboard’s Agent Version column.
lastAgentVersionIdlongfile_versions.id — the artifact catalog row VALC will stream from.
lastAgentVersionSha1hexString40-char SHA1; the agent verifies the streamed bytes against this before swapping the jar.

Deploy flow for an agent update:

  1. Admin uploads a new rr-valc-agent.jar into VALC → row in file_versions with component = 'agent'.
  2. VALC publishes SelfUpdateMessage to the agent’s receiver queue.
  3. Agent requests the bytes (SendFileMessage); VALC streams them (FileContentMessage).
  4. Agent verifies SHA1 against lastAgentVersionSha1hex.
  5. Agent swaps the jar on disk and exits. WinSW respawns it automatically.
  6. New agent jar reconnects via AgentConnection; subsequent dashboards reflect the new version through client_deploys.target_version_string.

This is the same flow Services jars use for redeploy — the only difference is component = 'agent' on the file_versions row vs component = 'services'.

The dashboard’s “Agent Version” column doesn’t ask the agent its version. Agents don’t self-report. VALC remembers what it pushed via client_deploys.target_version_string, and that’s what surfaces.

5a. When a clean install IS required

Exceptions, not the norm. Anything in the jar itself flows through SelfUpdateMessage; anything outside the jar needs manual intervention.

TriggerWhy self-update can’t cover itMitigation
JMS wire-format break If a change to rr-common removes a field the running agent reads, the old agent can’t deserialize VALC’s messages after VALC updates — including the SelfUpdateMessage that would have fixed it. SelfUpdateMessage itself is small and stable, but other messages exchanged with a not-yet-updated agent could break. Roll rr-common changes additively (new fields default to null on old readers; never remove or rename). Same policy V8 follows for endpoint shapes — see § 2.
WinSW wrapper config change If the .xml next to the service registration changes (JVM args, working dir, service name, log paths), self-update only swaps the jar — not the wrapper. Redeploy the wrapper and re-register the Windows service. Document the change in the deploy notes.
JDK floor change If the new agent jar requires a newer JDK than what’s on the customer host, the JVM rejects the bytecode before the new jar gets a chance to run. Install the new JDK on the customer host first, then push the jar.
First-time install on a new host No agent yet — nothing to self-update. Clean install via the installer.
Service registration corruption Edge case: the Windows service entry is broken (manual sc delete elsewhere, antivirus interference, etc.). sc commands to repair, or a reinstall to restore.

The rr-common JMS contract is the boundary to stay backward-compatible across. Treat it like a public API: add fields, don’t remove them.
