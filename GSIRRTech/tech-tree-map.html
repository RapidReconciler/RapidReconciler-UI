Patterns & observations

  • The tech tree is shallow. Hub → tech-client-management → procedure or start-here. Two clicks gets you anywhere. The deepest path is hub → workflow → procedure → "see also" sibling procedure (4 clicks).
  • Procedure pages are terminal. They back-link to the parent workflow, sometimes cross-link to a sibling procedure, and stop. They never spawn new pages.
  • The Help Desk is the densest cross-linking page. It deep-links into specific anchors on every procedure (24+ anchor refs across 3 procedure pages). It's not a hub — it's a symptom-to-procedure index.
  • Start-here pages vary in scope. The DBA's is narrow (just the DB install). The Network Tech's is broad (every tech procedure plus an unseen password-policy.html). Developer and Helpdesk Tech sit in the middle.
  • Two known orphans. password-policy.html is referenced only by start-here-network-tech. roles-responsibilities.html (in ../MiscDocs/) is referenced only by two start-here pages. Neither has been provided yet, so their own outbound links are unknown.
  • One tech→sales bridge exists. start-here-monitor links to ../GSIRRSales/rr-installation-prep.html. No other tech page references the sales side; the rest of the cross-folder traffic flows hub-to-folder, not folder-to-folder.
  • Folder casing was inconsistent and is now standardized to MiscDocs. The audit fixed two lowercase references in troubleshooting.html.
  • The breadcrumbs originally encoded a richer 5-section taxonomy (Installations / Migrations / Client Onboarding / Troubleshooting / Reference) that no longer exists on the parent. The audit collapsed all breadcrumbs to #procedures; if you want the old taxonomy back, that's a content rewrite, not a link fix.
