Orientation

What the role is — and what it isn’t

You own the database side of every customer install end-to-end. From the moment the customer’s pre-install email lands in rrsupport@getgsi.com through the completion email that triggers VALC, the install is yours. Customer I/T is on-session with you for the meeting; everyone else — Sales, Network Tech, Developer, Helpdesk Tech — is upstream or downstream of your work, not in it with you.

What you own

  • The 12-step install procedure documented in Installing the Production Database. Every install runs this sequence.
  • The 2-hour install meeting with the customer’s I/T contact. You drive; they provide access and credentials.
  • The two trigger emails: receiving the customer’s pre-install submission, and sending the database-ready notification that hands the install over to RR Network Tech for VALC.
  • Submission gate-checking. If a submission isn’t complete, you bounce it back — you don’t schedule against an incomplete checklist.

What you don’t own

  • The customer’s environment — their server provisioning, firewall, antivirus, JDE access. That’s Customer I/T’s pre-install work; you operate against what they’ve already prepared.
  • VALC, agent install, customer go-live — that’s RR Network Tech.
  • Schema design, the install zip itself, SSIS package authoring — that’s RR DB Developer.
  • Mailbox triage and Tier 1 support — that’s the Helpdesk Tech.
Draft — manager to validate

This guide assumes you already have working SQL Server, SSMS, and Visual Studio Community with the SSIS extension installed. It also doesn’t cover specifics like how installs are assigned to DBAs, your team’s shadowing convention for first installs, or what you do between installs. Confirm those with your manager before your first real assignment.

The four phases of an install

The 12 steps group into four distinct phases. Each phase has a different texture — some are async, some are live; some are checklists, some are working sessions; the gates between them are different in kind. Understanding the phase boundaries is the difference between feeling lost in a procedure and knowing exactly where you are.

  • Phase 1 · Intake (Steps 1–2) — async, before any meeting. Receive the customer’s pre-install email, gate-check it for completeness, schedule the install meeting only if the submission is complete.
  • Phase 2 · Setup (Steps 3–5) — pre-meeting prep plus the opening minutes of the meeting. Architecture review, OLE DB driver verification, fresh on-screen download of the install zip.
  • Phase 3 · Configure (Steps 6–10) — the bulk of the 2-hour meeting. SSISDB catalog setup, RR database creation scripts, SSIS package configuration, initial data load, security configuration. You drive in SSMS and Visual Studio; Customer I/T watches and provides credentials.
  • Phase 4 · Verify-and-Handoff (Steps 11–12) — post-meeting, async. Wait for Customer I/T to verify the initial scheduled load succeeded, then send the completion email that triggers RR Network Tech to start VALC.

The two important gates between phases:

  • Between Intake and Setup: the submission must be complete. Don’t schedule, don’t prepare, don’t do anything until it is. Bouncing a submission back for completion is normal and expected.
  • Between Configure and Verify-and-Handoff: the customer must confirm the initial load succeeded. Don’t send the completion email until that confirmation arrives. The VALC tech needs the database online and populated; notifying early creates re-work for them.
Phase 1 · Async
01
Phase 1
Intake
Steps 1–2

Before any meeting. The customer has finished their pre-install checklist and emailed rrsupport@getgsi.com; the Helpdesk Tech has forwarded it to you. Your job in this phase is to verify the submission is actually complete and either schedule the meeting or send it back for re-submission.

Step 1

Receive the pre-install submission

Async

The customer’s pre-install submission email arrives via the Helpdesk Tech — subject RapidReconciler Pre-Install Submission — {Customer Name}. The body is long and structured: TOPOLOGY SELECTED, SOURCE PLATFORM, SERVER NAMES & INTERNAL IPs, CHECKLIST SUMMARY, CHECKLIST DETAIL, JDE DATA DICTIONARY VARIABLES, JDE CONNECTION DETAILS, SUBMITTED BY.

Key points
  • If the customer was a previous PoC, the production install rebuilds from the PoC environment rather than starting clean. Note this from the submission and adjust accordingly.
  • The SUBMITTED BY block is your contact for the meeting invite — usually the customer’s I/T lead, sometimes their named DBA.
  • Don’t start work on the install yet. The next step is gate-checking.
Step 2

Gate-check the submission & schedule the install meeting

Async

Verify the submission is complete before scheduling. If anything’s missing, reply asking for re-submission. If complete, schedule a 2-hour web-meeting with the SUBMITTED BY contact, sending the standard 9-item agenda.

What “complete” means
  • 30 of 30 checklist items ticked in the CHECKLIST SUMMARY.
  • Topology selected — one of Configuration 1 (Co-located), 2 (Separate DB & App), or 3 (Separate SSIS).
  • Server names & IPs populated for every relevant server in the chosen topology.
  • JDE connection details with the trailing period on the table qualifier (proddta. not proddta).
  • JDE data dictionary variables — ECST, UNCS, PQOH, TRQT decimals filled in.

The 9-item meeting agenda

Approx Agenda item
0:00Introductions & topology confirmation against actual environment
0:10Pre-flight checks (server access, credentials, drivers)
0:15Download the install zip on-screen (Step 5)
0:20Create SSISDB catalog & RapidReconciler folder (Step 6)
0:30Run the four installation scripts (Step 7)
0:55Configure & deploy SSIS package (Step 8)
1:35Schedule SQL Agent job for the initial load (Step 9)
1:45Verify security configuration (Step 10)
1:55Wrap-up: what happens next, when to expect the VALC handoff
Critical scheduling note
  • Customer I/T must be available the full 2 hours, not just the start. They’re entering local credentials, answering environment questions, and confirming actions throughout. Surface this in the meeting invite explicitly.
Phase 2 · Pre-meeting + meeting open
02
Phase 2
Setup
Steps 3–5

The bridge between intake and the working session. Two of these steps are pre-meeting prep on your side; the third is the first thing you do once the meeting starts. Getting Setup right is what makes Configure go smoothly.

Step 3

Architecture review

Pre-meeting

Match the customer’s selected topology against their actual environment as described in the submission. Verify hardware/RAM/disk-space minimums against the chosen configuration. Catch mismatches now, not in the meeting.

Three configurations
  • Configuration 1 · Co-located — database, SSIS, and application all on one server. Smaller customers; simplest topology.
  • Configuration 2 · Separate DB & App — database server with co-located SSIS, plus a separate application server.
  • Configuration 3 · Separate SSIS — database, SSIS, and application all on different servers. Largest setups.

If the topology in the submission contradicts what the server inventory suggests — e.g. they selected Co-located but the SERVER NAMES block lists three separate servers — pause here and email the customer to reconcile before scheduling.

Step 4

Verify the OLE DB drivers

Pre-meeting

Confirm the correct OLE DB provider for the customer’s JDE source platform is staged on the SSIS host. Driver mismatches are the single most common cause of SSIS deployment failures — catching them at this step rather than during deployment saves 30+ minutes of meeting time.

By source platform
  • Oracle JDE — Oracle Data Access Components (ODAC). Both 32-bit and 64-bit if SSIS will run in either mode.
  • AS/400 / IBM i — iSeries Access (Client Access) drivers. Architecture-specific.
  • SQL Server JDE — native OLE DB driver, usually present already.

See the discipline section on driver architecture — getting 32-bit vs 64-bit right matters more than people expect.

Step 5

Download the install zip on-screen at the meeting

First 5 min of meeting

At the start of the install meeting, screen-share with the customer and download a fresh copy of the install zip directly to their database server from https://files.getgsi.com/rapidreconciler/install/RapidReconciler-Install-Latest.zip. Unzip on-screen.

Why this is its own step
  • Always fresh. Never reuse a zip from a prior install. RR DB Developer updates the staged version periodically; using an old copy means installing an old schema.
  • On-screen download eliminates an entire class of tickets. “The customer says the install zip is corrupted” or “the customer can’t find the install files” — both go away when you do the download yourself with the customer watching.
  • It’s your first read on their server access. If the customer can’t download from files.getgsi.com, you’ve found a firewall problem in the first five minutes rather than at Step 7.
Phase 3 · The working session
03
Phase 3
Configure
Steps 6–10

The meat of the install — the bulk of the 2-hour meeting. You’re driving in SSMS and Visual Studio against the customer’s database server (via screen-share or RDP); they’re providing credentials, answering environment questions, and watching. Five steps; about 90 minutes of meeting time.

Step 6

Create the SSISDB catalog & RapidReconciler folder

~10 min

On the target SQL Server, set up the Integration Services Catalog if it doesn’t already exist, then create a RapidReconciler folder inside it. This is where the deployed SSIS package will live in Step 8.

Notes
  • If SSISDB already exists (some customers have other SSIS workloads on the same server), don’t recreate it — just create the RapidReconciler folder inside the existing catalog.
  • Catalog creation requires sysadmin. The customer’s I/T contact must be using a sysadmin login or be available to enter sysadmin credentials at this step.
Step 7

Run the four installation scripts

~25 min

Run the four .sql files from the install zip in order. They create the database, all schema objects, the rruser SQL login, and the RapidReconciler_Prod SQL Agent job (with an empty package path — you’ll wire it up in Step 8).

The four scripts
  • 1 - RapidReconciler Database Creation Script.sql — creates RapidReconciler_Prod at 5 GB initial size. Hard-coded for Prod.
  • 2 - RapidReconciler Database Object Script {N}.sql — creates all schema objects (tables, views, procs). The {N} is the schema revision number.
  • 3 - RapidReconciler User Creation Script.sql — creates the rruser SQL login, db user mapping, and role assignments.
  • 4 - RapidReconciler SQL Agent Job Creation Script.sql — creates the SQL Agent job framework.
Step 8

Configure & deploy the SSIS package

~40 min

The longest step. In Visual Studio, create the solution, configure the JDE source connection manager (using the customer’s details from the submission), configure the RR destination, set the SSIS variables, deploy to the SSIS catalog, and point the SQL Agent job from Step 7 at the deployed package.

SSIS variables to set per customer
  • aaStartDateGr — data-extraction start date (typically 2 months prior to current fiscal year).
  • dbowner — JDE table qualifier with the trailing period (e.g. proddta.). The most common SSIS misconfiguration is forgetting the period.
  • DecExtCost, DecUnitCost, DecQTY, DecQtyCX — decimal places sourced from the customer’s JDE data dictionary (ECST, UNCS, PQOH, TRQT).

See discipline section on the trailing period — this is the single most common install bug.

Step 9

Initial data load

~10 min

Enable the SQL Agent job’s schedule against an off-peak run time agreed with Customer I/T. The first scheduled run will be the initial load — the data extract that pulls the customer’s JDE inventory data into RapidReconciler.

Notes
  • Don’t kick off the load manually in the meeting. Initial loads can take hours depending on data volume; schedule it for off-peak and confirm via the customer load verification gate (Phase 4).
  • The customer is the one who chooses the off-peak window since their JDE source server may have its own performance considerations.
Step 10

Configure security

~10 min

Verify rruser has the right permissions, the SQL Agent service account can access the SSIS catalog, and SSIS catalog roles are assigned correctly. Most of this was done implicitly by Steps 6–7; this step is the explicit verification pass before you wrap up the meeting.

Verify these
  • rruser has db_datareader, db_datawriter, and SQLAgentOperatorRole on msdb.
  • SQL Agent service account can read from the SSIS catalog (otherwise the scheduled job can’t execute the deployed package).
  • SSIS catalog roles: ssis_admin, ssis_logreader, dc_operator as appropriate for the customer’s administration model.
Phase 4 · Async, post-meeting
04
Phase 4
Verify-and-Handoff
Steps 11–12

The meeting’s done; the load is scheduled. Now you wait. Customer I/T runs the verification once the first scheduled load completes and emails you the result. Only after that confirmation do you send the completion email that triggers VALC.

Step 11

Wait on customer load verification

Async — hours to days

After the first scheduled run completes, Customer I/T runs a verification query against the diagnostic view in the database, confirms the load succeeded, and emails you the result. You don’t proceed to Step 12 until that email arrives.

Notes
  • The customer is doing this verification, not you. At the end of the meeting (Step 10 wrap-up), you point them at the diagnostic view and the columns to check (ErrorNum, UpdateCount) and ask them to email you back when they’ve confirmed.
  • If the customer reports back with a failure, this is where you triage. Common failure modes: trailing-period miss in dbowner, driver-architecture mismatch, JDE credential issue. The diagnostic view’s ErrorNum column points at the cause.
  • Don’t notify VALC early. RR Network Tech needs the database online and populated to run their step; an early notification creates re-work for them.
Step 12

Send the completion email

~5 min

With customer confirmation in hand, send the database-ready notification to rrsupport@getgsi.com. This email is the trigger for RR Network Tech to start the VALC install — it’s a structured email with a defined format.

Email format
  • Subject: RapidReconciler Database Ready — {Customer Name} — VALC Install Can Begin
  • Customer name — identifies the install for the network tech.
  • SQL port — only mention if non-default (i.e. not 1433). If default, omit.
  • rruser password — only mention if customer changed it from the default. If default, omit.
  • Confirmation that initial load was verified by Customer I/T.
  • Pre-flight checklist — topology-filtered list confirming each install step is complete.

See discipline section on completion email format. Network Tech parses these structured emails to populate VALC; deviating from the format means they have to come back and ask.

Operational Discipline

The six rules that keep installs from going sideways

The procedure works when the discipline below is followed. Most install failures aren’t novel — they’re the result of skipping one of these. Internalising them now means recognising the failure modes when you encounter them later.

Rule 1

Gate-check submissions before scheduling

If the submission isn’t complete, don’t schedule. Bouncing it back for re-submission is normal; it’s the customer’s job to provide a complete pre-install document and yours to enforce that they have.

The temptation is to schedule and figure out the missing pieces in the meeting. Don’t. Missing items typically point at preparation the customer hasn’t actually done — firewall ports not opened, service accounts not provisioned, JDE access not confirmed. Discovering those gaps live in the meeting wastes the customer’s time and yours.

Rule 2

Always download the install zip fresh

Never reuse a copy from a prior install. RR DB Developer updates the staged version on files.getgsi.com periodically — an out-of-date copy installs an out-of-date schema, which the customer’s database will then carry forever (or until a SQL Scripts push catches it up, which is extra work for everyone).

Doing the download on-screen with the customer is also your first read on their server’s outbound connectivity. Catching firewall issues in minute 1 of the meeting beats catching them in minute 25.

Rule 3

The trailing period on dbowner is mandatory

The dbowner SSIS variable holds the JDE table qualifier — e.g. proddta. SSIS expects it with a trailing period: proddta. not proddta.

Without the period, SSIS produces table references like proddtaF4111 instead of proddta.F4111, which fails. The error message points at “invalid object” rather than “missing period” — so the diagnosis takes longer than the fix.

Triple-check this when you set the variable in Step 8. The submission email shows it with the period; copy-paste rather than re-typing.

Rule 4

Get the OLE DB driver architecture right

32-bit vs 64-bit OLE DB drivers is the second most common SSIS deployment failure. The rule:

  • SSIS package execution can be either 32-bit or 64-bit, configurable per job step.
  • The matching driver architecture must be installed on the SSIS host before package deployment.
  • For Oracle and IBM i sources, install both 32-bit and 64-bit if there’s any doubt about which mode SSIS will run in.

Verifying this is Step 4’s job. The cost of installing both architectures is negligible; the cost of finding out at Step 8 that the wrong one is staged is 30+ meeting minutes.

Rule 5

Wait on customer load verification before sending the completion email

This is the gate between Phase 3 and Phase 4. It’s also the most easily skipped — the meeting felt successful, the customer seemed happy, and the scheduled job will surely run fine.

It usually does. But the load can fail in ways that the in-meeting verification didn’t catch — data volume that exceeds default timeouts, JDE-side credentials that are valid for read but not for the actual extract, decimal-handling errors that only show with real data.

Notifying RR Network Tech before the load is verified means they start VALC against a database that might not be fully populated, or that fails at the agent connection step. They come back and ask “is this customer actually ready?” — and the answer should never be “let me check.”

Rule 6

Send the completion email in the defined format

RR Network Tech parses the structured completion email when starting VALC — the customer name goes into the VALC client record, the non-default port (if any) goes into the database registration, the password (if non-default) gets used at the agent install. Free-form completion emails mean the network tech has to come back and ask for each piece.

The format isn’t complicated; it’s in Step 12 of the install procedure. Stick to it. The two omit-if-default fields (port, password) are deliberate — if you list a default value, the network tech treats it as non-default and configures VALC accordingly, which then fails.

Wrap-up

What to do next

You’ve walked the four phases and seen all 12 steps. Three suggested next steps before your first install:

  1. Read the full Installing the Production Database procedure end-to-end. This Start Here is a wayfinding guide; the procedure doc is your real reference. Knowing the procedure forwards and backwards before you touch a customer install is the difference between confidence and improvisation.
  2. Read the example pre-install submission email in the procedure doc. Recognising the structure lets you gate-check submissions in seconds rather than minutes — and gate-checking quickly is what makes the whole pipeline work.
  3. Shadow a senior DBA on at least one install before driving one yourself. The 12 steps are documented; the cadence between you and the customer, the small judgments at each phase boundary, the rhythm of a 2-hour meeting — those aren’t in any document. Watching once is worth reading three times.
