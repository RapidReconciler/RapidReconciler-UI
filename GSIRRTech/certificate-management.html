SSL Certificate & DNS Management

Reference for the wildcard SSL certificate lifecycle, DNS A-record configuration, renewal procedures, and the responsibilities for all RapidReconciler deployments under the *.getgsi.com domain.

Internal Document

For GSI staff only. Customers are not involved in certificate or DNS administration; both are managed entirely by GSI under the shared wildcard domain.

Owner: GSI Operations Audience: GSI Staff / I/T Certificate: *.getgsi.com

How It Works

The RapidReconciler user interface is secured using a wildcard SSL certificate (*.getgsi.com) provisioned and managed by GSI. This eliminates the need for individual customers to purchase, configure, or renew their own SSL certificates — reducing both cost and administrative overhead.

Each customer is assigned a unique domain URL (e.g. rrprod-customername.getgsi.com) configured in VALC during initial installation. GSI manages the DNS and certificate infrastructure required to make this URL resolve securely to the customer’s server.

Overview & Components

How the wildcard certificate model works and the four moving pieces that combine to secure each customer’s RapidReconciler URL.

How RapidReconciler SSL Works

The RapidReconciler user interface is secured using a wildcard SSL certificate (*.getgsi.com) provisioned and managed by GSI. This eliminates the need for individual customers to purchase, configure, or renew their own SSL certificates, reducing both cost and administrative overhead.

Each customer is assigned a unique domain URL (e.g. rrprod-customername.getgsi.com) configured in VALC during initial installation. GSI manages the DNS and certificate infrastructure required to make this URL resolve securely to the customer’s server.

Infrastructure Components

Four components combine to deliver a working secure URL for each customer. Each has a distinct owner; understanding which is which is the foundation for diagnosing certificate or connectivity issues.

ComponentOwnerDescription
*.getgsi.com wildcard cert GSI Secures all customer RapidReconciler URLs under the getgsi.com domain.
DNS ‘A’ record GSI I/T Resolves the customer’s domain URL to their server’s internal IP address.
Domain URL GSI (via VALC) Unique URL assigned to each customer during installation.
RapidReconciler Agent GSI Development Deployed to the customer server; must be updated when the certificate is renewed.
Why this matters

Three of these four components are touched on every install (cert is shared, the other three are per-customer). On a server migration or IP change, only the DNS A record needs updating — the cert and domain URL stay the same. On certificate renewal, the cert and the agent need updating — the DNS A record and domain URL stay the same.

DNS Configuration

How to request a DNS A-record change — for new customers, server migrations, or IP changes. GSI I/T owns the actual DNS work; tech submits the request with the right details.

DNS A-Record Management

GSI I/T maintains DNS records with the GSI Internet Service Provider (ISP). For each RapidReconciler customer, an A record maps the customer’s assigned domain URL to the internal IP address of their application server.

Example A Record
Domain URLInternal IP Address
rrprod-acmemanufacturing.getgsi.com192.168.1.100

When a New A Record Is Required

A new or updated A record is needed in the following situations:

SituationAction Required
New customer being set upAdd new A record
Customer migrates to a new serverUpdate A record with new IP
Customer’s internal IP address changesUpdate A record with new IP

Process

  1. Confirm the domain URL and internal IP address on the Client Details page in VALC.
  2. Submit the request to GSI I/T with the domain URL and internal IP address.
  3. GSI I/T adds or updates the A record with the ISP.
  4. Allow time for DNS propagation before testing connectivity.
Note

DNS propagation can take anywhere from a few minutes to several hours depending on the ISP and TTL settings. The RapidReconciler UI will not be accessible at the domain URL until propagation is complete.

See Server Migration → VALC & Cloudflare for the cutover sequencing when DNS changes are part of a migration. See Installing a Client in VALC → Network Access Requirements for the firewall and connectivity context the DNS update fits into.

SSL Renewal

Three-step procedure for renewing the wildcard *.getgsi.com certificate — one cert covers every customer, but each customer’s agent needs the renewed cert pushed to it.

Certificate Renewal Process

The *.getgsi.com wildcard certificate covers all subdomains under getgsi.com — a single certificate secures every customer’s RapidReconciler URL. GSI is responsible for procuring and renewing it.

Once procured, the renewal flows through three handoffs:

1
Step 1 of 3 · UI Developer
Backend Update

The UI developer updates the backend configuration with the renewed certificate. This ensures the RapidReconciler web interface continues to serve the correct certificate after renewal.

2
Step 2 of 3 · GSI Staff (via VALC)
Agent Deployment via VALC

Once the backend is updated, the renewed certificate must be deployed to each customer’s RapidReconciler Agent via the Manage Deploys process in VALC. This pushes the updated agent package to each customer server.

This step is per-customer, not global. For a customer base of any size, plan it as a sequenced rollout rather than a single batch — the verification in Step 3 needs to happen for each customer.

3
Step 3 of 3 · GSI Staff
Verification

After each customer’s deployment, verify:

  • The RapidReconciler UI loads correctly for the customer’s domain URL.
  • The browser shows a valid, non-expired certificate — issuer and expiry date reflect the renewed cert.
  • SSO sign-in works end-to-end. A valid cert chain doesn’t guarantee SSO is healthy; downstream pieces can be broken behind a green padlock.

Validate immediately after each customer’s deployment — don’t batch the verifications to the end of the rollout. The cost of catching a broken SSO mid-rollout is much lower than catching it after every customer is impacted.

Important — expiry has cascade effect

If the certificate is not deployed to the agent before the existing certificate expires, customers will receive browser security warnings and may be unable to access the RapidReconciler UI. Plan renewals well in advance of the expiry date — aim to start Step 1 at least 2–3 weeks before expiry.

This has actually happened. A delayed SSL deployment caused an SSO outage that impacted multiple customers at once — the rollout was left incomplete past expiry, and affected customers couldn’t access the application until the renewed cert reached their agent. Resolution was straightforward (deploy the cert, redeploy SSO), but the avoidable lesson was not finishing the rollout before expiry. Take from it:

  • Complete cert deployment across all customer agents before the existing cert expires — no customers left for “tomorrow.”
  • Validate SSO sign-in (not just the cert chain) immediately after each customer’s deployment. SSO failures can hide behind a valid cert if a downstream piece didn’t redeploy cleanly.
  • Communicate status proactively during the rollout window. A scheduled deployment touching many customers is a customer-impacting event — treat it as one.

Responsibilities & Troubleshooting

Reference: who owns each piece of the certificate & DNS infrastructure, and the four most common symptoms with their likely cause and remediation.

Who Does What

Four parties touch this infrastructure. Knowing which party owns which action is the difference between routing a request to the right person on the first try and chasing it through a chain of handoffs.

PartyResponsibilities
GSI
  • Procure and renew the wildcard SSL certificate
  • Configure the domain URL in VALC
  • Deploy updated agent via VALC Manage Deploys
  • Verify connectivity and certificate validity post-renewal
GSI I/T
  • Add or update DNS A records with the ISP
  • Coordinate on IP address changes
  • Resolve DNS propagation issues beyond 24 hours
GSI UI Developer
  • Update backend configuration after certificate renewal
  • Confirm web interface serves the renewed certificate correctly
GSI Staff
  • Submit DNS update requests with correct domain URL and IP
  • Trigger Manage Deploys after backend is confirmed updated
  • Perform post-deployment browser verification per customer

Common Issues

The four most common symptoms reported, mapped to likely cause and the action to take. Most issues fall into one of these patterns; if a symptom doesn’t match any of these, check the agent log files first.

SymptomLikely CauseAction
“Not Secure” or certificate warning in browser Certificate expired or not yet deployed to agent Verify certificate renewal and redeploy via VALC Manage Deploys
Domain URL does not resolve A record missing or incorrect Contact GSI I/T to verify the DNS entry
UI loads but agent is not connecting IP address change not reflected in VALC or DNS Update the internal IP in VALC and request a DNS update from GSI I/T
DNS propagation taking too long ISP TTL settings Allow additional time; contact GSI I/T if the issue persists beyond 24 hours
