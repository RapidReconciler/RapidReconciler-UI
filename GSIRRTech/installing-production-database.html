How to read this

Step 1 — Submission Email Lands in the rrsupport@getgsi.com Inbox

The database install kicks off when the customer's pre-install submission email arrives in the rrsupport@getgsi.com inbox. Check the inbox at the start of every install; the submission lands there with a subject line of the form RapidReconciler Pre-Install Submission — {Customer Name}. The email is generated when the customer completes the Tech Installation Prep guide on their own and contains every value the RR DBA needs to install and configure the database — topology, server names and internal IPs, source platform, JD Edwards connection details, decimal handling, and a complete environment readiness checklist.

Do not start without this email

The submission email in the rrsupport@getgsi.com inbox is the trigger for the database install. If you haven't received it, contact the customer's I/T lead and direct them to the Tech Installation Prep guide. Starting the install from a partial discovery call risks recording the wrong server names or IPs, which causes rework during SSIS deployment (Step 8) and the SQL Agent job configuration.

Same email, two roles

This same submission email also kicks off the Installing a Client in VALC process (performed by another tech in the same group). Both roles work from the same email in the rrsupport@getgsi.com inbox — the database side runs first, then VALC.

Before You Start

This walkthrough assumes you are an entry-level SQL Server DBA or a developer with comparable familiarity. A quick check of skills and inputs before beginning saves rework later.

Time required

The full installation typically takes about 2 hours. Plan to do it in one sitting alongside the customer's IT contact during a scheduled web meeting.

Required Skills

RapidReconciler is a Microsoft SQL Server based product. You should be comfortable with:

  • Running scripts in SQL Server Management Studio (SSMS)
  • Basic SQL Server database configuration (creating databases, logins, users, role memberships)
  • SQL Server Agent job configuration — opening jobs, editing steps, configuring schedules
  • SQL Server Integration Services (SSIS) — opening a package in Visual Studio, working with connection managers, deploying to a catalog

If any of these is unfamiliar, pair with a senior DBA on the first install.

Required Tools in the Customer's Environment

Per Step 4 (Review Architecture), these tools must be present somewhere in the customer's environment — either on the database server or the application server:

  • SQL Server Management Studio (SSMS)
  • Visual Studio Community with the SQL Server Integration Services extension installed
  • The appropriate OLE DB provider for the customer's JDE source (covered in Step 4 — Review Architecture)
Looking for the install files?

The install zip and download instructions are documented on the Step 5 — Download & Unzip Install Files page.

Step 5 — Download & Unzip the Install Files

At the start of the install meeting, after the customer has provided server access (Step 3) and you have reviewed the architecture (Step 4), download the GSI installation zip directly to the database server and unzip it. The role for this step shifted from the customer to the RR DBA — previously the customer was asked to download and unzip ahead of the meeting, but having the DBA do it on-screen during the meeting eliminates the most common variant of "missing files" tickets.

Owner: RR DBA (during the meeting, with the customer's I/T contact on the call)

Drive the download from the database server itself, not your workstation. The zip should be unzipped to a folder the SQL Server service account can read — a folder under C:\GSI\RapidReconciler-Install\ (or similar) on the database server is fine.

Download Location

Download the latest install zip from GSI's distribution location:

  • Primary download: https://files.getgsi.com/rapidreconciler/install/RapidReconciler-Install-Latest.zip
  • If the database server cannot reach this URL directly (locked-down environment), download to your workstation, then transfer to the database server using the customer's standard file-transfer method (SMB share, secured copy, etc.). Do not email the zip — it is too large for most mail systems and may be blocked by attachment scanners.
Verify the download is current

Always download a fresh copy at the start of each install rather than reusing a copy from a prior install. The install zip is updated periodically — the database object script in particular is versioned (the file ending ...178.sql reflects the schema version). Using an out-of-date zip means installing an out-of-date schema. If you need to confirm the current version before the meeting, contact rrsupport@getgsi.com.

What's in the Zip

The zip contains four .sql scripts and one .dtsx SSIS package:

FilePurposeUsed in
1 - RapidReconciler Database Creation Script.sqlCreates the database, mdf/ldf files, and sets initial sizeStep 7
2 - RapidReconciler Database Object Script 178.sqlCreates all database objects (tables, views, procedures)Step 7
3 - RapidReconciler User Creation Script.sqlCreates the rruser SQL loginStep 7
4 - RapidReconciler SQL Agent Job Creation Script.sqlCreates the SQL Agent jobStep 7
RapidReconciler-Prod.dtsxThe SSIS package to be deployed to the SSIS catalogStep 8

Unzip Procedure

  1. On the database server (via screen-share with the customer), open a browser and navigate to the download URL above.
  2. Save the zip to a known location — C:\GSI\RapidReconciler-Install\ is conventional.
  3. Right-click the downloaded zip and select Extract All. Extract to the same parent folder.
  4. Confirm all five files listed in the table above are present in the extracted folder.
  5. Leave the folder open in File Explorer — you'll be opening these files in SSMS and Visual Studio over the next several steps.
Note

If the customer's environment requires the install zip to be staged ahead of the meeting (e.g. download must go through change control), the customer's I/T contact can download it on the DBA's behalf in advance. Confirm during scheduling (Step 2) so there's no surprise at the start of the meeting.

Step 4 — Review Architecture & Requirements

Use the topology the customer selected on the pre-install submission email to confirm their environment matches the chosen configuration. Read this section before the install meeting — many install delays come from server requirements not being met or driver mismatches that aren't noticed until SSIS deployment fails.

Step 3 — Customer I/T provides server access (at the start of the meeting)

Step 3 in the overall flow has no separate procedure section because it isn't DBA work — it's owned by the customer's I/T contact at the start of the install meeting. What they need to provide:

  • Remote access to the database server (and the application/SSIS server, if separate per Configuration 2 or 3) — typically via screen-share with elevated rights, or RDP credentials.
  • SQL Server sysadmin credentials — needed for SSISDB catalog creation (Step 6) and database install (Step 7).
  • JD Edwards source access — the read-only SQL/Oracle login the SSIS package will use, plus the OLE DB driver installed on the server that hosts SSIS (per the topology). The driver is the most common cause of late-stage failure; verify it's there before Step 8.
  • An I/T contact on the call for the full 2 hours — not just to provide credentials at the start, but to answer environment questions during Steps 6–10 and to verify the initial load in Step 11.

If any of the above isn't ready when the meeting starts, pause and reschedule rather than push forward with workarounds — downstream steps depend on these being correct.

Configuration 1 — Co-located

Database, application services, and SSIS all run on a single dedicated server. Simplest setup; typical for smaller customers, pilots, or environments where IT prefers a single dedicated VM for RR.

Configuration 1 — Co-locatedOne dedicated server runs everything: SQL Server, the RR database, SSIS, and the RR application services.

Servers Involved

  • One dedicated RR server — hosts SQL Server, the RR database, SSIS, and the RR application services
  • Customer's existing JDE server — not part of the RR install, but RR connects to it via SSIS in read-only mode

Configuration 2 — Separate Database and Application Servers

Most common enterprise pattern. The database server (with SSIS) is separate from the application server. Standard pattern when the customer has a policy separation between database and application tiers.

Configuration 2 — Separate DB & AppDatabase server (with SSIS) on the left; separate application server on the right.

Servers Involved

  • Database server — hosts SQL Server, the RR database, and SSIS / Integration Services
  • Application server — hosts the RR application services
  • Customer's existing JDE server — not part of the RR install, but RR connects to it via SSIS in read-only mode

Configuration 3 — Separate SSIS Server

Uncommon. SSIS runs on its own server, separate from the database. Used when customer policy requires SSIS to run on a host other than the database server.

Configuration 3 — Separate SSISJDE server, SSIS server, database server, and application server — four servers total.

Servers Involved

  • SSIS server — hosts Integration Services only; pulls data from JDE
  • Database server — hosts SQL Server and the RR database
  • Application server — hosts the RR application services
  • Customer's existing JDE server — not part of the RR install, but the SSIS server connects to it in read-only mode

Disk Space

Estimate disk space based on the customer's transaction volume:

  • 100 GB per 1 million cardex (F4111) records per month
  • Estimate covers 1 year of data; records may be purged after year-end close

Example: a customer generating 2,000,000 cardex records/month requires 200 GB of disk space allocated to the RapidReconciler database.

Common cause of install delays

Driver mismatch (e.g. only 64-bit installed when both are needed) is a frequent cause of SSIS connection failures. Verify the OLE DB provider is installed correctly before the install meeting.

Notes on deployment
  • Port 443 outbound is used by RapidReconciler for JMS communications.
  • If the customer's submission email shows a topology that doesn't match what their I/T contact describes on the call, pause and reconcile before continuing — the entire install plan keys off this choice.

Step 2 — Schedule the 2-Hour Install Meeting

With the submission email in hand from Step 1, schedule a 2-hour web meeting with the customer's I/T contact. This is the meeting where steps 6–10 will be performed live, with the DBA driving and the customer's I/T contact on the call.

Pull the Contact from the Submission

Open the submission email in rrsupport@getgsi.com and locate the SUBMITTED BY block at the bottom. Use the name, email, and phone listed there to send the meeting invite. The submitted contact is the customer's I/T lead and will be your counterpart for the entire install.

Confirm the Submission Is Complete Before Sending the Invite

Before sending the meeting invite, do a quick pass over the submission to confirm it's complete:

  • Checklist Summary reads 30 of 30 items complete. Any unchecked item is a blocker.
  • Topology Selected is filled in (Configuration 1, 2, or 3) — you'll verify the customer's environment matches this in Step 4.
  • Server Names & Internal IPs populated for the topology selected.
  • JD Edwards — Connection Details populated, including the table qualifier (with trailing period).
If the submission is incomplete

If any checklist item is unchecked or any field is missing, do not schedule the meeting. Reply to the submission email asking the customer to complete the missing items in the Tech Installation Prep guide and re-submit. Scheduling on an incomplete submission is the most common cause of install meetings running over the planned 2 hours or having to be rescheduled.

What to Include in the Meeting Invite

  • Duration: 2 hours.
  • Attendees: the customer's I/T contact (from the SUBMITTED BY block) plus you. The customer's contact must be available for the full 2 hours — not just the start — to enter local credentials and answer environment questions during the install.
  • Format: web meeting with screen-share. The customer will share access to the database server (Step 3) so you can drive in SSMS and Visual Studio.
  • Prep ask for the customer: have the install zip downloaded and ready (covered in Step 5 — you'll handle the unzip together at the start of the meeting), and confirm sysadmin SQL credentials for the install.

Meeting Agenda

Include this agenda in the invite so the customer knows what to expect. Use the Copy agenda button below to copy the agenda as plain text suitable for pasting into a calendar invite.

Meeting agenda — copy & paste
  1. Confirm server access and review the topology against the submission (Steps 3–4)
  2. Download & unzip the install files on the database server (Step 5)
  3. Create the Integration Services Catalog (Step 6)
  4. Install the RapidReconciler database, create the rruser login, and create the RapidReconciler_Prod SQL Agent job (Step 7)
  5. Configure and deploy the SSIS package (Step 8)
  6. Perform the initial data load (Step 9)
  7. Confirm security permissions (Step 10)
Order-of-operations tip

Step 6 (creating the SSIS catalog) can be done before Step 7 (database install) if convenient, since they are independent. The remaining steps must be completed in the order shown above.

After the Invite Is Accepted

Once the customer accepts the meeting, your work between now and the meeting is in Step 4 — Review Architecture & Requirements. Step 4 happens before the meeting on the calendar but is logically after Step 3 (server access) in the flowchart, since the server access happens at the start of the meeting itself.

Step 6 — Create the SSISDB Catalog

The Integration Services Catalog (SSISDB) must exist on the target SQL Server before the RapidReconciler SSIS package can be deployed. If the catalog has already been created on this server, skip to Step 2 to create the RapidReconciler folder.

1
Step 1 of 2
Create the SSISDB Catalog
  1. Open SQL Server Management Studio and connect to the target SQL Server instance.
  2. In Object Explorer, expand the server node.
  3. Right-click Integration Services Catalogs and select Create Catalog.
  4. In the Create Catalog dialog:
    • Check Enable CLR Integration if not already enabled.
    • Check Enable automatic execution of Integration Services stored procedure at SQL Server startup.
    • Enter and confirm a Password to protect the database master key.
  5. Click OK to create the catalog.
  6. Confirm SSISDB now appears under Integration Services Catalogs in Object Explorer.
Important

Creation of the SSISDB catalog requires sysadmin privileges on the SQL Server instance. The password entered here will be required if the catalog ever needs to be restored from backup — store it in a secure location.

2
Step 2 of 2
Create the RapidReconciler Folder
  1. In Object Explorer, expand Integration Services Catalogs → SSISDB.
  2. Right-click SSISDB and select Create Folder.
  3. In the Create Folder dialog:
    • Set Folder name to RapidReconciler (or RR).
    • Optionally enter a description such as RapidReconciler SSIS packages.
  4. Click OK to create the folder.
  5. Confirm the RapidReconciler folder now appears under SSISDB in Object Explorer.
Note

This folder name must match the deployment path used in Step 8, sub-step 6 (Deploy the Project to the SSIS Catalog). The path /SSISDB/RapidReconciler/RapidReconciler references this folder as the first RapidReconciler segment.

Step 7 — Install the RapidReconciler Database

This tour runs the four installation scripts in sequence to create the database, install all database objects, create the rruser SQL login, and create the RapidReconciler_Prod SQL Agent job.

1
Step 1 of 5
Confirm the Installation Files Are on the Server

Confirm with the customer's IT contact that the GSI installation zip has been downloaded and unzipped on the target database server. The unzipped folder should contain the four .sql scripts and the RapidReconciler-Prod.dtsx SSIS package listed in Before You Start.

If the zip is not yet on the server, pause and have the customer download and unzip it before continuing.

2
Step 2 of 5
Create the Database
  1. Log on to the database server alongside the customer's IT contact.
  2. Open SQL Server Management Studio.
  3. Log in with an account that has sysadmin privileges.
  4. Open and run 1 - RapidReconciler Database Creation Script.sql.

The script will:

  • Create mdf and ldf files in the default locations specified by the server instance
  • Set the initial database size to 5 GB
  • Create the database with the default name RapidReconciler_Prod
Important — Change the database context

After this script completes, change the database context from master to RapidReconciler_Prod in the SSMS connection dropdown before opening any of the remaining scripts. Each subsequent script must run against RapidReconciler_Prod, not master.

3
Step 3 of 5
Add Database Objects

Run 2 - RapidReconciler Database Object Script 178.sql against RapidReconciler_Prod.

Wait for the Successfully Completed message before continuing. This script creates all tables, views, and stored procedures the application requires.

4
Step 4 of 5
Create the SQL User ID

Run 3 - RapidReconciler User Creation Script.sql.

This creates the rruser SQL login. These are the credentials the application uses to read and write data to the SQL database. The script handles both the server-level login creation and the database-level user mapping.

Permissions detail is covered in Security Requirements.

5
Step 5 of 5
Create the SQL Agent Job

Run 4 - RapidReconciler SQL Agent Job Creation Script.sql.

This creates the RapidReconciler_Prod SQL Agent job. The job's package path will be configured in Step 8 once the SSIS package is deployed, and the job's schedule will be enabled in Step 9.

Verify before continuing

In SSMS Object Explorer, expand SQL Server Agent → Jobs. Confirm the RapidReconciler_Prod job appears in the list. If it does not, re-check that the script ran without errors.

Step 8 — Configure & Deploy the SSIS Package

This is the longest step. You'll create a Visual Studio solution, add the prebuilt SSIS package, configure its connection managers and variables, deploy the project to the SSISDB catalog, and wire the deployed package up to the SQL Agent job created in Step 7 (Install Database).

Prerequisites

  • Visual Studio Community with the SQL Server Integration Services extension installed
  • The RapidReconciler-Prod.dtsx file extracted from the installation zip
  • The SSISDB catalog and RapidReconciler folder created in Step 6
  • The database, rruser login, and SQL Agent job created in Step 7
1
Step 1 of 8
Gather Configuration Information

Before opening Visual Studio, gather the following from the Tech Requirements document the customer's IT contact filled out:

JD Edwards Connection Details

  • Name or IP address of the JDE data server
  • Read-only credentials for the JDE database (use rapidrec/rapidrec if possible)
  • JDE server type: iSeries, Oracle, or Microsoft SQL Server

JDE Data Configuration

  • Table qualifier for JDE data (e.g. proddta. — note the trailing period)
  • Decimal places for: extended cost, unit cost, quantity on hand, transaction quantities
  • Customer's fiscal year start date (used to compute the aaStartDateGr variable — see Step 5)
2
Step 2 of 8
Create the Visual Studio Solution
  1. Launch Visual Studio Community. Run as administrator if possible to avoid permission issues during deployment.
  2. Select Create a new project.
  1. Search for and select Integration Services Project, then click Next.
  1. Set the Project name to RapidReconciler.
  2. Choose an appropriate location on the server to save the solution.
  3. Click Create.
  1. Once the solution loads, confirm the project appears in Solution Explorer as RapidReconciler.
3
Step 3 of 8
Add the Existing SSIS Package
  1. In Solution Explorer, expand the RapidReconciler project and right-click the SSIS Packages folder.
  2. Select Add Existing Package.
  3. In the Add Copy of Existing Package dialog, set Package location to File system, then browse to where the installation zip was extracted.
  4. Select RapidReconciler-Prod.dtsx and click OK.
  5. Confirm the package appears under the RapidReconciler project in Solution Explorer.
  6. Double-click the package to open it in the designer and verify the control flow and variables load.
  7. If prompted to upgrade the package, click OK to allow Visual Studio to update the package format.
Connection errors are expected at this stage

The package may show connection errors when it first opens — this is normal. The connection managers will be updated in Step 4. Passwords also need to be re-entered after deployment.

4
Step 4 of 8
Configure the Connection Managers

Connection managers appear at the bottom-center of the package designer.

JDE Source Connection

  1. Double-click the JDE source connection.
  2. Update the Server name or IP address to the JDE data server.
  3. Enter the database credentials gathered in Step 1 (rapidrec/rapidrec if available).
  4. Click Test Connection to verify connectivity to the JDE data source.
  5. Click OK to save changes.

RapidReconciler Destination Connection

  1. Double-click the RapidReconciler destination connection.
  2. Set the Server name to the database server hosting RapidReconciler_Prod.
  3. Select RapidReconciler_Prod from the database dropdown.
  4. Enter the rruser credentials created in Step 7.
  5. Click Test Connection to verify connectivity to RapidReconciler_Prod.
  6. Click OK to save changes.
5
Step 5 of 8
Configure the Variables
  1. Open the Variables window. If not visible, go to View → Other Windows → Variables.
  2. Reference Figure 1 (Step 4) for the variable names — they appear highlighted in yellow.
  3. Update each variable using the values gathered in Step 1:

Step 11 — Verify the Initial Load

The database is installed, the SSIS package is deployed and configured, and the SQL Agent job's schedule is enabled (Step 9). This step is the customer's I/T contact's job — they confirm the load succeeded and report the result back to the DBA.

Owner: Customer I/T (reports back to RR DBA)

Because the initial load can take longer than the 2-hour install meeting allows, the customer's I/T contact typically completes this verification on their own after the meeting ends and emails the result back to the DBA. The DBA does not need to keep the meeting open until the load finishes.

Verification Procedure — v_diagnostic4_b_to_c_log

After the first scheduled run completes, verification is done against the database view v_diagnostic4_b_to_c_log in RapidReconciler_Prod. This view records one row per data-capture operation performed by the SSIS package and is the single source of truth for whether the load succeeded.

  1. Open SSMS and connect to the database server.
  2. Expand Databases → RapidReconciler_Prod → Views and locate dbo.v_diagnostic4_b_to_c_log.
  3. Right-click the view and select Select Top 1000 Rows — or run a query against it directly: 
    SELECT *
FROM dbo.v_diagnostic4_b_to_c_log
WHERE Step = 'data capture'
ORDER BY Capture DESC, StartTime DESC;
  4. Limit the result to rows where Step = 'data capture' — these are the rows that report on the actual JDE pull. Other step values (e.g. Data Capture with the capital D, used for the start marker) are bookends, not the data rows themselves.
  5. Look at the UpdateCount column: each row should show a number that makes sense for the JDE table being captured. Cardex (F4111) for a mid-size customer is typically tens of thousands per month; item master (F4101) typically tens of thousands total; account ledger (F0911) similarly large. Tables with no recent activity may legitimately show 0.
  6. Look at the ErrorNum column: every row should be 0. Any non-zero value indicates a failure on that table.

What to Look For

ProcessName of the JDE-to-RR copy operation, e.g. 5-F0911 to RR F0911 New.One per JDE table; New rows are first-time inserts, Chg rows are change captures, Del rows are deletions.

If the numbers don't make sense or there are errors

If the UpdateCount values look way off (e.g. all zeros across the major tables, or counts that don't reflect the customer's actual JDE volume) or any row shows a non-zero ErrorNum, the load did not succeed. Contact rrsupport@getgsi.com for assistance — do not proceed to Step 12. Include the customer name, the timestamp of the failed run from the Capture column, and a screenshot of the diagnostic view rows in your support request.
