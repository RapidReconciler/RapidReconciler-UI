Agent services not running on application server

In Windows Services console, the rr-valc-agent service (display name: Rapid Reconciler 7 Agent) is stopped or fails to start. Common causes: a Windows Update reboot, the LocalSystem account hit a permission issue on the install root or the DB, a corrupted rr-valc-agent.xml config, or the WinSW recovery loop (restart 10s → restart 30s → reboot) ran through its three attempts and gave up. Often resolves with a service restart.

What you’re seeing

  • Services console (services.msc) shows Rapid Reconciler 7 Agent as Stopped or failing to start.
  • Symptom may surface elsewhere as agent disconnected on the VALC Clients grid, or database Unknown on the Databases tab.
  • Task Manager shows no java.exe processes (since the parent agent supervises every child, stopping the service terminates them all).
Only one service to look for

Older docs referenced a second service called RapidReconciler Services — that no longer exists. The current installer (post-WinSW migration) registers only the single rr-valc-agent service. What older docs called "RapidReconciler Services" is actually the agent’s spawned child java.exe processes (one per RR database), which appear in Task Manager but not in services.msc. See the RR Agent Reference for the parent/child process model.

Walk these checks

Check 1
Restart the service
  1. On the application server, open Services console (services.msc).
  2. Find Rapid Reconciler 7 Agent (service name rr-valc-agent).
  3. Right-click → Start (or Restart if it’s in a stuck state).
  4. Confirm the status reaches Running. Task Manager → Details should show one java.exe for the agent plus one per RR database within ~30 seconds.
Running? Stop here. Otherwise, continue.
Check 2
Inspect logs and credentials
  • Open C:\Program Files\Rapid Reconciler\logs\rr-valc-agent.wrapper.log — this is WinSW’s own log and captures service start/stop attempts plus the JVM launch arguments. If the wrapper couldn’t launch the JVM (missing JAR, bad path, JRE error), it’s logged here.
  • Then check today’s rr-valc-agent_YYYYMMDD.out.log and .err.log for the Java-side startup error if WinSW did launch the JVM but the agent died early. .err.log empty = JVM didn’t crash; non-empty almost always tells you what blew up.
  • By default the service runs as LocalSystem (the shipped WinSW config has no <serviceaccount>). If IT changed it post-install to a custom account, verify those credentials are still valid — a recent password rotation can break it.
  • Check Windows Event Viewer → Application log for related errors at the time of failure; WinSW emits Service-Control-Manager events here too.
If the wrapper log or .err.log indicates a config issue, escalate to a senior tech to investigate the agent installation.
