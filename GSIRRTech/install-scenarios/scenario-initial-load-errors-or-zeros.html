Initial load reports errors or zero counts in the diagnostic

The first scheduled run of the SQL Agent job completed, but the customer DBA’s Step 11 verification against v_diagnostic4_b_to_c_log shows either non-zero ErrorNum values or UpdateCount = 0 for major tables. The database install doc is explicit: do not proceed to the Step 12 VALC handoff until the underlying cause is identified and fixed.

What you’re seeing

  • SQL Agent job’s first run completed (history shows success or partial success).
  • Step 11 verification query against v_diagnostic4_b_to_c_log shows non-zero ErrorNum values, or UpdateCount = 0 across major tables.
  • Customer DBA is asking what to do next, or the GSI tech is mid-install and blocked before Step 12.

Walk these checks

Check 1
Read the full diagnostic with ErrorNum and ErrorMsg

The Step 11 query in the install doc filters tightly; for triage, run a wider query so you can see the error text alongside the counts.

  • In SSMS, connect to RapidReconciler_Prod.
  • Run:
    SELECT Capture, Step, Process, UpdateCount, ErrorNum, ErrorMsg FROM v_diagnostic4_b_to_c_log WHERE Step = 'data capture' ORDER BY Capture DESC
  • Group rows by ErrorNum — typically one error code dominates and one fix unblocks most of them.
Dominant error identified? Match it to one of the patterns below.
Check 2
Pattern: all UpdateCount = 0, ErrorNum = 0 (no error)

This is the dbowner-trailing-period symptom. The package ran cleanly but generated query strings against non-existent tables. Switch to the dedicated runbook:

  • SSIS package deployed but extracts no data
Matches this pattern? Follow the linked runbook, then re-run Step 11.
Check 3
Pattern: login or authentication failure

Error text contains Login failed, Cannot open user default database, or Authentication failed. The package’s connection managers can’t reach SQL Server with the credentials baked into the deployment.

  • Most common cause: the destination connection (RRLocal) is configured with the wrong rruser password (or rruser wasn’t created — see SQL script fails during database install, Check 4).
  • Open the package in Visual Studio, right-click the failing connection manager → Edit, re-enter the password, click Test Connection. Must succeed before deploying.
  • Redeploy (right-click project → Deploy), confirm the package timestamp in SSISDB updated, then re-execute.
Connection tests pass and redeployed? Re-run the job and re-check the diagnostic.
Check 4
Pattern: invalid object name or column not found

Error text contains Invalid object name, Cannot find object, Invalid column name, or Conversion failed. The package’s expected JDE schema doesn’t match what the customer actually has.

  • If every JDE table is missing — check dbowner first (see Check 2 link). Wrong schema prefix produces invalid-object errors on the first table the package touches.
  • If specific tables or columns are missing — the customer is on a non-standard JDE version or has customized columns. Note the exact object and column name from ErrorMsg and escalate to GSI DBA. Don’t guess at workarounds; schema-shape changes need DBA approval.
Identified as schema-shape? Escalate to GSI DBA with the failing object names.
Check 5
Pattern: timeout

Error text contains Timeout expired, execution timeout, or Query timeout expired. The JDE source is slow under the load of an initial extract, and the default 30-second command timeout isn’t enough.

  • First-time loads on larger customers commonly hit this; subsequent runs are smaller deltas and won’t.
  • In the package, increase the JDE source connection’s Command Timeout (or the relevant data-flow task’s timeout property) to 300 seconds or more. Save, redeploy.
  • If the same tables time out on subsequent attempts, escalate to GSI DBA — may indicate a missing index on the JDE side rather than a tuning issue on the RR side.
Timeout increased and redeployed? Re-run the job and re-check the diagnostic.
Check 6
Pattern: partial counts (some tables loaded, some didn’t)

Most tables show healthy UpdateCount but a specific table or two shows zero with no error. Usually means the source table is genuinely empty for this customer.

  • Identify the affected tables from the diagnostic.
  • From SSMS connected to the JDE source, run SELECT COUNT(*) FROM proddta.<table> for each.
  • If the source count is zero — confirm with the customer whether they use that JDE module (some customers don’t populate every table the package reads). If they don’t use it, the zero-count is expected and the install can proceed.
  • If the source count is non-zero but RR shows zero — treat as a schema or permissions issue and escalate to GSI DBA.
Source confirms empty? Document the table list and proceed. Otherwise escalate.
Check 7
Still failing after targeted fixes

If the diagnostic still shows errors or zero counts after working through the matching pattern above, escalate to the GSI DBA with:

  • The full diagnostic query output (copy the result grid — include Capture, Step, Process, UpdateCount, ErrorNum, ErrorMsg).
  • The customer’s JDE version and schema name.
  • Current package variable values (dbowner, aaStartDateGr, etc.).
  • Any redeployment timestamps so the DBA knows what version of the package they’re looking at.
Escalated? Do not proceed to Step 12 (VALC handoff) until the diagnostic shows healthy row counts and zero ErrorNum.
