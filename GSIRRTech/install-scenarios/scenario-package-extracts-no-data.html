SSIS package deployed but extracts no data

The package deployed cleanly and the SQL Agent job runs to completion with no errors, but the diagnostic log shows UpdateCount = 0 across every major table. Almost always the dbowner SSIS variable was set without the trailing period, so the generated source queries point at table names that don’t exist in JDE.

What you’re seeing

  • SSIS package deployed; SQL Agent job runs and reports success.
  • v_diagnostic4_b_to_c_log for Step = 'data capture' shows UpdateCount = 0 across all major tables.
  • ErrorNum is 0 — the package thinks it succeeded; it just received zero rows back from the source.
  • No companies, no inventory data, no transactions populated in the RR UI.

Walk these checks

Check 1
Verify dbowner has the trailing period

This is the single most common cause of zero-count extracts. The package builds source queries by string-concatenating dbowner in front of each JDE table name. Open any data-flow task (e.g. Companies → Copy F0006) and inspect its qryF0006 variable — the Expression property reads:

"Select MCCO, MCMCU, … FROM " + @[User::dbowner] + "F0006 Where MCMCU != ' ' "

If dbowner is proddta., the resolved query is SELECT … FROM proddta.F0006 — correct. If dbowner is proddta (no period), the resolved query becomes SELECT … FROM proddtaF0006 — an object that doesn’t exist. SSIS’s OLE DB Source treats “no matching rows” as success, so the package runs to completion with UpdateCount = 0 and no ErrorNum.

  1. Open the package in Visual Studio Community.
  2. Open the Variables pane (View → Other Windows → Variables).
  3. Find dbowner. The value must end with a period — e.g. proddta., not proddta. (The template default is dbo.; if you see that unchanged on a JDE customer, the variable was never configured.)
  4. If the period is missing or the schema is wrong, fix it, save the package (File → Save All), then redeploy (right-click project → Deploy).
Period present after fix? Re-run the job (or execute the package directly from SSISDB — see Check 4) and re-check the diagnostic.
Check 2
Confirm the JDE source actually has data in the configured schema

If dbowner is correct but counts are still zero, verify the schema actually contains rows.

  1. In SSMS, connect to the JDE source instance with the readonly login configured in the package (typically rapidrec / rapidrec).
  2. Run SELECT TOP 10 * FROM proddta.F4111 (replace proddta with whatever the customer’s schema actually is).
  3. If the query returns rows, the source is fine. If it errors with “invalid object name” or returns zero, you’re pointed at the wrong schema or the customer hasn’t populated their JDE tables yet.
  4. Coordinate with the customer DBA to confirm the correct schema name and that JDE production data exists.
Source schema correct and populated? Continue.
Check 3
Verify aaStartDateGr is reasonable

The aaStartDateGr variable bounds the date range the package pulls. If it’s set to a future date, no rows in the past will match, and counts will be zero with no error.

  • In the Variables pane, confirm aaStartDateGr is set at or before the customer’s fiscal-year start.
  • If unsure of the right value, ask the GSI DBA for the customer’s fiscal-year configuration before adjusting.
Start date sensible? Continue.
Check 4
Re-execute the package and re-check the diagnostic

After any fix, validate by executing the package directly from SSISDB (faster than waiting for the next scheduled job run):

  1. SSMS → Integration Services Catalogs → SSISDB → RapidReconciler → right-click RapidReconciler_Prod.dtsx → Execute.
  2. Wait for completion (review the report when prompted).
  3. Re-run the verification query: SELECT Capture, Step, Process, UpdateCount, ErrorNum FROM v_diagnostic4_b_to_c_log WHERE Step = 'data capture' ORDER BY Capture DESC.
  4. Confirm UpdateCount is non-zero for the major tables (cardex, item master, etc.) and ErrorNum is 0.
Counts non-zero? Initial load is healthy — continue to Step 11 verification and Step 12 VALC handoff.
Check 5
Still zero counts

If counts remain zero after fixing dbowner, verifying the source schema, and confirming the start date, escalate to the GSI DBA with:

  • The current values of all package variables (dbowner, aaStartDateGr, DecExtCost, DecUnitCost, DecQTY, DecQtyCX).
  • The JDE source connection details (server, schema, login).
  • The diagnostic query output from Check 4.
  • The customer’s JDE version — non-standard table layouts are a known cause and need DBA review.
Escalated? Do not proceed to Step 12 (VALC handoff) until the diagnostic shows healthy row counts.
