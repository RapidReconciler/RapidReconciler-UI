Post-deployment JDE import fails — rruser login error

Customer-side admin deployed RR to production successfully but the JDE import errors out a few minutes in with an rruser authentication failure. The first email-based response (“update your connection passwords”) is correct in principle but often points at the wrong layer — RR packages can have hidden rruser references inside Data Flow sources, Execute SQL Tasks, and package-internal connection managers that the customer’s deployment-time UI doesn’t expose. Updating values in Visual Studio is not sufficient by itself; redeployment and execution validation are required.

What you’re seeing

  • Customer-side admin completed a successful production deployment.
  • The JDE import task errors out a few minutes in with an rruser authentication failure.
  • Error references the SSIS package and a login failure for rruser.
  • Customer often says they updated the rruser password but it didn’t fix it.
  • The RapidReconciler_Prod SQL Agent job may already be disabled while troubleshooting.

Walk these checks

Check 1
First-pass: confirm deployment-time connection passwords

The most common fix — have the customer update their rruser connection passwords on the production deployment. This works for a fraction of cases without needing a live session.

  1. Reply recommending they update the rruser connection passwords on the production deployment.
  2. Set the expectation: if it doesn’t resolve within 24 hours, schedule a Teams session for Checks 2–5.
Resolved within 24h? Stop here. Otherwise, continue to Check 2 to set up the Teams session.
Check 2
Pre-Teams-session prep

Confirm everything is in place before the call — missing access or credentials wastes the customer’s time on the call.

  • Customer admin has access to the RapidReconciler SSIS solution (Visual Studio), production environment, and the SQL Server (e.g., prd.mgs.com).
  • Updated rruser credentials / password are in hand.
  • The RapidReconciler_Prod SQL Agent job is disabled while troubleshooting — so it doesn’t fire mid-fix and confuse the diagnosis.
  • You know which connections to look for: typically JDESource and any connection using rruser.
All in place? Continue to Check 3 on the call.
Check 3
Find and update ALL rruser connections in the package
Key lesson: RR packages can include hidden credential dependencies. Updating the primary config (e.g., jderr) is not enough — the package may also reference rruser in Data Flow sources, Execute SQL Tasks, or a package-internal connection manager. Miss any one of these and the import keeps failing the same way.

On the call, walk the customer through each layer:

  1. Open the SSIS solution in Visual Studio. Open the relevant .dtsx package (the JDE Import package).
  2. In the Connection Managers pane, locate every connection using rruser: JDESource, any database connection tied to the RR database, anything else with rruser in the credential.
  3. For each connection: right-click → Edit. Update server (e.g., prd.mgs.com), database (e.g., proddta for JDE), authentication type, and the username / password.
  4. Click Test Connection — this must succeed before moving on. If it fails, the credential or network is still wrong; don’t deploy yet.
  5. Walk through each Data Flow source / destination and each Execute SQL Task. Look for any other reference to rruser or legacy / test credentials. Update them too.
All connections testing successfully in Visual Studio? Continue to Check 4.
Check 4
Redeploy and validate the deployment landed

Visual Studio changes are not live until the project is redeployed to SSISDB. Validate the new package actually arrived — deployment can silently fail.

  1. Right-click the project → Deploy. Deploy to the target SQL Server / SSISDB. Complete the deployment wizard.
  2. Open SSMS → Integration Services Catalogs → SSISDB → find the Rapid Reconciler project.
  3. Confirm the package’s timestamp reflects the deployment you just completed. If the timestamp is older, the deploy didn’t land — redeploy.
Timestamp matches the just-completed deploy? Continue to Check 5.
Check 5
Execute, validate, and re-enable the schedule

Two execution paths — the SSMS direct execute is preferred during troubleshooting (faster feedback); the SQL Agent path matches what production will actually do.

  1. SSMS direct execute (preferred during troubleshooting): in Object Explorer, navigate to SSISDB → the package → right-click → Execute. Watch it complete with no auth errors.
  2. SQL Agent path (matches production behavior): SQL Server Agent → locate RapidReconciler_Prod → right-click → Start Job at Step. Watch it complete cleanly.
  3. Re-enable the RapidReconciler_Prod scheduled job (typically 6:30 AM).
  4. Confirm the next scheduled run completes successfully — check the next morning.

After the call, document which package and which connection managers needed updating. Future credential rotations at the same customer will likely need the same set of updates.

Both manual executions clean and the next scheduled run completes? Stop here.
