Post-migration login fails — app server can’t reach the database

After cutover (especially to AWS or another cloud), users get login failures in RapidReconciler even though the database itself is reachable from outside. The most common cause is the network path between the new app server and the DB server — Security Groups, Network ACLs, OS-level firewalls, or a stale IP somewhere in the config. The architecture expects only outbound from the client; verify each layer in turn.

What you’re seeing

  • One or more users (often a specific named user the customer mentions first) report login failures in RapidReconciler post-cutover.
  • The DB itself is reachable from at least one external client — the DBA can connect via SSMS or similar.
  • Customer confirms outbound traffic is open from their network and the app server is not joined to their domain.
  • Error wording varies: connection timeout, login failed for user 'rruser', or a TCP / named-pipe transport error.

Walk these checks

Check 1
Confirm post-migration IPs in app-side config
  • Verify the application server is pointing at the correct new DB IP (e.g., the new AWS internal IP).
  • Verify the DB server IP itself matches what the migration plan specified.
  • Search the app-server config for any stale hardcoded IP from the old environment in connection strings, app config files, or environment variables.
All IPs current? Continue.
Check 2
Test app server → DB network reachability
  • From the app server, run Test-NetConnection <db-host> -Port 1433 (PowerShell) or telnet <db-host> 1433.
  • If a hostname is used instead of an IP, confirm DNS resolution from the app server.
  • Confirm outbound traffic on the SQL port is permitted from the app server.
TCP open? Continue. Otherwise, the issue is at the network layer (Check 3).
Check 3
AWS Security Groups, NACLs, OS firewalls, VPC routing
  • The DB server’s AWS Security Group inbound rules permit SQL traffic (port 1433) from the app server’s IP or CIDR.
  • Network ACLs on the DB subnet permit the same.
  • Windows or Linux OS-level firewall on the DB server has the same allow rule.
  • Subnet / VPC routing between app server and DB server is in place.

Architecture note: RapidReconciler expects the customer side only requires outbound connectivity — no inbound port exposure on the client side. If a customer’s network admin is asking which ports to open inbound on their network, that’s a sign of a misconception worth correcting on the call.

Network path clear? Continue.
Check 4
Restart RR services, then read the failure wording
  • After any IP, connection-string, or config change on the app server, restart RapidReconciler Agent and RapidReconciler Services so the new values are picked up.
  • Check the application log for the failure mode wording — this tells you which layer is still broken:
    • connection timeout → network path (back to Check 2 / 3).
    • login failed for user 'rruser' → SQL auth (credentials, login enabled, Mixed Mode).
    • TCP / named-pipe transport error → SQL Server isn’t listening or the protocol is disabled.
  • Check SQL Server logs on the DB server for failed-login entries timestamped to your test attempts.
Wording points to a specific layer? Address it. Otherwise, continue to Check 5.
Check 5
End-to-end login test
  • Login as the affected user (and a known-good test account if available).
  • Confirm the UI loads after login and data renders without timeout.
  • If login still fails after Checks 1–4, capture: app log excerpt with the error, SQL Server log entries at the same timestamp, and the connection string in use. Escalate to network admin and DBA together — this class of failure usually needs both perspectives.
Login works end-to-end? Stop here.
