Orientation

What the role is — and what it isn’t

You take customers live. The RR DBA hands you a database that’s fully loaded and verified; you turn that into a customer who can log in and use RapidReconciler. After go-live, you’re also the front line for that customer’s ongoing needs — license renewals, additional databases, password-policy changes, deploys, and most troubleshooting. Two modes, one role.

What you own

  • The 13-step VALC install procedure documented in Installing a Client in VALC. Every install runs this sequence.
  • The customer go-live moment. Step 13 ends with a handoff email containing the customer’s login URL and initial GSI Admin credentials. That email is the customer’s first usable touchpoint with the live product.
  • Most live-customer change requests. Renewals, additional licenses, additional databases, password policy, agent updates, migrations — all routed to you by Helpdesk Tech.
  • The GSI network/Cloudflare side of every customer setup — DNS A-records, the wildcard *.getgsi.com certificate, Cloudflare cutovers during migrations.
  • First-line troubleshooting via agent log files. You escalate to RR DBA when the diagnosis points at the database or SSIS side, and to RR DB Developer when it points at schema or product code.

What you don’t own

  • The database install itself — that’s RR DBA. You start work only after their database-ready notification arrives.
  • Customer-side user administration (adding users, password resets, permission changes). The customer’s own designated RapidReconciler administrator owns this; redirect when these requests come to you.
  • Schema changes, install-zip authoring, SSIS package source — that’s RR DB Developer.
  • Mailbox triage — that’s the Helpdesk Tech.

This guide doesn’t cover specifics like how installs and live-customer work interleave day-to-day, who in GSI I/T owns the DNS/cert request workflow on their end, after-hours coverage for live-customer issues, or how customer load is divided between multiple Network Techs. Confirm those with your manager before your first shift.

Two operational modes

Most roles in this knowledge base have one shape. Yours has two, and they feel different from each other day-to-day. Knowing which mode you’re in helps you use the right mental frame.

Mode 1: New install

Triggered by RR DBA’s database-ready notification arriving in rrsupport@getgsi.com. You run the 13-step VALC install procedure end-to-end. Most of the work is async (Phases 1, 2, and 4); Phase 3 has a live working session with Customer I/T. The install completes when the customer logs in for the first time. Each new install is bounded and has a defined endpoint — the Customer Live terminator.

Mode 2: Live-customer support

Triggered by anything from an existing customer reaching the support mailbox — renewals, change requests, password-policy activations, additional databases, migrations, troubleshooting. The work is unbounded and ongoing; a customer is “done” only when they cancel. Most live-customer requests are short and tactical, but server migrations and post-go-live troubleshooting can be days of focused work.

How the modes interact

On any given day you’ll do some of both. The install is sequential and time-bounded; the live work is interrupt-driven. Most teams handle the tension by reserving specific blocks for install meetings and triaging live-customer requests in between. How your team does it specifically is something to discuss with your manager.

Part 1 · New-Install Procedure

The 13 steps in four phases

The full VALC install groups into four phases, mirroring the RR DBA install structure. The phase names are deliberately the same — both roles run the same shape of work, just on different sides of the seam.

  • Phase 1 · Intake (Steps 1–5) — async, before any meeting. Receive both trigger emails, create the VALC client record, configure the GSI Admin user and modules, submit DNS/cert request.
  • Phase 2 · Setup (Steps 6–7) — scheduling and meeting prep. Short phase, but a real boundary — work moves from solo to live.
  • Phase 3 · Configure (Steps 8–10) — the live working session. Agent install (shared with Customer I/T), SQL connection validation (gate), database registration.
  • Phase 4 · Verify-and-Handoff (Steps 11–13) — databases-online check, company licensing, customer handoff email. Customer Live terminator.

Two important gates between phases:

  • Between Intake and Setup: the DNS A-record needs propagation time before the agent install. Don’t schedule the install meeting before the cert & DNS request is in flight.
  • Within Configure (at Step 9): if SQL Server connectivity fails at the agent install, you cannot proceed to database registration. The work loops back to Customer I/T for connectivity troubleshooting before resuming.
01
Phase 1
Intake
Steps 1–5

Async, before any meeting. Two trigger emails arrive (one originally from Customer I/T, one from RR DBA), and you do all the setup work in VALC and on the network side that doesn’t need the customer in real time.

Step 1

Receive customer pre-install submission email

Async

The customer’s pre-install submission email — the same one that started the RR DBA’s install workflow days earlier — comes to you via the Helpdesk Tech when the database-ready notification arrives. You need both: the original submission has the topology, server IPs, and JDE details; the database-ready notification adds non-default port and password if any.

Key points
  • Don’t start work without both emails. If only the database-ready notification has arrived, ask the Helpdesk Tech to forward the pre-install submission too — the topology and server IPs are mandatory inputs to Step 2.
  • The pre-install submission has a long structured body: TOPOLOGY SELECTED, SERVER NAMES & INTERNAL IPs, JDE CONNECTION DETAILS, SUBMITTED BY. The SUBMITTED BY contact is who you’ll schedule the agent-install working session with at Step 6.
  • The database-ready notification has subject RapidReconciler Database Ready — {Customer} — VALC Install Can Begin. Look for non-default SQL port or non-default rruser password — both are noted only when non-default.
Step 2

Create the VALC client record

Async

On the VALC Clients page, create the new client. Set Client Name, License Start Date, License End Date, Version, JDE Version, HTTPS = True, Maximum Companies Allowed (from sales contract), and Agent Protocol = SSL.

Where the values come from
  • Maximum Companies Allowed — from the sales contract. This is the number of JDE companies the customer purchased; you’ll license specific company numbers at Step 12.
  • License Start/End Dates — from the sales contract. End date drives lockout enforcement; renewals later in the customer’s lifecycle update this field.
  • Version & JDE Version — from the customer’s pre-install submission.
  • Agent Protocol = SSL — always SSL. Don’t toggle this.
Step 3

Create the initial GSI Admin user

Async

On the new client’s User Accounts tab, create a placeholder admin user used during agent install and later handed off to the customer’s designated RR administrator at Step 13.

Field values
  • Full Name: GSI Admin
  • Email: gsiadmin@{customerdomain} — use the customer’s actual domain so the address looks plausible to them later.
  • Password: a temporary one you note for the handoff email at Step 13. The customer will be required to change at first login.

This user is functionally a bootstrap. After Step 13, the customer’s designated RapidReconciler administrator takes over user management and the GSI Admin user becomes a dormant fallback.

Step 4

Configure modules and additional fields

Async

On the Client Details tab, fill the Agent External and Internal IP addresses, the Domain URL, and enable the module tabs the customer should see at launch.

Values
  • Agent External/Internal IPs — from the pre-install submission’s SERVER NAMES & INTERNAL IPs block.
  • Domain URL: rrprod-{customername}.getgsi.com — the customer-facing URL they’ll log in to. Generate it from a sanitised customer name.
  • Modules to enable initially: Inventory and Admin tabs to start. Additional modules get enabled later as live-customer changes when the customer purchases them.
Step 5

Submit DNS A-record & certificate request

Async · off-team

Submit a DNS A-record addition for the customer’s domain (rrprod-{customername}.getgsi.com) pointing to the agent internal IP. The wildcard *.getgsi.com certificate then covers the customer’s URL automatically — no per-customer cert provisioning needed.

Critical timing
  • DNS propagation takes time. Allow propagation before scheduling the agent install meeting at Step 6 — if DNS hasn’t propagated, the agent install at Step 8 will fail at the URL-resolution step.
  • The wildcard cert is managed centrally; you don’t generate or apply a per-customer certificate. See SSL Certificate & DNS Management for the full reference on certificate renewal and DNS management.
Draft — manager to validate

The exact submission channel for the DNS request (ticket queue, email to GSI I/T, Cloudflare access?) and propagation time expectations aren’t covered in source documentation. Confirm with your manager before your first install.

02
Phase 2
Setup
Steps 6–7

The bridge between solo VALC work and the live working session. Short phase — just scheduling and the moment the customer joins — but a real boundary in the rhythm of the install.

Step 6

Schedule meeting with Customer I/T

Async

Schedule a working session with the customer’s I/T contact (the SUBMITTED BY person from the pre-install submission) to do the agent install on their application server. There’s no detailed step documentation for this — it’s essentially calendar coordination.

Notes
  • The agent install is a shared task: you drive the VALC side; Customer I/T drives the customer-server side. Both must be on the call together.
  • Step 5’s DNS A-record needs to have propagated by meeting time. If you scheduled too tightly, the install will fail at Step 8.
  • Allow ~1 hour for the meeting. Most of it is the agent install itself plus the SQL Server validation that follows.
Step 7

Customer I/T joins the working session

Live

Customer I/T joins the meeting from their side. Confirm they have local administrator credentials on the customer’s application server — the agent installer needs them in Step 8. Confirm screen-share or RDP access so you can see what they’re doing.

Notes
  • This step has no separate procedure detail in the source — it’s the moment the live working session begins.
  • If the customer is missing local admin credentials, pause now rather than starting Step 8 and hitting it mid-install. Most customers can produce these in a few minutes; the alternative is rescheduling.
03
Phase 3
Configure
Steps 8–10

The live working session. You’re driving in VALC; Customer I/T is driving on their application server; both of you are on the same call making each other’s pieces fit together. Three steps — one shared, one gate, one solo.

Step 8

Install agent on customer server

Shared with Customer I/T

The longest step in the procedure. Walk Customer I/T through downloading the 64-bit agent from https://rapidreconciler.getgsi.com, running the installer with local administrator credentials on the application server, and confirming Installation Complete in the browser.

Sequence
  • Download: the customer downloads the 64-bit installer from the public RapidReconciler URL. Confirm with them which architecture — 64-bit is standard.
  • Run: the customer runs the installer with local admin credentials on their application server. Watch the install via screen-share.
  • SQL connection prompt: mid-install, the agent will prompt for SQL Server details. You provide these (database server internal IP, port, rruser credentials — using the values from the database-ready notification at Step 1).
  • Confirmation: the customer sees Installation Complete in the browser. The VALC Clients grid updates to show Connected with Services and Agent versions populated.
Step 9

Validate SQL Server connection

Gate · failure loops to Customer I/T

Confirm the agent has a working SQL Server connection. The validation is what the agent does immediately post-install — if it succeeds, the VALC Clients grid shows Connected with Services and Agent versions populated. If it fails, the agent shows Disconnected and you can’t proceed.

If validation fails
  • Don’t proceed to Step 10. Database registration requires a working agent connection.
  • Common causes: firewall closing port 1433 (or the customer’s custom SQL port), rruser credentials wrong (check the database-ready notification for non-default password), DNS propagation incomplete (Step 5 timing problem).
  • Loop back to Customer I/T for connectivity troubleshooting; you stay on the call to re-validate once they’ve made changes. The flowchart explicitly shows this loop — it’s expected, not exceptional.
Step 10

Register database in VALC

Live

On the new client’s Databases tab, create one record per registered database with all required fields. Repeat for each Test/Dev/Train database the customer has — the DBA prepares additional databases via backup/restore from Production.

Required fields per database
  • Address: database server internal IP.
  • Port: 1433 unless the database-ready notification said otherwise.
  • Username / Password: rruser and its password (default unless specified).
  • Name: the database name (typically RapidReconciler_Prod for Production).
  • Category: Production / Test / etc.
  • Job Name: RapidReconciler_Prod (the SQL Agent job created by RR DBA at install Step 7).
  • Max Memory: typically the customer’s server-tier default; check the customer’s server spec.
04
Phase 4
Verify-and-Handoff
Steps 11–13

Verification and customer handoff. You confirm everything is working, license the customer’s purchased company numbers, and send the email that gives them their first login. Customer Live happens at the end of this phase.

Step 11

Verify databases show Online

Live

On the Databases tab, confirm every registered database row shows Online (green). Investigate Unknown or Stopped states using the agent log files at C:\Program Files\Rapid Reconciler\logs\ on the customer’s application server.

Status meanings
  • Online (green) — agent has the database connection and is reporting healthy. This is the goal state.
  • Unknown — agent hasn’t reported on this database yet. Wait a moment and re-check; if persistent, check log files.
  • Stopped — agent reports an error on the database connection. Check log files.
Step 12

License company numbers

Live

On the new client’s Companies tab, check the boxes for each 5-digit JDE company number per the sales contract. Repeat for each registered database (Test/Dev/Train inherit the same licensing).

Notes
  • Source of truth: the sales contract names exact 5-digit company numbers. Don’t guess or extrapolate from what looks like a pattern in the database — there may be additional companies in the customer’s JDE that aren’t licensed.
  • Maximum Companies Allowed (set at Step 2) caps how many you can license. If the contract names more companies than the Maximum, that’s a sales-contract / VALC mismatch — pause and reconcile before proceeding.
Step 13

Customer handoff — the go-live email

Async

Send the customer’s I/T contact and designated RapidReconciler administrator a handoff email containing: login URL, initial GSI Admin credentials with first-login password reset, pointer to the User Accounts tab for ongoing user management, password-policy status, and the support contact (rrsupport@getgsi.com).

Notes
  • Confirm first customer login — verify the customer has logged in at least once before considering the install closed. The Customer Live terminator on the tech flowchart implies a successful first login, not just a sent handoff email.
  • The customer’s designated RapidReconciler administrator is named in the discovery call participants list. They’re the post-handoff owner of user management on the customer side — the GSI Admin user is now their fallback, not their primary tool.
Part 2 · Live-Customer Mode

What live-customer work looks like

After a customer goes live they’re yours. Most of what they need is short and tactical — renewals, license adjustments, password changes, deploys. Some of it is bigger — server migrations, post-go-live troubleshooting. All of it routes to you via the Helpdesk Tech at rrsupport@getgsi.com.

This section is a tour of the recurring task types — what each is, when you’d reach for it, and where the canonical reference lives. The references are the source of truth for procedure detail; this Start Here just orients you to existence and purpose.

Where these requests come from

The Helpdesk Tech reads inbound emails to rrsupport@getgsi.com, classifies them, and forwards anything live-customer-related to you. If something arrives directly that should have come through the Helpdesk Tech, route it back to them — the routing layer exists for a reason.

License renewals

Annual

When a customer’s renewal contract is signed, Sales emails the support mailbox and the Helpdesk Tech forwards it to you. Update the License End Date in VALC on the client’s record. If the renewal also adds company licenses or modules, that’s a separate change but goes to the same client record.

When you’ll reach for it: annually per customer, sometimes batched at quarter-ends. Missing a License End Date update is a customer-facing outage — users get locked out at the old expiry — so don’t let these sit.

Live-customer change requests

Frequent

The catch-all category: adding company licenses, enabling additional module tabs, updating database connection details (IP, port, credentials), deactivating a client (Client Active toggle on cancellation). Most of these are one-tab edits in VALC against the customer’s client record.

When you’ll reach for it: the bulk of inbound from existing customers. The Common Requests reference is the ownership matrix — some requests are GSI tasks, others get redirected to the customer’s own RR administrator. Read it once early; it’ll save you from over-helping on user-administration questions that aren’t yours.

Reference: Common Requests on the VALC client-install page.

Password policy toggle

Per request

Customers may request the complex password policy be enabled (8-character minimum, complexity rules, 90-day expiry, 10-password history). Per-customer toggle on the Client Details tab in VALC. Once enabled, every existing user is forced to reset at next login.

When you’ll reach for it: when an inbound asks for activation, with three required pieces of information — company name, the customer’s RR administrator name, and a requested activation date. If any are missing, ask before flipping the toggle.

Reference: Password Policy.

Adding a Test, Dev, or Training database

Occasional

Existing customers grow into the product and ask to add additional databases. Coordinate with the customer’s DBA to back up RapidReconciler_Prod and restore it under the new name on their SQL Server — the install scripts can’t be re-run because they’re hard-coded for Prod. Then register the new database in VALC’s Databases tab.

When you’ll reach for it: usually as customers mature. The pattern is: customer-side DBA does the SQL work, you do the VALC registration. If the customer’s DBA is unsure about the backup/restore approach, that’s the only thing you walk them through.

Manage Deploys (Services / Agent updates)

Periodic

Push Services and Agent version updates to customer environments via the Manage Deploys feature in VALC. RR DB Developer cuts new versions; you push them out to the customer base.

When you’ll reach for it: when a new Services or Agent version is staged and the rollout is on your plate. Same multi-customer rollout discipline as RR DB Developer’s SQL Scripts work applies — small batches, monitor results, investigate failures before continuing.

Posting Message Users broadcasts

Periodic

VALC’s Message Users page lets you post system-wide notices for maintenance windows, deploys, or version changes. Messages are seen by all customers, so wording stays customer-neutral — no internal jargon, no customer-specific context.

When you’ll reach for it: ahead of any operational event customers should know about — maintenance windows, agent or services deployments, advance notice of certificate renewal cutovers. See the discipline section on broadcast wording.

Server migrations

Big project

Existing customers occasionally move to new infrastructure — new database server, new application server, new topology. The migration is days of focused work: backup/restore on the SQL side, SSIS package re-deployment, SQL Agent job re-creation, agent re-install on the new application server, VALC update, Cloudflare DNS cutover, post-migration verification.

When you’ll reach for it: infrequently per customer, but each one is real work. Treat every migration as a mini-install — same discipline, same gates.

Reference: Server Migration.

SSL & DNS management

Periodic

The wildcard *.getgsi.com certificate covers every customer URL; renewing it is a periodic task that touches all customers at once. Plus per-customer DNS A-record changes during migrations or domain changes. Cloudflare is the DNS host.

When you’ll reach for it: certificate renewals (the doc has a three-step procedure), DNS A-record changes during migrations, troubleshooting DNS-related Disconnected agents.

Reference: SSL Certificate & DNS Management.

First-line troubleshooting

As needed

When customers report issues, you’re the first read. Agent log files at C:\Program Files\Rapid Reconciler\logs\ on the customer’s application server are the primary diagnostic source — the daily files are rr-valc-agent_YYYYMMDD.out (info) and .err (errors). Common patterns: Disconnected clients, Unknown/Stopped database rows, agent installs hanging at Validating Data, Setup Step not advancing, post-Cloudflare data-not-flowing issues.

When you’ll reach for it: any inbound that describes a customer problem — the Helpdesk Tech will route it to you. The Troubleshooting Hub covers the diagnostic patterns you’ll see most often. Escalate to RR DBA when the diagnosis points at the database or SSIS side, and to RR Developer when it points at schema or product code.

Reference: Troubleshooting Hub.

How escalations arrive from Helpdesk

Helpdesk Tech can hand off through two channels, both of which include a pre-filled escalation summary with the symptom, what they tried at Tier 1, and the customer’s context:

  • JSM ticket in the RapidReconciler Support project (issue type Support) — for issues needing formal tracking, cross-shift handoff, or SLA visibility. Watch the project queue. The customer’s original email is in the description (or attached, if it had screenshots).
  • Reply-all on the customer’s email thread — for fast-moving conversations or phone-driven cases. The Helpdesk Tech’s escalation note is inline; the original customer message is in the thread above.

If you receive a JSM ticket and decide it’s actually database / SSIS / schema work, reassign it; don’t close-and-recreate. The RapidReconciler Support project is the entry point regardless of where it ends up.

Part 3 · Operational Discipline

The six rules that keep both modes safe

The procedure works when the discipline below is followed. Some of these are install-mode specific; others apply across both modes. Internalising them now means recognising the failure modes when you encounter them later.

Rule 1

Pair both trigger emails before starting an install

The VALC install needs two emails to be actionable: the customer’s original pre-install submission (with topology, server IPs, JDE details) and RR DBA’s database-ready notification (with non-default port and password if any).

If only one has arrived, ask the Helpdesk Tech to forward the other before starting. Working from the database-ready notification alone means missing topology and server IPs — mandatory inputs to Steps 2 and 4. Working from the pre-install submission alone means missing the non-default port or password and discovering at Step 8 that your config is wrong.

Rule 2

Submit the DNS request early, schedule the meeting late

Step 5 (DNS A-record submission) and Step 6 (scheduling the agent install meeting) have a propagation gap between them. If you do them too close together, DNS hasn’t propagated by meeting time and Step 8 fails at URL resolution — you reschedule with the customer, which is avoidable.

Submit the DNS request as early as Phase 1 lets you (immediately after Step 4 is fine), then schedule the meeting for far enough out that propagation is comfortable. If you must schedule tightly, verify DNS propagation before the meeting starts.

Rule 3

Honor the Step 9 SQL connection gate

If SQL Server connectivity fails at Step 9, do not proceed to database registration. The agent in a Disconnected state can’t serve the database it’s registered against, so registering anyway just creates a broken record that you’ll have to fix later.

The procedure explicitly shows the loop back to Customer I/T for connectivity troubleshooting — firewall, credentials, DNS — before resuming. This is the single most common pause point in a VALC install. Don’t treat it as a failure; treat it as expected.

Rule 4

Redirect user-administration requests to the customer’s own RR administrator

Customers will email asking you to add a user, reset a password, change someone’s permissions, or unlock an account. None of that is your work — it’s the customer’s designated RapidReconciler administrator’s job. Redirect with a polite reply pointing them at their own admin.

Why this matters: GSI doesn’t handle user administration on the customer’s behalf unless explicitly contracted to. Doing it occasionally as a favor sets the wrong precedent and creates support work that scales linearly with the customer base. The Common Requests reference is the ownership matrix; lean on it.

Rule 5

Broadcasts are global; wording stays customer-neutral

The Message Users page broadcasts to every customer simultaneously. There’s no per-customer targeting at this layer. That means:

  • No internal jargon — customers don’t know what a “deploy” or “agent push” is. Use plain operational language.
  • No customer-specific context — never include a specific customer’s name, environment, or issue.
  • No internal blame — even a soft “due to a vendor issue” reads worse than a neutral “during scheduled maintenance.”
  • Read it twice before posting. Once it’s out, every customer has seen it.
Rule 6

Logs first, escalate second

When a customer reports an issue or a database row shows Stopped, your first move is the agent log files at C:\Program Files\Rapid Reconciler\logs\. Most issues are diagnosable from .out and .err files alone — connectivity errors, credential failures, certificate issues all leave clear traces.

Escalation rules:

  • To RR DBA when the failure points at the database or SSIS side — load failures, schema errors, SQL Agent job problems.
  • To RR DB Developer when the failure points at the schema or product code itself — bug reports, unexpected data behavior reproducible across customers.
  • Don’t escalate without logs. “The customer says it’s broken” without logs costs the receiving role 30 minutes asking for them. Pull the logs first; escalate with them attached.
Wrap-up

What to do next

You’ve walked the procedure, the live-customer mode, and the discipline. Three suggested next steps before your first install or first live-customer ticket:

  1. Read the full Installing a Client in VALC procedure end-to-end. This Start Here is wayfinding; the procedure doc is your reference. Knowing it forwards and backwards before you touch a customer install is the difference between confidence and improvisation.
  2. Read the four references the live-customer section points at: Common Requests, Password Policy, Server Migration, Troubleshooting. None of them is long; together they cover most of what shows up in your inbox post-go-live.
  3. Shadow a senior Network Tech on at least one install before driving one yourself. The 13 steps are documented; the cadence with Customer I/T, the rhythm of the live working session, the small judgments at each phase boundary — those aren’t in any document. Watching once is worth reading three times.
