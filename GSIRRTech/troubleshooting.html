Tier 2 · Network Tech Install & Agent

Agent install hangs at "Validating Data"

Tier-1 difficulty: Hard — Debug app-server-to-DB connectivity, SQL credentials, firewall.
Symptom: During agent install, the browser displays "Validating Data" indefinitely instead of progressing to the SQL Server connection prompt or "Deploying".
Likely causes
  • Application server cannot reach the database server on the configured SQL port
  • SQL credentials provided in the connection prompt are incorrect
  • RapidReconciler_Prod database is not yet created on SQL Server, or schema is incomplete
  • Firewall on the database server is blocking inbound traffic on port 1433 (or the customer's configured port)
  • SQL Server Browser service is not running on the database server
Troubleshooting steps
  1. From the application server, attempt a manual connection to the database server on the configured port to verify reachability.
  2. Confirm the database, the rruser login, and SQL Authentication mode are all in place per the pre-install database setup.
  3. If the prompt is reached but still fails, double-check the values being entered match the customer's SQL configuration exactly.
  4. Check Windows Event Viewer on the application server for RapidReconciler-related errors.
  5. Check today's agent .out and .err log files (see Diagnostic Tools).
  6. If the install needs to be restarted, ensure Setup Step is set back to "Created" in VALC before retrying.
In coordination with Customer I/T (server access)

Agent download prompt does not appear after login

Tier-1 difficulty: Easy — Single VALC field to verify: Setup Step = 'Created'.
Symptom: Customer logs in to rapidreconciler.getgsi.com with GSIADMIN credentials but no agent download prompt is shown.
Likely causes
  • Setup Step on the Client Details tab is not set to "Created"
  • Login was performed with the wrong credentials
  • Browser cache is showing a stale view
Troubleshooting steps
  1. In VALC, open Manage Client → Client Details tab for the customer.
  2. Verify Setup Step is set to "Created". If not, change it to "Created" and save.
  3. Have the customer hard-refresh the browser (Ctrl+F5 / Cmd+Shift+R) and log in again with the GSIADMIN credentials.
  4. If the prompt still does not appear, check that no other browser session has the customer logged in elsewhere.

Tier 1 · Helpdesk Tech Install & Agent

Clients grid row not updating to "Connected" after install

Tier-1 difficulty: Easy — Read the Clients grid; usually a refresh or re-login resolves it.
Symptom: Agent install reported "Installation Complete", but on the VALC Clients grid the row's Status badge stays at the previous value instead of switching to Connected.
Likely causes
  • DNS for the customer's .getgsi.com subdomain has not yet propagated after Cloudflare update
  • RR Agent and RR Services are installed but one or both services are not running
  • VALC has not refreshed — the underlying agent has connected but the grid row hasn't updated yet
  • Agent External IP on the Client Details tab does not match the customer's actual external IP
Troubleshooting steps
  1. Refresh the Clients grid in your browser (the page does not auto-refresh).
  2. Wait a few minutes if the install just completed.
  3. From the customer's application server, try resolving the customer's .getgsi.com URL — if DNS still returns the old IP, the Cloudflare update hasn't propagated yet.
  4. On the application server, confirm both "RapidReconciler Agent" and "RapidReconciler Services" are running in the Windows Services console.
  5. Verify the Agent External IP Address on the Client Details tab matches the customer's actual external IP.
  6. If the row still hasn't updated, check today's agent .out log file for connection errors back to VALC.

Tier 1 · Helpdesk Tech Application & Data

Inventory Validation light is red — variance / out of balance

Tier-1 difficulty: Easy — Walk customer through the popup, ask if prior-period batches are posted, escalate the Excel if needed.
Symptom: Customer reports a variance, "out of balance" message, or that the Inventory Validation indicator at the top of the Reconciliation page has turned red. The Variance Calculation panel typically shows a Carry Forward amount that matches the Unreconciled Variance. Customer phrasings often include "the red light came on," "this just popped up," or "we have a $X variance for [period]."
Likely causes
  • One or more GL batches in the prior period are unposted — by far the most common cause; the product itself surfaces this as the most likely scenario in the indicator's tooltip.
  • A backdated transaction has been posted into a now-closed prior period — affects the carry-forward calculation. Identifiable from the Excel report exported by clicking the red indicator. Escalate the Excel to Network Tech; they forward to RR Developer for transaction-level review if needed.
  • Other data-capture issue at period close — rare, no obvious customer-side cause. Same escalation path as above.
Troubleshooting steps
  1. Acknowledge the variance with the customer. Note the period (e.g. April 2024) and the variance amount. Ask for a screenshot of the Reconciliation page including the Variance Calculation panel if not already attached.
  2. Read the indicator’s own tooltip back to the customer. Hovering over the red Inventory Validation indicator shows a popup that says: “This indicator is red if the roll forward from the prior period fails. This is most likely due to one or more un-posted GL batches. If all batches in the prior period have been posted, please contact us at rrsupport@getgsi.com for assistance.” Many customers don’t notice this message — pointing them at it sometimes resolves the case before any further troubleshooting.
  3. Ask the customer to verify whether all GL batches in the prior period have been posted. This is a customer-side check in their JDE/ERP, not in RapidReconciler. The product’s own tooltip says this is the most likely cause and is correct in our experience.
  4. If unposted batches are found: have the customer post the batches, then refresh the Reconciliation page and confirm the Inventory Validation indicator goes green. If green, resolved — close the loop with the customer. Roughly 80% of cases end here.
  5. If the customer confirms all prior-period batches are already posted: ask them to click the red Inventory Validation indicator. This exports an Excel report of the carry-forward calculation. Have them email the Excel to rrsupport@getgsi.com (or attach it to the existing support thread).
  6. Escalate to Network Tech (Tier 2). Do not analyze the Excel at Tier 1 — the analysis is product/data work and a Network Tech is the right first read on whether to forward to RR Developer (Tier 3) for transaction-level review. Use the Create JSM ticket button below; the customer’s email and the Excel attachment travel with the escalation. Network Tech routes onward as needed.
  7. If the customer reports no obvious cause from their side either (no unposted batches, no recent backdated entries they can identify), escalate via the same path. The Excel is still the artifact the next-tier tech needs — ask the customer to send it regardless.
Tier 1 · Helpdesk Tech Application & Data

System Status light is red — job crashed

Tier-1 difficulty: Easy — Walk the customer through clicking the indicator, collect the Excel export, attach to a JSM ticket, and escalate.
Symptom: Customer reports the System Status indicator at the top of the Reconciliation page has turned red. This signals that the underlying job has crashed. Customer phrasings often include "system status is red," "the red light is on," "the job failed," or "something crashed."
Likely causes
  • Disk space issue on the application or database server — the job ran out of room while writing intermediate or output data and aborted. Common after long-running periods without housekeeping.
  • Data processing error — the job hit a record or condition it couldn't handle (bad data shape, unexpected null, type mismatch, calculation overflow, etc.) and threw before completing.
  • Other backend failure — rare; same escalation path. Tier 2 reads the Excel and routes onward if needed.
Troubleshooting steps
  1. Acknowledge the alert with the customer. Note the date/time the indicator turned red, the company/business unit they were viewing, and ask for a screenshot of the Reconciliation page header (showing the System Status indicator) if not already attached.
  2. Ask the customer to click the red System Status indicator. This exports an Excel report containing the crash details. Have them email the Excel to rrsupport@getgsi.com (or attach it to the existing support thread).
  3. Once the Excel arrives, create a JSM ticket for the case and attach the Excel file to it. Do not analyze the Excel at Tier 1 — Tier 2 (Network Tech) is the first read on whether it’s a disk-space issue, a data error, or something else.
  4. Escalate the JSM ticket to Tier 2 (Network Tech). Use the Create JSM ticket button below; the customer’s email and the Excel attachment travel with the escalation. Network Tech routes onward to RR Developer (Tier 3) if transaction-level review is needed.
  5. If the customer cannot get the Excel to export (e.g., clicking the indicator does nothing), capture a screenshot of the indicator and any browser console error, and escalate without the Excel — note in the ticket that the export step failed so Tier 2 knows to start from the agent logs instead.
Tier 1 · Helpdesk Tech Database & SSIS

Database status shows "Unknown" or "Stopped"

Tier-1 difficulty: Easy — Read VALC Databases tab; signals what to check next.
Symptom: On the Databases tab, a row shows Unknown (orange) or Stopped (orange) instead of Online (green).
Likely causes
  • Connection details on the Edit Database popup are wrong — IP, port, username, password, or name don't match the actual SQL Server
  • RR Agent service on the customer's application server is stopped or has lost network access
  • Firewall on the database server is blocking the agent's connection on the configured port
  • The physical database doesn't exist on SQL Server (registration in VALC succeeded but the underlying database wasn't created or has been removed)
  • SQL Server Authentication is disabled or the rruser login was not created
Troubleshooting steps
  1. Open the Edit Database popup for the failing row in VALC and verify every field matches the actual database configuration.
  2. Confirm the RR Agent service is running on the customer's application server. If not, start it.
  3. Check today's agent .out log file on the application server for connection errors or stack traces.
  4. From the application server, attempt a manual connection to the database server on the configured port (e.g., telnet 10.x.x.x 1433) to confirm reachability.
  5. Confirm the customer's DBA has created the RapidReconciler_Prod database and the rruser SQL Authentication login.
  6. If everything looks correct and status remains failed, escalate to the GSI database administrator.
In coordination with RR Network Tech (VALC) and Customer I/T (server access)

Tier 1 · Helpdesk Tech Database & SSIS

Companies tab is empty after install

Tier-1 difficulty: Easy — Read VALC Companies tab; usually a config oversight.
Symptom: Database shows Online on the Databases tab, but selecting it from the dropdown on the Companies tab shows no company numbers to license.
Likely causes
  • The first sync from JDE has not yet completed — Online status confirms VALC can reach the database, but company numbers populate from JDE through the SSIS package's first run
  • The SQL Agent job is not running or is disabled on the database server
  • The SSIS package's source connection (to JDE) is misconfigured, even though the destination (to RapidReconciler_Prod) is working
  • The customer's JDE environment has no companies in the source tables that the SSIS package reads
Troubleshooting steps
  1. On the Companies tab, check the "Last Company Sync" timestamp at the top-right. If it shows recent activity, give the sync a few more minutes and refresh the page.
  2. On the database server, verify the RapidReconciler SQL Agent job is enabled and has run successfully at least once.
  3. Check the SQL Agent job history for errors on the most recent run.
  4. If the job is failing or has never run, escalate to the GSI database administrator to investigate the SSIS package and JDE connection.

Tier 3 · RR Developer Database & SSIS

RapidReconciler data is stale — SQL Agent job not running

Tier-1 difficulty: Moderate — Check SQL Agent service + job history in SSMS.
Symptom: Data in RapidReconciler is older than expected and isn't refreshing. The customer reports yesterday's (or older) figures rather than current data, even though the SSIS package was deployed correctly and previously ran fine.
Likely causes
  • SQL Server Agent service on the database server is stopped or set to Manual / Disabled (most common after a server restart or patch reboot).
  • The SQL Agent job that runs the SSIS package is disabled on the schedule.
  • The SQL Agent job is enabled and on schedule, but the most recent runs have failed (job history shows failures rather than successes).
  • The SSIS package itself errors out mid-run — credentials expired, source JDE connection broken, or proxy account permissions changed.
  • The job hasn't fired yet because the next scheduled run time hasn't arrived (verify against the schedule, not just "it should be running").
Troubleshooting steps
  1. On the database server, open SQL Server Configuration Manager (or services.msc) and confirm SQL Server Agent is running. If stopped, start it; if set to Manual, change to Automatic so it survives the next reboot.
  2. In SSMS, expand SQL Server Agent → Jobs, locate the RapidReconciler / SSIS job, and confirm it is Enabled.
  3. Right-click the job → View History. Check the last several runs: are they succeeding, failing, or not running at all? A run that simply isn't happening points back to the Agent service or the schedule; runs that fail point to the package or its credentials.
  4. If recent runs failed, expand the failed run in history and read the step-level error message — it identifies whether the failure is in the SSIS package execution or in a downstream T-SQL step.
  5. If the job is healthy but stale data persists, run the job manually (right-click → Start Job at Step) and watch it complete. Confirm fresh data lands in RapidReconciler_Prod.
  6. Verify the schedule on the job (right-click → Properties → Schedules) matches the agreed cadence with the customer.

Tier 3 · RR Developer Database & SSIS

SSIS package deployment fails — OLE DB driver error

Tier-1 difficulty: Hard — Resolve OLE DB driver/architecture mismatch on the server.
Symptom: During SSIS package deployment, an error indicates the OLE DB provider for JDE is not installed or cannot be found.
Likely causes
  • Driver mismatch — only 64-bit installed when both 32-bit and 64-bit are needed
  • OLE DB provider was not installed on the database server
  • Wrong version of the OLE DB provider for the JDE database type
Troubleshooting steps
  1. On the database server, verify both 32-bit and 64-bit versions of the required OLE DB provider are installed.
  2. Re-install the OLE DB provider with both architectures if missing.
  3. After installation, re-run the SSIS package deployment.
  4. Verify driver installation before the install meeting next time — this is a known cause of install delays.

Tier 3 · RR Developer Database & SSIS

SSIS catalog deployment fails — permission denied

Tier-1 difficulty: Escalate — Catalog/SSISDB permission work on the SQL server; escalate to RR DBA.
Symptom: When deploying the SSIS package to the SSISDB catalog, deployment fails with a permission/authentication error.
Likely causes
  • The deploying account does not have sysadmin or ssis_admin privileges
  • Windows Authentication was not used for deployment (required)
  • SSISDB catalog has not been created on the target SQL Server instance
Troubleshooting steps
  1. Confirm you are deploying using Windows Authentication, not SQL Authentication.
  2. Verify the deploying account has sysadmin or ssis_admin privileges on the target SQL Server instance.
  3. Confirm the SSISDB catalog exists — if not, create it first.
  4. If the SQL Agent job steps will run under a proxy account, ensure the proxy has access to any UNC paths used by the package.

Tier 1 · Helpdesk Tech Network & Connectivity

User sees no companies after admin granted access (session refresh)

Tier-1 difficulty: Trivial — Click the refresh icon on the widget; if that fails, log out and back in.
Symptom: Customer admin reports that a user with access to one or more companies clicks into a tab (Reconciliation, Inventory, etc.) and sees no companies in the company selector. The user is set to Active, the customer admin has confirmed in Admin → Users that the right companies and tab columns are checked under Authorized Companies, and they have saved the user. From the GSI side, everything in VALC looks healthy.
Likely causes
  • The user's session is holding stale security settings. The RR app reads Authorized Companies / Authorized Tabs into the session at login (and into individual widgets when they first render); changes the customer admin makes after that point won't appear until the session re-reads them.
  • Cause 1's most common flavor: the user was already logged in when the customer admin updated their permissions. Their browser has the old permission set cached on the page.
  • The widget on the page the user is looking at hasn't refreshed its filter/company list since the permission change — widgets pull their list once when the page first loads.
  • Less common: the customer admin closed the Authorized Companies dialog with the X instead of clicking Save, so the change wasn't actually persisted (verify before assuming session cache).
Troubleshooting steps
  1. Confirm with the customer admin that the user's Authorized Companies are saved correctly: open Admin → Users, click the edit (pencil) icon for the user, and check the Authorized Companies section. Confirm the right companies are ticked under the column that matches the tab the user is trying to use (Reconciliation has its own column, Inventory has its own, etc.). If the boxes are unticked, tick them and click Save — that's the fix; stop here.
  2. If the boxes are correctly ticked and saved, the user's session is the issue. Have the user click the refresh icon in the top-right corner of the widget (filter / company-selector area) on the page they're on. This forces the widget to re-read security settings from the RR app and the missing companies should appear.
  3. If the widget refresh doesn't surface the companies, have the user log out and log back in. A fresh login pulls the current security settings from the server and bypasses anything the browser had cached for the previous session.
  4. If neither the widget refresh nor a re-login works, have the user hard-refresh the browser (Ctrl+F5 on Windows / Cmd+Shift+R on Mac) on the affected page, then log in again. This clears any cached page assets that might still be holding old permissions.
  5. If after a clean re-login the user still sees no companies on the tab, re-open the user record in Admin → Users on the customer admin's side and verify one more time that the boxes were saved — closing the Authorized Companies dialog with the X instead of Save can drop the changes silently.
  6. If none of the above resolves it, capture a screenshot of the user's Authorized Companies dialog (as the customer admin sees it) along with a screenshot of what the user sees on the tab, and email both to rrsupport@getgsi.com. At that point the issue is more likely on the back end than session caching.
RR Network Tech walks the customer admin and end user through this on a screenshare or phone call; no GSI tools or VALC access required. The customer admin owns user setup in Admin → Users; the end user owns the in-app refresh / re-login.

Browser blocking local network access (Edge / Chrome permission)

Tier-1 difficulty: Easy — Customer-guided: toggle a browser site permission.

Customer sees no data — checking the browser console (DevTools)

Tier-1 difficulty: Moderate — Open DevTools, compare console against healthy baseline.
Symptom: Customer reports no data after the basic checks (VPN connection, hard reload, Database tab healthy in VALC) come back clean. Opening the browser DevTools Console is the next step to see whether the page is silently failing — JS errors, blocked requests, CORS rejections, or the agent script not loading at all.
Likely causes
  • Agent script (base.js) failed to load — common when the customer's network blocks the agent's hostname or the cert chain is broken (no URL visited log lines at all).
  • The page is making AJAX calls to an endpoint that returns 4xx/5xx — the customer can log in but every data fetch fails server-side.
  • CORS block or mixed-content error — the page is HTTPS but is trying to call an HTTP endpoint, or an origin isn't whitelisted.
  • JavaScript exception thrown early in app bootstrap — usually visible as a red error in Console with a stack trace pointing at base.js or the AngularJS controller.
  • Customer is running a browser extension (ad-blocker, privacy tool, corporate DLP) that's stripping requests — visible as net::ERR_BLOCKED_BY_CLIENT in the console.
  • Browser is genuinely fine and the console is clean — in which case the failure is back-end (covered by the SQL Agent, SSIS, or Cloudflare scenarios).
Troubleshooting steps
  1. Have the customer open DevTools: in Edge or Chrome press F12 (or Ctrl+Shift+I), then click the Console tab.
  2. Ask the customer to clear the console (the ⊘ icon at top left), then have them perform the action that fails — usually login, then navigating to the page where data should appear.
  3. Compare against the healthy-baseline pattern. A working session shows lines like URL visited /page/login and URL visited /app/inventory/reconciliation from base.js — that confirms the agent is loaded and the customer is navigating the app.
  4. Ignore the known noise: warnings about 'popover' attribute with an invalid value are AngularJS template noise from the UI and are harmless. 404s on fontawesome-webfont.woff or Simple-Line-Icons.woff are cosmetic font fallbacks and do not affect data flow.
  5. Look for the actual signal: any red error from base.js with a stack trace, any 4xx/5xx on the application's own API calls (not the .woff files), CORS or Mixed Content warnings, net::ERR_BLOCKED_BY_CLIENT entries from a browser extension, or the complete absence of URL visited lines from base.js.
  6. Have the customer screenshot or copy the relevant errors and email them to rrsupport@getgsi.com. Right-click in the console → Save as… exports the full log if a longer trace is needed.
  7. If the console shows no base.js activity at all, the agent script never loaded — jump to Agent services are not running and Browser shows “Not Secure”.
  8. If the console shows red errors on application API calls (4xx/5xx), the agent loaded but the back end is unhappy — check SQL Agent job not running and the database scenarios.
  9. If the console shows net::ERR_BLOCKED_BY_CLIENT or CORS blocks, the customer's browser environment is interfering — ad-blocker, corporate DLP, or proxy. Have them retry in an incognito window with extensions disabled to confirm.
RR Network Tech walks the customer through DevTools on a screenshare; the customer's I/T may need to whitelist agent hostnames if extensions or DLP are stripping requests.

Tier 2 · Network Tech Certificate & DNS

Per-customer hostname fails to resolve (ERR_NAME_NOT_RESOLVED in console)

Tier-1 difficulty: Moderate — The cross-network test triages customer-side vs. GSI-side in about two minutes.
Symptom: Customer reports no data. Opening DevTools Console shows red net::ERR_NAME_NOT_RESOLVED errors on requests to a per-customer RapidReconciler hostname (e.g. rrprod-<customercode>.getgsi.com on a non-standard port). The login page may load fine but data fetches fail because the front-end can't reach the customer-specific backend. The error is a DNS failure — the browser tried to look up the hostname and got nothing back.
Likely causes
  • Customer-side DNS not resolving the per-customer subdomain — the most common cause. A DNS filter, a split-horizon DNS policy, or a recent change to the customer's network config has broken resolution for the deeper GSI subdomain even though the public getgsi.com still works.
  • Split-tunnel VPN routing — the customer's VPN is up but isn't routing the per-customer subdomain through it. The public domain resolves on the public internet; the deeper subdomain is only resolvable over the VPN.
  • GSI-side DNS issue — the A record for the per-customer hostname on GSI's authoritative DNS is broken or missing. This will affect every user at that customer simultaneously.
  • Customer-side firewall / proxy sinkhole — a recent firewall or web-filter change is intercepting the hostname before DNS resolution completes (uncommon but observed).
  • Local hosts file or DNS cache on the user's machine — a stale entry pointing at an old IP, or a poisoned local DNS cache (rare; follows from the cross-network test result).
Troubleshooting steps
  1. Confirm the failing URL pattern in the customer's DevTools Console. Note the exact per-customer hostname and port (e.g. rrprod-<customercode>.getgsi.com:NNNNN). Capture a screenshot of the console so the network signature is preserved if the issue is intermittent.
  2. Run the cross-network triage. Have the customer test from another machine on the same network, then from a different network (mobile hotspot or personal phone tether is fine). This is the single highest-value test — about two minutes and it tells you which side owns the problem.
  3. Interpret the test:
    • Same-network machine fails the same way + different network works → customer-side. Their corporate network is blocking or failing to resolve the hostname.
    • Same-network machine fails the same way + different network also fails → GSI-side. The DNS record is broken or missing on GSI's authoritative DNS.
    • Same-network machine works + this user's machine fails → user-machine local. Stale DNS cache, hosts file, or browser profile.
  4. Check whether multiple users at the same customer are reporting identical errors right now. If yes, treat as GSI-side until proven otherwise (a customer-network issue typically wouldn't cause every user at the customer to fail simultaneously).
  5. If the verdict is customer-side: have the customer's I/T explicitly whitelist the per-customer hostname (and the port) on their DNS filter, web filter, and firewall. Ask whether they recently rolled out a DNS-filtering product (Cisco Umbrella, Zscaler, Cloudflare Gateway, DNSFilter, etc.) or changed VPN config (split- vs. full-tunnel). After whitelisting, have the customer hard-refresh and recheck the console.
  6. If the verdict is GSI-side: escalate to GSI I/T (Network Tech) to verify the A record for the per-customer hostname on GSI's authoritative DNS, and to confirm the per-customer backend host is up. Do not have the customer keep retrying — the issue is not on their end.
  7. If the verdict is user-machine local: have the user run ipconfig /flushdns (Windows) or sudo dscacheutil -flushcache (Mac), check their hosts file (C:\Windows\System32\drivers\etc\hosts on Windows; /etc/hosts on Mac/Linux) for a stale entry containing the GSI hostname, and retry. If still failing, have them try in a private/incognito window and a different browser to rule out browser-profile DNS-over-HTTPS misconfiguration.
  8. Once resolved, note in the ticket which path resolved it (customer I/T whitelist, GSI DNS fix, local DNS flush). This helps the next person when the same pattern surfaces from a different customer.
RR Network Tech runs the cross-network triage with the customer; if it points GSI-side, escalates to GSI I/T (RR Developer) for DNS record verification.
