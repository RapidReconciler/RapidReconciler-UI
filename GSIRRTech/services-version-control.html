Services version control workflow

How to cut a new Services jar release: tag the commit, let the GitHub Action build + publish the jar, then push it to customers via the VALC deploy panel.

Overview

The RapidReconciler-Agent repo ships Services jar releases on git tags. Despite the repo name, the produced artifact is the per-DB Services jar (client-services-X.Y.Z.jar) — the green-field rewrite of v359’s Services component, not the JMS Agent jar itself. Day-to-day work on Dev stays on 0.1.0-SNAPSHOT; when a chunk is release-worthy, a maintainer tags the commit, and the rest is automatic:

  • GitHub Actions rebuilds the jar with the tag’s version baked into @project.version@.
  • The built jar lands on the repo’s Releases page, ready to be uploaded into VALC.
  • Auto-generated release notes summarize commits since the previous tag.

Once a customer’s Services instance receives the new jar (via VALC’s deploy panel and SelfUpdateMessage), the dashboard’s Services Version column updates — the value lives in client_deploys.target_version_string, not in any Services self-report.

Scope: this doc covers the Services release flow. The literal Agent jar (rr-valc-agent.jar, deploy slot component='agent') is the legacy v359 JMS broker client — its source lives outside our repos and it has no release flow on our side. The pinned reference for that artifact is RapidReconciler-Agent/artifacts/v359/rr-valc-agent.jar. See v359 vs new agent § 1 for the three-component breakdown.

If you haven’t yet, read v359 vs new agent § 5 for the self-update plumbing. This doc covers the release side; that one covers the deploy side.

1. Versioning convention

Semantic versioning, with a -SNAPSHOT default for the Dev branch.

WhereLooks likeWhat it means
Dev branch HEAD0.1.0-SNAPSHOTPermanent placeholder. Every merged PR ships under this version until a tag fires the Action. The Services jar’s /health endpoint reports this string verbatim.
Released tagv0.2.0Maven version becomes 0.2.0; the leading v is stripped by the Action. Jar baked in as 0.2.0; release published on GitHub.
Pre-release tagv0.2.0-rc1, v0.2.0-betaSame flow, but the GitHub Release is flagged Pre-release. Useful for letting a single customer pilot a build.

What goes in each component

  • Major (X.0.0): breaking changes to the JMS contract (rr-common) or a forced minimum JDK bump. Triggers customer-side action.
  • Minor (0.X.0): new endpoints, new fields on existing endpoints, additive Services jar features. Customers update by deploy; no install action required.
  • Patch (0.0.X): bug fixes, performance, no API surface change.

2. Cut a release

The whole dance is two actions inside Visual Studio: create a tag on the right commit, then push it. The Action takes care of the rest.

Prefer the command line? The CLI equivalent is two commands: git tag v0.2.0 && git push origin v0.2.0. The rest of this section walks through the Visual Studio UI.

Prefer Claude Code? Run /release agent (or /release valc) in any session. The command checks the repo state, proposes the next version, fast-forwards main to Dev, cuts the tag, watches the Action complete, and verifies the jar landed on the Release page. Pass an explicit version as the second argument (/release agent v0.2.1) to skip the proposal step.

The command file lives at ~/.claude/commands/release.md — user-scoped, so it works from any repo’s working directory. See § 7 if it isn’t available in a fresh session.

1

Open the repo in Visual Studio and switch to Dev

If the solution isn’t already open, use File → Open → Open from Git Repository or File → Recent Projects to load RapidReconciler-Agent.

Confirm the active branch in the bottom-right status bar — it should read Dev. If not, click the branch name and switch via the popup.

2

Confirm the working tree is clean

Open View → Git Changes (Ctrl+0,G). The Changes list should be empty — no pending stages, no unsaved files. Outgoing commits should be zero (Dev synced with origin).

Run tests in Test → Test Explorer or via the integrated terminal (View → Terminal, then mvn -B test). Don’t cut the tag if tests are red.

3

Pick the version

Open View → Git Repository (Ctrl+0,R) for the full history view. Scroll the local history graph or expand Tags in the left panel to see the previous release tag.

Review the commits between that tag and the current Dev HEAD — right-click the previous tag and choose Compare with Branch → Dev to see exactly what’s landed.

Decide major / minor / patch using the table in § 1.

4

Create the tag

In the Git Repository window, find the commit at the tip of Dev (top of the history graph). Right-click it and choose Create Tag.

In the dialog:

  • Name: v0.2.0 (always lead with v; the Action strips it).
  • Message: a short summary — this is the annotated tag’s message and shows up in git log. E.g. Services v0.2.0 — DMAAI overlay endpoints, audit-detail expansion.

Click Create. The tag now exists locally; it hasn’t been pushed yet.

5

Push the tag

Still in the Git Repository window, expand Tags in the left panel. The new v0.2.0 entry appears with a small “up-arrow” marker showing it’s local-only.

Right-click v0.2.0 and choose Push Tag. (Alternatively, Git → Push Tags from the menu bar pushes every unpushed tag at once — useful if you batched several.)

Pushing the tag is what triggers .github/workflows/release.yml on GitHub. This is the only externally-visible step.

6

Watch the Action

Visual Studio doesn’t surface GitHub Actions natively. Open the repo in a browser, go to Actions → Release, and watch the run.

Typical build: 1–2 minutes. When it finishes you’ll see a new entry under Releases with the jar attached and auto-generated release notes filled in.

Visual Studio 2022 only. Earlier VS versions had Team Explorer instead of the modern Git tooling described above. The steps differ; this doc assumes VS 2022.

3. What the Action does

The release workflow at .github/workflows/release.yml runs on every v* tag push. Step by step:

StepWhat it does
CheckoutPulls the tagged commit (not main / Dev HEAD).
Setup JDK 21Temurin distribution; matches the Services jar’s runtime floor. Maven cache is restored from the previous run.
Derive versionStrips the leading v from refs/tags/v0.2.0 → 0.2.0. Stored as a step output.
Set Maven versionmvn versions:set -DnewVersion=0.2.0 -DgenerateBackupPoms=false — rewrites the <version> in pom.xml in place. Not committed; only used for this build.
Buildmvn -B -DskipTests package. Tests already ran before tagging (Step 1 above); skipping here keeps the release fast.
Publishsoftprops/action-gh-release@v2 opens a GitHub Release named after the tag, attaches target/client-services-0.2.0.jar, and auto-generates release notes from PRs merged since the last tag.

The Action doesn’t commit the version bump back to the repo — that’s deliberate. Dev branch HEAD stays on -SNAPSHOT, no chore commits cluttering history.

4. After the release

The jar on the GitHub Release page isn’t the Services install path on its own — it has to land in VALC’s artifact catalog (the file_versions table) before a customer’s Agent can stream it down to a Services instance.

  1. Open the new GitHub Release; download the attached client-services-X.Y.Z.jar.
  2. In production VALC, navigate to the artifact upload page; upload the jar with component = services. VALC records the SHA1, file size, and version string into file_versions.
  3. For each client you want to push the new version to: open Manage Deploy, select the new version, click Deploy. VALC publishes SelfUpdateMessage to the Agent’s queue, the Agent streams the jar to the per-DB Services slot per v359 vs new agent § 5, and a row appears in client_deploys with status = PENDING.
  4. Within a few seconds the deploy progresses through IN_TRANSFER → INSTALLING → SUCCEEDED. The mini-VALC dashboard’s Services Version column updates to the new value once the SUCCEEDED row lands (it’s pulled from client_deploys.target_version_string).

Watch the component slot. The artifact upload form asks for a component value. Services jars (this doc’s artifact) go into component='services'. The other slot, component='agent', is reserved for rr-valc-agent.jar — the legacy v359 JMS broker client. Uploading a Services jar into the agent slot will fail SHA1 verification on the customer side because the Agent’s HealthService expects a different Actuator contract than the Services jar exposes.

Customer deploys are not automatic on every release. Releasing publishes the jar; deploying pushes it. This split is intentional — gives you a chance to pilot a release on one or two clients before fanout.

5. Pre-releases & rollbacks

Pre-release

Tag with a suffix:

git tag v0.2.0-rc1
git push origin v0.2.0-rc1

The Action’s prerelease: field is set to contains(version, '-'), so any tag with a hyphen lands as a Pre-release on GitHub. Same artifact path; just a UI flag.

Rollback

Two options, depending on what went wrong:

  • Bad jar, deploys not yet sent: delete the GitHub Release (don’t delete the tag — future reference). Cut a new patch release (v0.2.1) with the fix.
  • Bad jar, already pushed to customers: redeploy the previous version. VALC’s artifact catalog keeps old versions; pick the prior one in Manage Deploy. The Agent streams the older jar to the Services slot and the per-DB instance restarts onto it.

Once a tag is pushed and the Action publishes, don’t delete the tag. Tags are part of the audit trail; reissuing a deleted tag at a different commit confuses everyone.

6. Mini-VALC parallel

The mini-VALC repo (RapidReconciler-Valc) has the same release.yml workflow. Same tag flow, same Action behavior, same Release page. The only difference is the artifact filename: valc-X.Y.Z.jar instead of client-services-X.Y.Z.jar.

Different consumer model, though. Mini-VALC isn’t a customer-deployed artifact — it’s the parallel dev-side broker + dashboard that replaces the central VALC SPA for local development. Its release jar doesn’t land in file_versions; it’s just downloaded and run directly via setup/run-mini-valc.ps1 or its successor. Mini-VALC’s version surfaces on its /actuator/info endpoint and in the dashboard sidebar footer once that’s wired through.

Tag-cutting cadence is independent of the Services repo — release each repo when its work is release-worthy.

7. Troubleshooting

SymptomCauseFix
Action ran, no jar attached to the Release The Maven build failed silently — check the Action’s logs for the Build jar step. Re-run the workflow once the underlying issue is fixed (you can’t fix it just by re-tagging; the same tag won’t re-trigger). Cut a new patch tag instead.
target/client-services-X.Y.Z.jar not found at the publish step The versions:set step didn’t apply, or the Spring Boot repackager produced a differently-named artifact. Confirm the pom uses ${project.version} in <finalName> (or omit finalName entirely). Re-tag with a new patch.
Tag pushed, no Action run Tag didn’t match the v* glob (e.g. 0.2.0 instead of v0.2.0). Push a new tag with the leading v.
Release notes empty or missing PR links generate_release_notes: true needs at least one merge between the previous tag and the new one. If no PRs landed between tags, the auto-generated notes will be terse. Edit the Release on GitHub to add narrative if needed.
Customer’s Services instance never picks up the new version The deploy step (after the release) wasn’t run, or SelfUpdateMessage hasn’t reached the customer’s Agent. Check client_deploys for a row with the target version; if absent, go to Manage Deploy and trigger the push. If the row exists but is stuck in PENDING / IN_TRANSFER, the customer’s Agent may be offline — check its JMS heartbeat in VALC’s clients list.
Services jar uploaded but customer health check fails immediately The jar was uploaded into the wrong artifact-catalog slot (component='agent' instead of component='services'). Re-upload into the correct slot. The Agent’s self-update path verifies Actuator endpoints specific to that component; a Services jar in the agent slot will fail the contract check.
/release command not available in Claude Code The command file is user-scoped at ~/.claude/commands/release.md; missing on a new machine, or the session started before the file was created. Confirm release.md exists under %USERPROFILE%\.claude\commands\. If missing, copy it from any machine that has it (it’s a single self-contained markdown file). Restart the Claude Code session to pick up newly-added commands.
