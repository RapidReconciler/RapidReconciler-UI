Overview

The RapidReconciler_Prod.dtsx package is the production data-extraction component of RapidReconciler. It runs as a scheduled SQL Agent job on the customer's SSIS server, reads roughly 30 tables from the customer's JD Edwards source database, and writes them into the customer's rrv7-<customer> RapidReconciler database. The RR Agent then reads from this RR-local database and serves it to the RapidReconciler UI.

Why the package exists

RapidReconciler does inventory and financial reconciliation against JDE data, but it never reads directly from the customer's JDE production database. Two reasons:

  • Performance isolation. JDE production runs the customer's day-to-day operations; running RR reconciliation queries directly against it would compete with production transactions.
  • Schema portability. RR's queries are shaped around its own copy of the data; the SSIS package is the seam that lets RR's schema stay stable while the customer's JDE schema varies (Oracle vs SQL Server JDE, proddta vs jdeprod vs crpdta schema, fiscal-year calendars, decimal-precision dictionaries).

The package is the bridge: it reads JDE through a read-only login (rapidrec/rapidrec by convention), transforms numeric columns into RR-canonical decimal precision via the Dec* multiplier variables, and writes into the RR database where the agent picks up.

Where the package sits in the install flow

The package is delivered to the GSI DBA as part of the install zip (alongside the four SQL scripts that build the database). During Installing the Production Database, the DBA:

  1. Creates the RapidReconciler_Prod database, the rruser SQL login, and the RapidReconciler_Prod SQL Agent job by running the four install scripts (Step 7).
  2. Opens the package in Visual Studio Community, configures the 6 install-time variables (Step 8 Step 5 of 8) and the 2 OLE DB connection managers (Step 8 Step 4 of 8), and deploys it to the SSISDB catalog at /SSISDB/RapidReconciler/RapidReconciler (Step 8 Step 6–8 of 8).
  3. Points the SQL Agent job's step at the deployed package and enables the schedule (Step 9).
  4. Customer I/T waits for the first scheduled run to populate the database, then verifies via v_diagnostic4_b_to_c_log (Step 11).

From then on the package runs on the SQL Agent schedule (typically nightly during off-peak hours) and the agent serves whatever's in the RR database.

Runtime lifecycle of one job run

Each scheduled run follows the same shape:

  1. Initialize — test both connections, log the package-start banner, read enabled-module flags from usertabnames into ModInv and ModRnv.
  2. Companies — copy business unit / fiscal pattern / currency tables. Compute Julian start date from Gregorian aaStartDateGr.
  3. General Ledger — copy chart of accounts and GL distributions; populate the Cache - Objects cache from F0901 object codes; use it as a lookup to filter F0902 and F0911 down to inventory-relevant accounts only.
  4. Inventory — copy the item-ledger (F4111 — cardex), watermarked by maxUKID from prior runs so each run is incremental.
  5. Inventory Supplemental — copy item master, branch, costs, lot, UoM tables. Populate the Cache - Short Items cache.
  6. Orders — copy purchase order and sales order detail.
  7. Receipts — copy receipts and vouchers.

Each table copy writes a row to v_diagnostic4_b_to_c_log with the start/end timestamps, row count, and an ErrorNum (0 = success). Step 11 of the install procedure is the customer DBA querying this log to confirm the load was healthy.

Connection managers

The package defines four connection managers. Two are real OLE DB connections (JDE source, RR destination); two are SSIS in-memory cache connections used by the GL/inventory lookup chain. The OLE DB connections are configured by the GSI tech during Step 8 of the install procedure; the cache connections are internal and need no configuration.

JDESource — JDE production source

OLE DB connection to the customer's JDE production database. Read-only by convention — the package never writes back to JDE.

Template connection string (as shipped)

Data Source=.;Initial Catalog=jde_treatt;Provider=SQLNCLI11.1;Integrated Security=SSPI;

Note: the template ships pointed at a sample customer (jde_treatt) on the local instance with Windows Authentication. The install tech replaces both the Data Source and Initial Catalog with the customer's actual JDE server and database, and switches the authentication to SQL Authentication using the rapidrec/rapidrec read-only login (Step 8 Step 4 of 8 in the install procedure).

i
Provider note

Both OLE DB connections use the SQL Server Native Client 11 (SQLNCLI11.1) provider. This is the standard SQL Server driver for SSIS in SSDT/VS 2017+, not the JDE-specific OLE DB provider that the SSIS driver mismatch scenario warns about. The JDE-specific provider applies only when the customer's JDE source is on Oracle or AS400 — if the JDE source is on SQL Server (as it is for many JDE-on-SQL customers), SQLNCLI11.1 handles both connections.

Process flow

The package's control flow is 7 top-level sequence containers executed in package-XML order. Each container holds one or more data-flow tasks (and in the case of Initialize, only SQL tasks). Within each data-flow task, an OLE DB Source reads from JDE via JDESource, optional transformations apply, and an OLE DB Destination writes to RRLocal.

JDE source tables

Every JDE F-table the package reads, grouped by sequence container. For each table the column list (from the actual SSIS source query in the package), the WHERE-clause filter, and a brief description.

i
All source queries use the dbowner concat pattern

Each query is built as "SELECT … FROM " + @[User::dbowner] + "F<number> [WHERE …]". When dbowner is proddta., the resolved query is SELECT … FROM proddta.F0006 …; when dbowner is missing the trailing period, the resolved query becomes SELECT … FROM proddtaF0006 … and silently returns zero rows. See Key mechanisms → The dbowner pattern.

Companies container tables

F0006 — Business Unit Master

SELECT MCCO, MCMCU, MCMCUS, MCDL01,
       MCRP01, MCRP02, … MCRP30
FROM   <dbowner>F0006
WHERE  MCMCU != ' '

Company / business unit / cost-center hierarchy. The MCRP01..MCRP30 columns are JDE category codes used for reporting and reconciliation grouping.

F0008 — Date Fiscal Patterns

SELECT CDDTPN, CDFY, CDDFYJ,
       CDD01J, CDD02J, … CDD14J
FROM   <dbowner>F0008
WHERE  CDDFYJ > 99999

Fiscal year calendar with period-end dates in Julian format. Drives period-by-period reconciliation cuts.

F0010 — Company Master

Currency code and fiscal-year-start info per company.

F0015 — Currency Exchange Rates (DISABLED)

SELECT CXCRCD, CXCRDC, CXAN8, CXRTTYP, CXEFT,
       CXCLMETH, CXCRCM, CXTRCR, CXCRR, CXCRRD, CXCSR
FROM   <dbowner>F0015
WHERE  CXEFT > 99999 AND CXEFT >= @[User::aaStartF0015]

Disabled in the template — replaced by F1113 for most JDE customers.

F1113 — Currency Exchange Rates (alternate)

Alternate exchange-rate table used in newer JDE deployments.

F41001 — Inventory Constants

SELECT DISTINCT CIMCU, CIAN8
FROM   <dbowner>F41001
WHERE  CIMCU != ' '

Branch-level inventory constants. Defines which branches RR considers part of the customer's inventory footprint.

General Ledger container tables

F0011 — Batch Control Records

SELECT ICICUT, ICICU, ICUSER, ICIAPP, ICDICJ, ICIST
FROM   <dbowner>F0011
WHERE  ICDICJ >= @[User::minbatch]
   AND ICICUT IN ('0','O','S','ST','IB','T','N','NC','G','V','I')

Batch headers for GL postings. The ICICUT filter restricts to batch types that affect inventory and AP/AR posting.

F0901 — Account Master

SELECT GMCO, GMAID, GMMCU,
       GMOBJ, GMSUB, GMDL01
FROM   <dbowner>F0901

Chart of accounts. The GMOBJ values feed into Cache - Objects for downstream filtering.

F0902 — Account Balances

GL account balances by period. Filtered against Cache - Objects to exclude accounts whose object code isn't in the cache (the Filter out non inventory accounts lookup transform).

F0911 — Account Ledger / GL Distribution

SELECT GLKCO, GLDCT, GLDOC, GLDGJ, GLJELN,
       LTRIM(RTRIM(GLEXTL)) AS GLEXTL,
       GLPOST, GLICU
FROM   <dbowner>F0911 (NOLOCK)
WHERE  GLDGJ >= '@[User::minGLDATEGr]'
ORDER  BY GLKCO, GLDCT, GLDOC, GLDGJ, GLJELN, GLEXTL

The GL distribution detail. Filtered to the rolling RefreshDays window (default 35 days back). Also filtered against Cache - Objects.

F4095 — Account AAI

SELECT DISTINCT MLCO, MLANUM, MLDCTO, MLDCT,
       MLGLPT, MLMCU, MLOBJ, MLSUB, MLCOST
FROM   <dbowner>F4095

Automatic Accounting Instructions — the mapping table that tells JDE which GL accounts to post to for each transaction type.

F4096 — Item AAI

SELECT FAANUM, FAAST, FACO, FAITEM, FAOBJF, FAOBJT,

Key mechanisms

Three mechanisms that drive the package's behavior and that anyone diagnosing a failure needs to understand: the dbowner string-concat pattern, Gregorian→Julian date conversion via qryJulianStart, and the cache & lookup chain that filters GL accounts down to inventory-relevant ones.

The dbowner string-concat pattern

Every source query in the package is built dynamically by string-concatenating the dbowner variable into a SQL command. The literal query text isn't stored anywhere; what's stored is the expression that builds it.

Example: the qryF0006 expression

Each data-flow task has a qry<table> variable whose EvaluateAsExpression property is true. For F0006, that expression is:

"Select MCCO, MCMCU, MCMCUS, MCDL01, MCRP01..MCRP30 FROM " + @[User::dbowner] + "F0006 Where MCMCU != ' ' "

At runtime SSIS evaluates this expression by substituting the current dbowner value:

  • If dbowner = "proddta.": "Select … FROM proddta.F0006 Where …" — correct.
  • If dbowner = "proddta" (no trailing period): "Select … FROM proddtaF0006 Where …" — an object that doesn't exist.

The OLE DB Source component then sends the resolved query to JDESource. If the table doesn't exist, OLE DB returns zero rows but does not raise an error — SSIS treats "no matching rows" as a successful query. The data-flow task completes, UpdateCount is logged as 0, and ErrorNum is 0. From the diagnostic table's perspective the run looks healthy; from the customer's perspective no data ever shows up in RR.

!
This is why forgetting the trailing period is the #1 install failure mode

The package fails silently and the diagnostic doesn't surface it as an error. See SSIS package deployed but extracts no data for the dedicated runbook.

Schema variants seen across customers

JDE deploymentTypical dbowner value
JDE on Oracle (most common)proddta.
JDE on SQL Server (older)proddta. or jdeprod.
JDE on SQL Server (newer, with schema-named users)jdeprod. or crpdta.
JDE with all tables in the default schema (rare)dbo. (the template default — matches no real customer)

If the install tech sees the template default dbo. in a deployed customer's package, that's almost certainly the variable having never been configured. The schema value comes from the JDE Data Configuration section of the pre-install submission email.

Gregorian to Julian date conversion

JDE stores dates internally in a custom 6-character Julian format: CYYDDD where C = century flag (1 for 2000s, 0 for 1900s), YY = year, DDD = day of year. So 1 May 2026 in JDE Julian is 126121 (century 1, year 26, day 121).

The user-facing variable aaStartDateGr is in Gregorian YYYY-MM-DD format. The package converts it to Julian via the qryJulianStart expression in the Companies sequence container:

declare @juliandate_out as nchar(6),
        @century as int,
        @yy as int,
        @dayofyear as int,
        @gregorian as datetime

select @gregorian = "  + @[User::aaStartDateGr]  + "

select @century = case
         when datepart(yyyy, @gregorian) > 2000 then 100000
         else 0
       end

select @yy = cast(
         (substring(cast(datepart(yyyy, @gregorian) as nvarchar(4)), 3, 2))
         as int
       )

select @dayofyear = datepart(dayofyear, @gregorian)

select @juliandate_out = @century + @yy * 1000 + @dayofyear

select @juliandate_out as start_date_jul

This T-SQL block is itself the result of an SSIS expression that interpolates the aaStartDateGr value. It's executed by an Execute SQL Task that returns start_date_jul into the Inventory container's startdatejul variable, which then feeds the F4111 cardex query:

WHERE ILUKID > @[User::maxUKID]
  AND ILIPCD != 'X'
  AND ILCRDJ >= @[User::startdatejul]

That's why the aaStartDateGr format matters: SSIS embeds the value as a literal in a T-SQL block and lets SQL Server parse it. If the date isn't YYYY-MM-DD, SELECT @gregorian = <value> either implicitly converts in an unexpected way or errors out at the SQL layer.

Cache & lookup chain (GL filtering)

RR cares about a specific slice of the customer's chart of accounts — the accounts that touch inventory. Pulling all of F0902 and F0911 unfiltered would be wasteful (chart-of-accounts can have tens of thousands of GL accounts; only a fraction touch inventory). The package's design uses an in-memory cache to filter:

  1. Populate: The Load Objects data-flow task in the General Ledger container reads GMOBJ values from F0901 (whose own filter restricts to the relevant chart-of-accounts subset) and writes them to Cache - Objects.
  2. Consume in F0902: The Copy F0902 task includes a Lookup Transform (Filter out non inventory accounts) that joins each F0902 row against Cache - Objects on the object code. Rows whose object code isn't in the cache are routed to the no-match output (which has no downstream destination), effectively filtering them out.
  3. Consume in F0911: The Copy F0911 task has the same Lookup Transform pattern (Filter out unused accounts), filtering GL distribution rows to those that hit cached object codes.

The cache lives only in memory for the duration of one package run — it's not persisted between runs. Each scheduled execution rebuilds the cache from F0901 first, then uses it for downstream filtering.

i
Why this matters for diagnostics

If F0901 is missing or empty in the resolved query (e.g. due to a dbowner issue), the cache is empty, the lookups match nothing, and downstream F0902 and F0911 also load zero rows. The dbowner symptom isn't just "F0006 is empty" — it cascades into "the entire General Ledger container is empty" via the cache.

Cache - Short Items: similar pattern, different table

The Cache - Short Items cache is populated from the F4101 Item Master via the Short Items data flow in the Inventory Supplemental container. The downstream consumer is reserved for future use (the cache is populated but not yet read by any active lookup transform in the shipped template). Engineering may activate downstream usage in a future package version — the cache infrastructure is there.

Deployment & runtime

How the package gets from the install zip to running in production on the customer's SQL Agent schedule. The install procedure has the full step-by-step; this section is the operational summary plus the verification path.

Deployment path

The customer's DBA performs the deployment from a workstation that can reach both the source .dtsx file (the install zip) and the target SQL Server hosting the SSIS catalog. The workstation needs Visual Studio Community with the SQL Server Integration Services Projects extension installed (free download from Microsoft Marketplace; both 2019 and 2022 versions of VS work).

  1. Open in Visual Studio. Create a new Integration Services Project named RapidReconciler; add the existing package RapidReconciler_Prod.dtsx via right-click Add Existing Package.
  2. Configure connection managers. Update JDESource with the customer's JDE server/database/credentials; update RRLocal with the RR-local server and the rruser credentials. Test both connections before proceeding.
  3. Configure variables. Set the 6 install-configurable variables in the Variables window per the customer's pre-install submission email (dbowner, aaStartDateGr, the four Dec* precision multipliers).
  4. Deploy to SSISDB. Right-click the project → Deploy. The Integration Services Deployment Wizard opens; target is the customer's SQL Server with Windows Authentication (Step 1 of the wizard), and the catalog path is /SSISDB/RapidReconciler/RapidReconciler. The deploying account needs sysadmin or ssis_admin on the target instance.
  5. Verify deployment landed. SSMS → Integration Services Catalogs → SSISDB → RapidReconciler → right-click the package — confirm the timestamp reflects the just-completed deployment.
!
Re-deployments must re-configure

SSIS doesn't persist variable values or connection-string parameters across re-deployments by default — if a re-deployment is needed (e.g. after fixing the dbowner variable), the values configured at design time are what get deployed. Save the VS project before deploying so the variables persist across sessions.

Verification — the diagnostic table

Every data-flow task writes a row to v_diagnostic4_b_to_c_log in RapidReconciler_Prod with the run timestamp, row counts, and error code. After each scheduled run, the customer DBA queries this view to confirm the load was healthy:

SELECT Capture, Step, Process,
       UpdateCount, ErrorNum, ErrorMsg
FROM   v_diagnostic4_b_to_c_log
WHERE  Step = 'data capture'
ORDER  BY Capture DESC

Healthy run characteristics:

  • One row per JDE table copied at Step = 'data capture'.
  • ErrorNum = 0 on every row.
  • UpdateCount non-zero for active tables (F4111, F0911, F4111, F4101, F4102 for typical mid-volume customers). Some tables may legitimately show zero (e.g. F4801 if the customer doesn't use work orders).

If any of these is false, see Initial load reports errors or zero counts.

Incremental vs. initial-load behavior

The package is designed to run incrementally: subsequent runs pull only new transactions since the last successful run, using maxUKID watermarks for F4111 and date-window filters for F0911/F4801/F3106. This keeps each run fast (typically minutes for steady-state load).

The first run after install is the exception: there's no prior maxUKID and the date windows are wide (aaStartDateGr is the 2-months-prior-to-fiscal-year-start anchor). This first run does a near-full historical pull and can take hours for high-volume customers. After the first successful run, every subsequent run is incremental.

Some tasks (Copy F0911 Init, Copy F41002 Init1, Copy F43121 Init) are initial-load variants that run only when InitLoad = 1. Engineering can manually flip InitLoad to 1 to force a full reload of those tables outside the normal schedule; it's not part of the standard install procedure.

Logging & error handling

The package uses verbose logging (DTS:LoggingMode="2"). All task starts/ends, variable changes, and errors are emitted to the SSIS server's standard log destinations (typically SSISDB's catalog.execution_data tables).

The package has one event handler defined — OnError at package scope — but its body is empty. Error propagation relies on SSIS's default behavior: any task that errors propagates up the precedence-constraint chain and fails the containing container, ultimately failing the package run. The OnError handler structure exists for future engineering to wire in (e.g. an alert email or external monitoring hook) without re-architecting the package.

Troubleshooting

This package has its own dedicated set of install-scenario runbooks under the install troubleshooting reference. Each is keyed on a specific failure symptom and walks the diagnostic path step by step.

Primary SSIS-specific scenarios

SQL script fails during database install. One of the four install scripts (creation, objects, rruser, SQL Agent job) errors in SSMS. Six-check runbook organized by which script failed. Common causes per script: CREATE FILE encountered operating system error, Database already exists, Login failed, Login already exists, Password validation failed, SQL Server Agent service not running, job already used. SSIS package deployed but extracts no data. Package deploys cleanly, SQL Agent job runs to completion, ErrorNum = 0, but UpdateCount = 0 across every major table. Almost always the dbowner trailing-period issue. Five-check runbook with the actual qryF0006 expression and the OLE DB "no-rows = success" mechanism explained. Initial load reports errors or zero counts. Step 11 verification finds non-zero ErrorNum or zero UpdateCount for major tables. Seven-check runbook with five pattern-matched diagnoses: zero-count-no-error (the dbowner case), login/auth failure, invalid-object-name (schema mismatch), timeout, partial counts.

Deployment-specific scenarios

SSIS catalog deployment fails — permission denied. Deployment from Visual Studio errors before package upload completes. Three checks: Windows Authentication vs SQL Authentication, deploying account privileges (sysadmin / ssis_admin), SSISDB catalog existence. SSIS package deployment fails — OLE DB driver error. Deployment-time error referencing a missing or unrecognized OLE DB provider for JDE. Common with JDE-on-Oracle or JDE-on-AS400 customers where the JDE-specific driver wasn't installed in both 32- and 64-bit on the SSIS host.

Related runtime scenarios

Companies tab is empty after install. Database is online but the Companies tab in the RR UI shows no companies. Often a SQL Agent job that hasn't run yet, or an SSIS source connection issue that's not the dbowner case. Post-deployment JDE import fails — rruser login error. Customer-side admin's production deployment encounters an rruser authentication failure during the import. Five-check runbook covering hidden rruser references in package internals (data-flow source credentials, Execute SQL Task connections, internal connection managers) that the deployment UI doesn't expose. Migration complete but data not flowing. Post-migration to a new server, customer can reach the RR UI but no new data appears. Often SQL Agent schedule not enabled on the new server, or SSIS package connection strings still pointing at the old server.
