Overview

The RapidReconciler_Prod.dtsx package is the production data-extraction component of RapidReconciler. It runs as a scheduled SQL Agent job on the customer's SSIS server, reads roughly 30 tables from the customer's JD Edwards source database, transforms the data along the way (JDE Julian dates back to Gregorian, JDE-scaled numeric columns to RR-canonical decimal precision), and writes them into the customer's RapidReconciler_Prod database (the default name created by Script 1 of the install zip). RR data services — an internal-network-only Java process that runs alongside the RR Agent on the customer's application server — then reads from RapidReconciler_Prod and serves the data through to the RapidReconciler UI.

New to SSIS? Microsoft's Integration Services packages primer covers the underlying concepts (packages, control flow, data flow, connection managers, event handlers, variables) before you dive into RR's specifics below.

Why the package exists

RapidReconciler does inventory and financial reconciliation against JDE data, but it never reads directly from the customer's JDE production database. Two reasons:

  • Performance isolation. JDE production runs the customer's day-to-day operations; running RR reconciliation queries directly against it would compete with production transactions.
  • Schema portability. RR's queries are shaped around its own copy of the data; the SSIS package is the seam that lets RR's schema stay stable while the customer's JDE schema varies (Oracle vs SQL Server JDE, proddta vs jdeprod vs crpdta schema, fiscal-year calendars, decimal-precision dictionaries).

The package is the bridge: it reads JDE through a read-only login (rapidrec/rapidrec by convention), transforms numeric columns into RR-canonical decimal precision via the Dec* divisor variables (JDE stores numeric values as integers with implied decimals; dividing by the appropriate power of 10 reconstitutes the real decimal), and writes into RapidReconciler_Prod where RR data services picks up.

Where the package sits in the install flow

The package is delivered to the GSI DBA as part of the install zip (alongside the four SQL scripts that build the database). During Installing the Production Database, the DBA:

  1. Creates the RapidReconciler_Prod database, the rruser SQL login, and the RapidReconciler_Prod SQL Agent job by running the four install scripts (Step 7).
  2. Opens the package in Visual Studio Community, configures the 6 install-time variables (Step 8 Step 5 of 8) and the 2 OLE DB connection managers (Step 8 Step 4 of 8), and deploys it to the SSISDB catalog at /SSISDB/RapidReconciler/RapidReconciler (Step 8 Step 6–8 of 8).
  3. Points the SQL Agent job's step at the deployed package and enables the schedule (Step 9).
  4. Customer I/T waits for the first scheduled run to populate the database, then verifies via v_diagnostic4_b_to_c_log (Step 11).

From then on the package runs on the SQL Agent schedule (typically nightly during off-peak hours) and the agent serves whatever's in the RR database.

Runtime lifecycle of one job run

Each scheduled run follows the same shape:

  1. Initialize — test both connections, log the package-start banner, read enabled-module flags from usertabnames into ModInv and ModRnv.
  2. Companies — copy business unit / fiscal pattern / currency tables. Compute Julian start date from Gregorian aaStartDateGr.
  3. Inventory — copy the item-ledger (F4111 — cardex), watermarked by maxUKID from prior runs so each run is incremental.
  4. Inventory Supplemental — copy item master, branch, costs, lot, UoM tables. Populate the Cache - Short Items cache.
  5. Orders — copy purchase order and sales order detail, plus the Work Orders sub-container's F3106 and F4801.
  6. General Ledger — copy chart of accounts and GL distributions; populate the Cache - Objects cache from F0901 object codes; use it as a lookup to filter F0902 and F0911 down to inventory-relevant accounts only.
  7. Receipts — copy the F43121 PO Receiver / Voucher Detail table.

Each table copy writes a row to v_diagnostic4_b_to_c_log with the start/end timestamps, row count, and an ErrorNum (0 = success). Step 11 of the install procedure is the customer DBA querying this log to confirm the load was healthy.

SSIS features RR's package deliberately doesn't use

A general-purpose SSIS package can opt into a wide set of platform features (configurations, parameters, checkpoint restart, package-level transactions, digital signatures, custom log providers, multiple event handlers). RR's package uses very few of them — the design intentionally stays on the simple end of the surface. Knowing what isn't here helps an experienced SSIS dev orient quickly to the design boundaries.

SSIS featureRR's packageWhy
Configurations (external property files) Not used The 6 install-configurable variables (dbowner, aaStartDateGr, the four Dec* divisors) are set at design time in Visual Studio and baked into the deployed package. Re-deploy to change. A configurations file would add a moving part the customer DBA has to track on top of the SSISDB-deployed package; the trade-off isn't worth it for 6 values that change at most once per install.
Project / package Parameters Not used Same reason — the install-configurable values live as package-level variables, not project Parameters. Parameters would surface them in the Integration Services Catalog UI for runtime overrides; that's not the install model RR uses.
Checkpoint restart Not used (no CheckpointFileName set) The package's incremental design (maxUKID watermarks, date-window filters) makes a failed run safe to re-run from the top — the next run picks up where the last one left off without re-doing successful steady-state work. Checkpoint files would add complexity (file persistence, cleanup) without buying much.
Package-level transactions (TransactionOption) Default (Supported); no transaction span declared Each data-flow task commits independently to RRLocal through OLE DB Destinations. Durability is per-task at the SQL Server level; the package doesn't enroll multiple tasks into a single distributed transaction. A failure mid-run leaves successful tables loaded and failed tables to retry — aligned with the incremental design above.
Custom Log Providers Not used Logging relies on two layers: SSISDB's built-in execution logging (catalog.execution_data tables) for SSIS engine-level start/end/error events, and the application-level v_diagnostic4_b_to_c_log table written by the package's own SQL tasks for per-table row counts and error codes. A custom Log Provider would duplicate one of those without adding signal.
Multiple event handlers One (OnError at package scope, empty body) Error propagation relies on SSIS's default behavior — any failed task fails its containing container, which fails the package run, which surfaces in SSISDB execution status and in the diagnostic table's ErrorNum. The empty OnError handler is a placeholder for future engineering to wire in alerting (e.g. an email task) without restructuring the package.
Digital signature Not signed The deployed package isn't digitally signed and the SSIS server isn't configured to check signatures. Trust is established by the install procedure (the DBA deploys a package they received from GSI) rather than by signature verification.

Connection managers

The package defines four connection managers. Two are real OLE DB connections (JDE source, RR destination); two are SSIS in-memory cache connections used by the GL/inventory lookup chain. The OLE DB connections are configured by the GSI tech during Step 8 of the install procedure; the cache connections are internal and need no configuration.

JDESource — JDE production source

OLE DB connection to the customer's JDE production database. Read-only by convention — the package never writes back to JDE.

Template connection string (as shipped)

Note: the template ships pointed at a sample customer (jde_treatt) on the local instance with Windows Authentication. The install tech replaces both the Data Source and Initial Catalog with the customer's actual JDE server and database, and switches the authentication to SQL Authentication using the rapidrec/rapidrec read-only login (Step 8 Step 4 of 8 in the install procedure).

Provider note

Both OLE DB connections use the SQL Server Native Client 11 (SQLNCLI11.1) provider. This is the standard SQL Server driver for SSIS in SSDT/VS 2017+, not the JDE-specific OLE DB provider that the SSIS driver mismatch scenario warns about. The JDE-specific provider applies only when the customer's JDE source is on Oracle or AS400 — if the JDE source is on SQL Server (as it is for many JDE-on-SQL customers), SQLNCLI11.1 handles both connections.

Process flow

The package's control flow is 7 top-level sequence containers plus 2 standalone SQL tasks (one disabled by default, one wrapping up the run), chained by precedence constraints at the package level. Each container holds one or more data-flow tasks (and in the case of Initialize, only SQL tasks). Within each data-flow task, an OLE DB Source reads from JDE via JDESource, optional transformations apply, and an OLE DB Destination writes to RRLocal. The top-level executables run in this order:

1. Initialize

Pre-flight container. No data flows — only SQL tasks that prep the run.

1a. Truncate destination tables (disabled by default)

A standalone top-level SQL task sitting between Initialize and Companies in the control flow. Marked disabled in the shipped template — the caption next to it in the VS designer reads: “Leave disabled unless backing up the start date after the initial load.”

The use case is rare: if a customer needs to re-load history from an earlier aaStartDateGr than they initially configured, simply running the package with the new earlier date isn't enough — the incremental-watermark logic (especially maxUKID for F4111) would prevent re-pulling older transactions. Enabling this task wipes the RR-local destination tables so the next run rebuilds from scratch using the new (earlier) start date. Don't enable for routine runs.

Key mechanisms

Three mechanisms that drive the package's behavior and that anyone diagnosing a failure needs to understand: the dbowner string-concat pattern, the per-task Dates and Decimals Derived Column transform (where Julian dates become Gregorian and JDE-scaled numerics get divided into real decimals), and the cache & lookup chain that filters GL accounts down to inventory-relevant ones.

The dbowner string-concat pattern

Every source query in the package is built dynamically by string-concatenating the dbowner variable into a SQL command. The literal query text isn't stored anywhere; what's stored is the expression that builds it.

Example: the qryF0006 expression

Each data-flow task has a qry<table> variable whose EvaluateAsExpression property is true. For F0006, that expression is:

"Select MCCO, MCMCU, MCMCUS, MCDL01, MCRP01..MCRP30 FROM " + @[User::dbowner] + "F0006 Where MCMCU != ' ' "

At runtime SSIS evaluates this expression by substituting the current dbowner value:

  • If dbowner = "proddta.": "Select … FROM proddta.F0006 Where …" — correct.
  • If dbowner = "proddta" (no trailing period): "Select … FROM proddtaF0006 Where …" — an object that doesn't exist.

The OLE DB Source component then sends the resolved query to JDESource. If the table doesn't exist, OLE DB returns zero rows but does not raise an error — SSIS treats "no matching rows" as a successful query. The data-flow task completes, UpdateCount is logged as 0, and ErrorNum is 0. From the diagnostic table's perspective the run looks healthy; from the customer's perspective no data ever shows up in RR.

!
This is why forgetting the trailing period is the #1 install failure mode

The package fails silently and the diagnostic doesn't surface it as an error. See SSIS package deployed but extracts no data for the dedicated runbook.

The Dates and Decimals Derived Column transform

Every data-flow task that copies a JDE table with date columns or scaled-numeric columns includes a Derived Column transformation named Dates and Decimals wired between the OLE DB Source and the OLE DB Destination. This is where both the Julian-to-Gregorian date conversion and the decimal-divisor application happen, in a single transform per task.

The transform adds new columns to the data-flow buffer rather than overwriting the originals — output columns get a prefix indicating their type:

  • Grg prefix — Gregorian datetime version of a JDE Julian date column. E.g. strILDGL (the Julian string from F4111) becomes GrgILDGL (a DT_DBTIMESTAMP).
  • flt prefix — double-precision float version of a JDE-scaled numeric column, with the appropriate divisor already applied. E.g. ILTRQT becomes fltILTRQT (a DT_R8).

The destination then writes both the original JDE-format columns and the derived columns into the RR-local table — preserving the source data for traceability while giving downstream queries readable date/decimal columns to work with.

Julian-to-Gregorian conversion expression

Each Julian date column is converted by a SSIS expression of this shape (taken from the F4111 task, where strILDGL is the Julian string and GrgILDGL is the derived Gregorian datetime):

DATEADD("Day",
        (DT_I4)(SUBSTRING(strILDGL, 4, 3)) - 1,
        DATEADD("Year",
                (DT_I4)((LEN(strILDGL) == 5)
                        ? SUBSTRING(strILDGL, 1, 2)
                        : SUBSTRING(strILDGL, 1, 3)),
                (DT_DBDATE)"1900-01-01"))

Read inside-out:

  1. Start with the anchor date 1900-01-01.
  2. Extract the year portion of the Julian string. JDE Julian dates are 5 or 6 characters: 5-char form YYDDD for 20th-century dates (year portion is the first 2 chars), 6-char form CYYDDD for 21st-century dates (year portion is the first 3 chars, with the leading 1 century flag included). The conditional (LEN == 5) ? SUBSTRING(1,2) : SUBSTRING(1,3) picks the right slice for each.
  3. Add that year portion to 1900-01-01 via DATEADD("Year", …). The 5-char case adds a 2-digit value (e.g. 26 → 1926); the 6-char case adds a 3-digit value (e.g. 126 → 2026). The encoding does the century math for free — no century-flag conditional needed.
  4. Extract the day-of-year portion (always the last 3 chars: SUBSTRING(4, 3)), subtract 1 to align with the Jan-1-is-day-1 convention, and add via DATEADD("Day", …).

Worked example for 1 May 2026:

  • JDE Julian: 126121 (6 chars).
  • Year portion: SUBSTRING("126121", 1, 3) = 126.
  • Day portion: SUBSTRING("126121", 4, 3) = 121.
  • Year add: 1900-01-01 + 126 years = 2026-01-01.
  • Day add: 2026-01-01 + 120 days = 2026-05-01. ✓

The same expression shape is used for every Grg* derived column — only the source-column name changes. Date columns this conversion is applied to include F4111 ILCRDJ + ILDGL; F4108 IOMMEJ, IOUPMJ, IOBBDJ, IOBODJ, IODLEJ, IOOHDJ, IOSBDJ; F0008 CDDFYJ, CDD01J–CDD14J; F0911 GLDGJ; F3106 SDDGJ, SDDICJ; F4801 WADRQJ; F4102 IBUPMJ; and others.

Decimal divisor application

The same Derived Column transform also applies the Dec* divisor variables to JDE-scaled numeric columns, again via a simple expression per column. Examples from the F4111 task:

fltILNID  =  (DT_R8) ILLNID  / 1000
fltILTRQT =  (DT_R8) ILTRQT / @[User::DecQty]
fltILPAID =  (DT_R8) ILPAID / @[User::DecExtCost]
fltILUNCS =  (DT_R8) ILUNCS / @[User::DecUnitCost]

Each source column is cast to DT_R8 (double-precision float) and divided by either:

  • A package variable — the four customer-configurable Dec* divisors (DecQty, DecQtyCX, DecExtCost, DecUnitCost) for the columns whose decimal precision varies between customers (cost and quantity columns), or
  • A hardcoded constant — e.g. ILLNID / 1000 for the line ID number, which always has 3 implied decimals in JDE regardless of customer.

Whether a given numeric column uses a variable or a hardcoded divisor depends on the column itself: JDE data-dictionary fields that customers can customize (cost / quantity precision) use the variables; structural fields whose precision is fixed by JDE (line IDs, batch numbers, etc.) use constants.

i
Internal note: aaStartDateGr is not a per-row conversion

The aaStartDateGr package variable is in Gregorian YYYY-MM-DD form, and at runtime the package computes a Julian equivalent from it via the qryJulianStart expression in the Companies container, which then feeds the F4111 cardex filter WHERE ILCRDJ >= @[User::startdatejul]. This is a one-time runtime computation that produces a single filter value — not a row-by-row data conversion. It's documented under the Variables reference for the aaStartDateGr entry; the only operational consequence here is that aaStartDateGr must stay in YYYY-MM-DD form for SQL Server's datetime parser to accept it.

Cache & lookup chain (GL filtering)

RR cares about a specific slice of the customer's chart of accounts — the accounts that touch inventory. Pulling all of F0902 and F0911 unfiltered would be wasteful (chart-of-accounts can have tens of thousands of GL accounts; only a fraction touch inventory). The package's design uses an in-memory cache to filter:

  1. Populate: The Load Objects data-flow task in the General Ledger container reads GMOBJ values from F0901 (whose own filter restricts to the relevant chart-of-accounts subset) and writes them to Cache - Objects.
  2. Consume in F0902: The Copy F0902 task includes a Lookup Transform (Filter out non inventory accounts) that joins each F0902 row against Cache - Objects on the object code. Rows whose object code isn't in the cache are routed to the no-match output (which has no downstream destination), effectively filtering them out.
  3. Consume in F0911: The Copy F0911 task has the same Lookup Transform pattern (Filter out unused accounts), filtering GL distribution rows to those that hit cached object codes.

The cache lives only in memory for the duration of one package run — it's not persisted between runs. Each scheduled execution rebuilds the cache from F0901 first, then uses it for downstream filtering.

i
Why this matters for diagnostics

If F0901 is missing or empty in the resolved query (e.g. due to a dbowner issue), the cache is empty, the lookups match nothing, and downstream F0902 and F0911 also load zero rows. The dbowner symptom isn't just "F0006 is empty" — it cascades into "the entire General Ledger container is empty" via the cache.

Cache - Short Items: same pattern, different scope

The Cache - Short Items cache follows the same populate-then-lookup pattern as Cache - Objects, but in service of a different filtering job:

  1. Populate: The Short Items data flow in the Inventory Supplemental container runs SELECT DISTINCT ibitm FROM F4102 and writes each distinct item number into Cache - Short Items. F4102 (Item Branch File) is the set of "item is set up at this branch" records, so the cache effectively holds the list of items that exist in some branch.
  2. Consume in F4101: The Copy F4101 task has a Lookup Transform that joins each F4101 row against Cache - Short Items on the item number. Rows whose item number isn't in the cache are routed away from the destination, dropping master records for items that don't appear in any branch.

Same cache lifecycle as Cache - Objects — in-memory only, rebuilt each run. The motivation is performance: a customer's F4101 Item Master can be enormous (everything they've ever stocked or planned to stock), and most of those rows aren't relevant to RR's reconciliation. Filtering down to the items that are present in F4102 keeps the RR-local Item Master at a usable size without losing anything the application cares about.

How the package gets from the install zip to running in production on the customer's SQL Agent schedule. The install procedure has the full step-by-step; this section is the operational summary plus the verification path.

The customer's DBA performs the deployment from a workstation that can reach both the source .dtsx file (the install zip) and the target SQL Server hosting the SSIS catalog. The workstation needs Visual Studio Community with the SQL Server Integration Services Projects extension installed (free download from Microsoft Marketplace; both 2019 and 2022 versions of VS work).

  1. Open in Visual Studio. Create a new Integration Services Project named RapidReconciler; add the existing package RapidReconciler_Prod.dtsx via right-click Add Existing Package.
  2. Configure connection managers. Update JDESource with the customer's JDE server/database/credentials; update RRLocal with the RR-local server and the rruser credentials. Test both connections before proceeding.
  3. Configure variables. Set the 6 install-configurable variables in the Variables window per the customer's pre-install submission email (dbowner, aaStartDateGr, the four Dec* precision divisors).
  4. Deploy to SSISDB. Right-click the project → Deploy. The Integration Services Deployment Wizard opens; target is the customer's SQL Server with Windows Authentication (Step 1 of the wizard), and the catalog path is /SSISDB/RapidReconciler/RapidReconciler. The deploying account needs sysadmin or ssis_admin on the target instance.
  5. Verify deployment landed. SSMS → Integration Services Catalogs → SSISDB → RapidReconciler → right-click the package — confirm the timestamp reflects the just-completed deployment.
!
Re-deployments must re-configure

SSIS doesn't persist variable values or connection-string parameters across re-deployments by default — if a re-deployment is needed (e.g. after fixing the dbowner variable), the values configured at design time are what get deployed. Save the VS project before deploying so the variables persist across sessions.

Verification — the diagnostic table

Every data-flow task writes a row to v_diagnostic4_b_to_c_log in RapidReconciler_Prod with the run timestamp, row counts, and error code. After each scheduled run, the customer DBA queries this view to confirm the load was healthy:

SELECT Capture, Step, Process,
       UpdateCount, ErrorNum, ErrorMsg
FROM   v_diagnostic4_b_to_c_log
WHERE  Step = 'data capture'
ORDER  BY Capture DESC

Healthy run characteristics:

  • One row per JDE table copied at Step = 'data capture'.
  • ErrorNum = 0 on every row.
  • UpdateCount non-zero for active tables (F4111, F0911, F4111, F4101, F4102 for typical mid-volume customers). Some tables may legitimately show zero (e.g. F4801 if the customer doesn't use work orders).

If any of these is false, see Initial load reports errors or zero counts.

Incremental vs. initial-load behavior

The package is designed to run incrementally: subsequent runs pull only new transactions since the last successful run, using maxUKID watermarks for F4111 and date-window filters for F0911/F4801/F3106. This keeps each run fast (typically minutes for steady-state load).

The first run after install is the exception: there's no prior maxUKID and the date windows are wide (aaStartDateGr is the 2-months-prior-to-fiscal-year-start anchor). This first run does a near-full historical pull and can take hours for high-volume customers. After the first successful run, every subsequent run is incremental.

Some tasks (Copy F0911 Init, Copy F41002 Init1, Copy F43121 Init) are initial-load variants that run only when InitLoad = 1. Engineering can manually flip InitLoad to 1 to force a full reload of those tables outside the normal schedule; it's not part of the standard install procedure.

Logging & error handling

The package uses verbose logging (DTS:LoggingMode="2"). All task starts/ends, variable changes, and errors are emitted to the SSIS server's standard log destinations (typically SSISDB's catalog.execution_data tables).

The package has one event handler defined — OnError at package scope — but its body is empty. Error propagation relies on SSIS's default behavior: any task that errors propagates up the precedence-constraint chain and fails the containing container, ultimately failing the package run. The OnError handler structure exists for future engineering to wire in (e.g. an alert email or external monitoring hook) without re-architecting the package.

Troubleshooting

This package has its own dedicated set of install-scenario runbooks under the install troubleshooting reference. Each is keyed on a specific failure symptom and walks the diagnostic path step by step.

Primary SSIS-specific scenarios

SQL script fails during database install. One of the four install scripts (creation, objects, rruser, SQL Agent job) errors in SSMS. Six-check runbook organized by which script failed. Common causes per script: CREATE FILE encountered operating system error, Database already exists, Login failed, Login already exists, Password validation failed, SQL Server Agent service not running, job already used. SSIS package deployed but extracts no data. Package deploys cleanly, SQL Agent job runs to completion, ErrorNum = 0, but UpdateCount = 0 across every major table. Almost always the dbowner trailing-period issue. Five-check runbook with the actual qryF0006 expression and the OLE DB "no-rows = success" mechanism explained. Initial load reports errors or zero counts. Step 11 verification finds non-zero ErrorNum or zero UpdateCount for major tables. Seven-check runbook with five pattern-matched diagnoses: zero-count-no-error (the dbowner case), login/auth failure, invalid-object-name (schema mismatch), timeout, partial counts.

Deployment-specific scenarios

SSIS catalog deployment fails — permission denied. Deployment from Visual Studio errors before package upload completes. Three checks: Windows Authentication vs SQL Authentication, deploying account privileges (sysadmin / ssis_admin), SSISDB catalog existence. SSIS package deployment fails — OLE DB driver error. Deployment-time error referencing a missing or unrecognized OLE DB provider for JDE. Common with JDE-on-Oracle or JDE-on-AS400 customers where the JDE-specific driver wasn't installed in both 32- and 64-bit on the SSIS host.

Related runtime scenarios

Companies tab is empty after install. Database is online but the Companies tab in the RR UI shows no companies. Often a SQL Agent job that hasn't run yet, or an SSIS source connection issue that's not the dbowner case. Post-deployment JDE import fails — rruser login error. Customer-side admin's production deployment encounters an rruser authentication failure during the import. Five-check runbook covering hidden rruser references in package internals (data-flow source credentials, Execute SQL Task connections, internal connection managers) that the deployment UI doesn't expose. Migration complete but data not flowing. Post-migration to a new server, customer can reach the RR UI but no new data appears. Often SQL Agent schedule not enabled on the new server, or SSIS package connection strings still pointing at the old server.

Design notes

i
F1113 and F0015 — why both exist in the package

Both F1113 and F0015 are JDE currency-conversion-rate tables. Neither is functionally required by RapidReconciler — RR states everything in the company's base currency, so multi-currency conversion isn't part of the reconciliation. The tables are pulled (F1113 enabled, F0015 disabled) for completeness and possible future use; no downstream RR logic depends on either. If a future install needs to drop them entirely to slim the extract, both can be disabled safely.

i
Cache - Short Items — what it filters and why

Performance optimization. F4101 (Item Master) and F4102 (Item Branch) can be enormous — many JDE customers have item-master rows for products they no longer stock or never stocked at any branch. RR only needs items that are actually used, so the package builds Cache - Short Items from the distinct list of item numbers found in F4102 (and conceptually corresponds to the set of items active in F4111 cardex / F41021 location records), then uses it as a Lookup-Transform filter in the F4101 copy task to drop master rows for items that aren't in the cache. The result is a much smaller, app-relevant F4101 in RapidReconciler_Prod.

i
F4096 variants — FA_1RT (new, default) vs FA@RT (old, fallback)

Two parallel F4096 data-flow tasks exist because Oracle renamed an F4096 column during a JD Edwards upgrade several years ago: the column is FA_1RT on newer JDE versions and FA@RT on older versions. The package ships with F4096_FA1RT (new column name) enabled by default; F4096_FA@RT (old column name) is disabled by default but kept in place to support installations still running an older JDE that wasn't migrated. If the F4096 copy errors out with a missing-column error on FA_1RT, the installer disables F4096_FA1RT and enables F4096_FA@RT to fall back to the legacy column name.
