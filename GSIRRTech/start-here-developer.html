Orientation

What the role is — and what it isn’t

You author RapidReconciler at the schema and code level. You don’t install it for individual customers, you don’t troubleshoot their environments, and you don’t hold their hand through migrations — that’s RR DBA and RR Network Tech. Your work is upstream of theirs: every install zip they download, every SQL update they push, every SSIS package configuration they touch — you wrote it.

What you own

  • The install zip. The customer-distributed package containing the four installation scripts and the SSIS package. Versioned and updated periodically.
  • Schema changes and patches. SQL update scripts that get applied across the live customer base via the VALC SQL Scripts page.
  • The SSIS package source. RapidReconciler-Prod.dtsx — the SSIS package the DBA configures and deploys at every install.
  • Escalation target for product-code issues. When RR DBA or RR Network Tech hit a failure that points at the schema or the product (not at customer-environment configuration), they come to you.

What you don’t own

  • Individual customer installs — that’s RR DBA.
  • VALC client setup, agent install, customer handoff — that’s RR Network Tech.
  • Mailbox triage at rrsupport@getgsi.com — that’s the RR Support Monitor.
  • Customer-environment troubleshooting (driver mismatches, firewall issues, DNS propagation) — that’s the front-line tech roles.

This guide doesn’t cover specifics like your local development environment (IDE choice, source-control tooling, branching strategy), the release process for cutting a new install-zip version, or how escalations actually reach you (Slack channel, ticket queue, email). Confirm those with your manager before working on a real change — they’re team-specific and not in the source documentation.

Where you fit in the workflow

The install workflow has four GSI roles working in sequence: RR DBA handles intake and the database side; RR Network Tech picks up VALC and takes the customer live; RR Support Monitor watches the mailbox and routes between them. RR DB Developer is the role none of those three loops in for normal installs — you sit one layer up.

You become directly relevant in three situations:

  1. A new install zip is needed — the schema has been updated, a fix has been merged, and the next set of installs should pick up the new version. You cut the new zip and stage it at files.getgsi.com; the next DBA who runs an install downloads it fresh per the “always download a fresh copy” rule in the install procedure.
  2. An update needs to be pushed across existing customers — e.g. a bug fix, a new view, a schema migration. You apply it via the VALC SQL Scripts page (covered below).
  3. An install or migration fails in a way that points at the product rather than the customer environment. RR DBA or RR Network Tech escalates to you for diagnosis.

For broader context on how the four tech roles relate, see the Roles & Responsibilities doc — particularly the “Hand-offs in” sections of the RR DBA and RR Network Tech profile cards. They both list “escalation to RR DB Developer” as a hand-off path.

Part 1 · Tool Tour

The VALC SQL Scripts page

The SQL Scripts page is your primary operational tool — the only place in VALC where you, as Developer, do customer-facing work. It lets you apply update scripts (schema changes, data fixes, patches) across customer databases without logging in to each customer’s SQL Server directly.

The page is reserved for RR DB developers. Other tech-team roles can see it exists but don’t use it. Working through it confidently is the single most important capability of this onboarding guide.

Page layout at a glance

The SQL Scripts page divides into four working areas. The next four sections cover each area in turn, then a final section walks the end-to-end execution flow.

  • Database list (left) — every database registered across every customer client in VALC, with a checkbox per row.
  • Filters (top-left) — Database Selection (by Category) and Revision Number (freeform text).
  • Script editor (top-right) — Choose File to load a .sql file, review its contents, then EXECUTE.
  • Results panel (bottom-right) — pass/fail line per database after execution completes.
Area 1

Database list (left)

The database list shows every RapidReconciler database registered across every customer in VALC. Each row is one database, formatted as:

{ClientName} - {DatabaseName} - {RevisionNumber}

For example: Acme Mfg - RapidReconciler_Prod - 178 — the Production database for Acme Manufacturing currently at schema revision 178.

How to use it: check the box next to each database you want to target with a script. You can multi-select. The list can be long — this is where the filters come in.

Why each row shows the revision number: different customers may be at different revisions if updates have been rolled out in batches or some customers have been skipped (e.g. on extended maintenance windows). Knowing the revision before pushing a script tells you whether the script applies cleanly or whether the database needs an interim update first.

Area 2

Filters (top-left)

Two filters narrow the database list so you don’t hunt or accidentally check the wrong row.

Database Selection (Category)

Filters by the database’s Category — a tag set when the database is registered in VALC and surfaced here for filtering. Five values:

Production Test Archive Change Control Inactive

Category doesn’t affect database behavior — it’s purely an organizational tag. Production is the live customer database; Test/Dev/Train are siblings used by customers who’ve added them to their setup; Change Control and Archive are used as configured by the team.

Revision Number

Freeform text field — type any portion of a revision number to filter the list to matching rows. Useful for targeting only customers at a specific schema version when an update applies cleanly only to certain revisions.

Area 3

Script editor (top-right)

Where you load and review the script before running it.

  • Choose File — opens a file browser. Select the .sql file you want to apply. Once loaded, its contents appear in the dark editor area below.
  • Editor area — read-only display of the loaded script. Always read the script here before clicking EXECUTE, even if you trust the source. This is your last review checkpoint before the script touches every selected database.
  • EXECUTE — runs the script against every database checked in the database list. There is no per-database confirmation; clicking EXECUTE commits to the full batch.
Area 4

Results panel (bottom-right)

After execution completes, the Results panel populates with a pass/fail line per database in the batch.

What “pass” means: the script executed against that database without throwing an error. It does not mean the script did what you intended — only that it ran cleanly. For schema migrations, follow up with a verification query against a sample of the targeted databases.

What “fail” means: something went wrong on that database. Investigate before re-running — the failure mode for re-executing a partially-applied script is making the problem worse, not fixing it. The Operational Discipline section below covers what to do.

Flow

End-to-end execution flow

The five-step flow when applying any update script:

  1. Filter the list using Category and/or Revision Number so you’re only looking at the rows you might target.
  2. Select databases by checking the boxes. Limit to 5 at a time — covered in the discipline section.
  3. Load the script via Choose File and review the contents in the editor area.
  4. Click EXECUTE and wait for the run to complete.
  5. Review the Results panel — pass/fail per database. Investigate any failures before re-running.
Part 2 · Artifact Tour

The artifacts you maintain

You own three things in source: the install zip (a packaging concern), the four installation scripts inside it (the install procedure’s building blocks), and the SSIS package (data-extraction logic). Each has a different update cadence and a different set of consumers.

Install zip — RapidReconciler-Install-Latest.zip

Distribution

The customer-distributed package every new RR install starts from. Staged at https://files.getgsi.com/rapidreconciler/install/RapidReconciler-Install-Latest.zip and downloaded fresh by the RR DBA at the start of each install meeting.

The install procedure explicitly tells DBAs to always download a fresh copy at the start of each install rather than reusing a copy from a prior install — using an out-of-date zip means installing an out-of-date schema. That “always fresh” rule is the contract between you and them: when you stage a new version of the zip, every install from that moment forward picks it up.

Your role: cut new versions when the contained scripts or SSIS package change. Stage the new zip at the canonical URL. Maintain backwards-compatibility considerations for in-flight customer installs.

The four installation scripts

Schema & setup

The four .sql files inside the install zip. They run in sequence during the database install (DB Install Step 7) and create the database, every schema object, the rruser SQL login, and the RapidReconciler_Prod SQL Agent job.

File What it creates Notes
1 - RapidReconciler Database Creation Script.sql The RapidReconciler_Prod database itself, with the .mdf/.ldf files in default locations and an initial 5 GB size. Hard-coded for Prod. Test/Dev/Train databases must be created via backup-restore from Prod — never by re-running this script with a different name.
2 - RapidReconciler Database Object Script {N}.sql All schema objects: tables, views, stored procedures. The {N} in the filename is the schema revision (e.g. 178). This is the file you’ll touch most often when the schema evolves.
3 - RapidReconciler User Creation Script.sql The rruser SQL login plus the database-level user mapping with db_datareader, db_datawriter, and SQLAgentOperatorRole. Mixed-mode SQL authentication required. The default password is rruser; customers may change it.
4 - RapidReconciler SQL Agent Job Creation Script.sql The RapidReconciler_Prod SQL Server Agent job. Job’s package path is empty at creation — the DBA points it at the deployed SSIS package later in the install.
Your role: author and maintain all four. The Object Script is versioned (file name includes the revision number); the others are stable but evolve when authentication, role memberships, or job structure changes.

RapidReconciler-Prod.dtsx — the SSIS package

Data extraction

The SSIS package the DBA configures and deploys to the SSIS catalog at every install (DB Install Step 8). It pulls data from the customer’s JD Edwards source into RapidReconciler_Prod on a schedule.

The DBA configures it per customer — updating the JDE source connection manager, the RR destination connection manager, and these variables:

  • aaStartDateGr — data-extraction start date (set to 2 months prior to current fiscal year)
  • dbowner — JDE table qualifier (e.g. proddta.) with the trailing period — the most common SSIS misconfiguration
  • DecExtCost, DecUnitCost, DecQTY, DecQtyCX — decimal places for cost and quantity columns, sourced from the customer’s JDE data dictionary (ECST, UNCS, PQOH, TRQT)
Your role: author and maintain the package. The DBA configures customer-specific values, but the control flow, data-flow tasks, transformations, and underlying SQL are yours.

Versioning

Reference

The Database Object Script is the canonical version stamp — its filename includes the schema revision number (e.g. 178). The same revision number surfaces in VALC’s database list ({ClientName} - {DatabaseName} - {RevisionNumber}) so the SQL Scripts page filter can target databases at a specific revision.

Conceptually: customer database revision = the schema version that was last applied to it. Push an update script via SQL Scripts and the database’s revision advances. Customers running at older revisions are visible from the SQL Scripts filter, which is how you find “everyone still on 175 that needs the 178 update.”

Part 3 · Operational Discipline

The five rules that keep multi-customer rollouts safe

The SQL Scripts page is powerful enough to break every customer at once. The discipline below is what keeps that from happening. None of these rules is novel — they’re standard practice for any environment where one mistake fans out to many tenants — but operating the SQL Scripts page without internalising them is the failure mode this section exists to prevent.

Rule 1

Always review the script in the editor before clicking EXECUTE

Even when you trust the source. Even when you wrote it yourself five minutes ago. Even when it’s the third time you’re running “the same” script today.

The editor area exists for exactly this purpose: it’s the last visible checkpoint before the script touches every selected database. Skipping the review is how the wrong file gets executed (and the wrong file looks identical to the right file in a file browser).

Rule 2

Run scripts against Test databases first when possible

Customer Test, Dev, and Training databases are visible alongside Production in the database list. Use the Category filter to target Test rows specifically — run the script against those, verify behaviour, then promote to Production once the result is what you expect.

Not every customer has a Test database, so “when possible” is the qualifier. For customers with Production-only setups, use a known-friendly customer’s Test database as a proxy if the script’s behaviour is portable.

Rule 3

Limit each execution to 5 databases at a time

The recommended batch size in the SQL Scripts documentation is 5. The reasoning:

  • Smaller batches mean smaller blast radius if the script fails on a database with unexpected state.
  • The Results panel is more readable with 5 lines than with 50.
  • Rollback is tractable for a small batch — running through 5 databases manually to verify or revert is achievable; running through 50 is not.

For larger rollouts, batch the work into multiple executions. Yes, this means the SQL Scripts page is intentionally a slow tool for large rollouts. That’s a feature, not a bug.

Rule 4

If a script fails on one or more databases, investigate before re-running

Re-executing a partially-applied script can compound the problem. If a schema migration created an index on three databases before failing on the fourth, re-running the script means it tries to re-create the index on the three that succeeded — and now you have two failures plus one new one.

The investigation:

  • Read the failure message in the Results panel.
  • Look at the failing databases — what’s different about them? Different revision? Larger data volume? Customer-specific schema modification?
  • Decide whether the right move is fixing the script, fixing the failed databases manually, or rolling back the partial application before retrying.
  • Only re-run after you can articulate why the second run will succeed where the first didn’t.
Rule 5

Receive escalations from RR DBA and RR Network Tech with the right framing

The other tech roles escalate to you when a failure points at the schema or product code rather than at customer-environment configuration. Examples:

  • RR DBA: “The Object Script 178 fails at this customer with a constraint violation that doesn’t happen on others.”
  • RR Network Tech: “The Companies tab in VALC is empty even though the database is Online — the SSIS package extracts no data.”

When you receive these:

  • Ask for the agent log file if the symptom is at the agent layer (RR Network Tech’s Troubleshooting reference covers what to grab).
  • Ask for the customer’s revision number — visible in VALC’s database list.
  • Reproduce against a Test database at the same revision before changing anything in Production.
  • If a hotfix is needed, decide whether it goes out as a one-off SQL Scripts execution against the affected customer, or whether it needs a new install-zip revision because it affects the schema baseline.

Wrap-up

What to do next

You’ve walked the tool, the artifacts, and the discipline. Three suggested next steps before you push your first script:

  1. Open the VALC SQL Scripts page in read-only mode and explore. Filter by Category, scan the database list, look at the revision-number distribution across the customer base. Don’t check any boxes — just learn the lay of the land.
  2. Read the install procedures top-to-bottom. Installing the Production Database uses the four installation scripts; Installing a Client in VALC registers the resulting database in VALC. Knowing how the install consumes your artifacts means knowing what to be careful about when you change them.
  3. Pair on the first real change. Schedule time with another developer (or your manager) to walk through your first SQL Scripts execution before doing it solo. The ritual of reviewing a script with someone else’s eyes is the discipline you’ll want forever — it just takes one pairing session to build the habit.
