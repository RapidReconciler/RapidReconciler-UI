Sanitisation checklist · work through this before publishing
  • Replace the customer's company name everywhere with a pseudonym (e.g. Acme Corp, Beta Industries, Gamma Holdings). Never the real name.
  • Replace customer contact names with first-initial + last-initial or short pseudonym (P. Flores, J. Smith). Avoid real first names on the public site.
  • Replace customer email addresses with name@<customer-pseudonym>.example — the .example TLD is reserved by IANA and never resolves.
  • Replace any internal GSI hostnames with patterns: <gsi-backend>.cloudapp.net, <customer-host>.getgsi.com. Never publish a real hostname.
  • Replace IP addresses with <azure-ip> or <customer-ip>. Never publish a real IP.
  • Replace port numbers with <port> if the port is internally meaningful, otherwise leave generic ports (80/443) as-is.
  • Replace ticket numbers with INC<XXXXXXX>, RS-<XXXX>, etc.
  • Replace exact dates with the date only (drop time-of-day if the case is older than a month). Replace specific times in the walkthrough with relative timestamps (9:47 AM is fine; the actual day is what gets fuzzed).
  • If a screenshot is attached, replace it with a sanitised description in the .attachment block — do not embed the original image.
  • Re-read the whole page once more. Search the file for any remaining instances of the real customer name — it's the most common slip.
  • Register the case on walked-tickets.html. Add a new <a class="case-card"> entry, copying Case 001 as your model. Fill in the symptom, blurb, and meta fields — and a data-search attribute. Include error phrases, technical terms (host fragments, error wording), the customer-facing symptom in plain English, the path number, and any role names. The data-search string is what the troubleshooting page's drop-the-email matcher scores against to decide whether your case should surface for an incoming ticket. Without it the case is invisible to the matcher.
