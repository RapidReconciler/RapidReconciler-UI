Case 003 · 2025-03-07 · Path 2 (live) · No card matched

Post-deployment JDE import fails — rruser login error.

A customer-side admin has just deployed RR to production. The deployment itself reported success, but running Import JDE data errors out a few minutes in with what looks like an rruser authentication failure. The first Path 2 attempt — "update your connection passwords" — was correct in principle but pointed at the wrong layer. The customer updated the deployment-time connections; the actual culprit was a separate connection string buried inside the SSIS package itself. The fix needed a live Teams session to walk the customer to the right place.

Section 2

The walkthrough — two email rounds, then a live session.

Day 1
9:15 AM
arrives

Read the first inbound. Pattern-match to the usual post-deployment culprit.

Powell's first email (not shown above — that was the followup) describes a fresh production deployment of RR. Deployment itself reported success. Powell ran Import JDE data, watched it process for a few minutes, and got an authentication-style error. Connection settings looked right at deploy time.

This is a recognisable post-deployment pattern. SSIS connection-manager passwords can get blanked out when a package is moved between environments or when the package's protection level doesn't preserve sensitive data across the redeploy. The standard first-line fix is "update the connection passwords and redeploy." It works often enough to be worth trying as Path 2 before going deeper.

Day 1
9:23 AM

Reply with the standard first-line fix.

I write the Path 2 reply. Looking back, the wording is more vague than it should have been — it says "update your connection settings" without specifying that there are two layers (deployment-time connections that the customer can edit in the deployment wizard, AND the package-internal connections that live inside the .dtsx file). The lesson surfaces in retrospect.

To:C. Powell
Subject:RE: RR Question
Hi, Thanks for the deployment notes. The error you're seeing on the JDE import after a successful deployment is most often a connection-password issue: SSIS connections in the RR package can have their passwords blanked out when the package is moved between environments or its protection level doesn't preserve sensitive data across redeploy. The fix is to go into the connection settings, re-enter the password on each connection, save, and redeploy the package. Take a database backup first if you haven't already. Once the package is redeployed cleanly, re-run "Import JDE data" — that should resolve it. Let me know how it goes. Thanks, Sam GSI Helpdesk
Day 2
7:09 PM

Followup arrives. The diagnostic gold is buried in the second-to-last sentence.

Powell did everything I suggested — updated server and database connection values, redeployed cleanly, took a backup — and the import still fails. The followup is shown above; the line that matters is: "the error log points at an rruser login failure, which is confusing me — I never explicitly configured an rruser account."

That's the diagnostic signal. rruser isn't an account the customer is supposed to know about — it's configured by GSI inside the SSIS package itself, in a Connection Manager that's separate from the deployment-time connections the customer just edited. When a customer says "I never set up rruser," they're telling you they're looking at the wrong layer. The fix is the same as I already recommended — update the connection password — but it has to be applied inside the package, in Visual Studio, not in the deployment wizard.

Powell is technical enough to drive Visual Studio with me on the call.

Section 3

Resolution & lessons — what the live session uncovered, what made this case different.

What the live session uncovered

The rruser connection's password was empty inside the SSIS package — one layer below where the customer was looking.

An RR package has two distinct sets of connections. The deployment-time connections (server name, database name, credentials configured in the deployment wizard) are what most customers find first — the customer had updated these correctly. The package-internal connections live inside the .dtsx file and are visible only when opening the package in Visual Studio's Connection Managers pane. The rruser entry there had its password field cleared — a known SSIS quirk that happens when a package's protection level doesn't preserve sensitive data across moves or redeploys. Re-entering the password, saving the package, and redeploying made the JDE import complete cleanly. The fix was structurally identical to Sam's first email recommendation; it just needed to be applied at the right layer, which the customer couldn't reach without screen-share guidance.

What this case taught us
  • An RR package has two layers of "connections," and Tier 1 needs to specify which one explicitly. The deployment-time connections (visible in the deployment wizard) and the package-internal connections (visible only inside Visual Studio's Connection Managers pane) are different. When recommending "update your connection passwords," the reply should call out both layers from the start. The first email in this case was vague enough that the customer fixed only the visible layer. Updated reply template: when post-deployment auth issues are reported, mention both layers in the first response.
  • "I never configured an rruser account" is the diagnostic signal. rruser is GSI's, configured inside the SSIS package; customers don't (and shouldn't) configure it directly. When a customer says they don't recognise the account, they're telling you they're looking at the wrong layer. Recognise the signal and pivot to package internals immediately — either with explicit step-by-step guidance or, more reliably, a live session.
  • Post-deployment auth failures with a few-minutes delay almost always mean SSIS package internals. Initial deployment-time connections succeed quickly — the deployment itself reports success, the database is reachable. The failure shows up minutes into actual data-import processing because that's when the package's internal connections (including rruser) are exercised. If a customer reports a fast-success-then-delayed-failure pattern after a deployment, jump straight to package internals; don't waste a round on deployment-time connections.
  • When email-based Path 2 fails, escalate to live before sending a third round. Sam's second reply could have been a long step-by-step "open Visual Studio, find Connection Managers, etc." but the right call was to suggest Teams. The hands-on diagnosis costs less of everyone's time than another back-and-forth email round. Keep this trigger in mind: if the first Path 2 attempt fails AND the customer's followup contains a signal they're missing GSI-side context (like "I never set up rruser"), escalate to live.
  • Documentation gaps surfaced. Provisional new Tier 1 scenario card: "Post-deployment JDE import fails — rruser login error → SSIS package's rruser Connection Manager needs password updated, package redeployed." Provisional new reply template: post-deployment-rruser-package-connection — should mention both connection layers explicitly so the customer doesn't have to come back for round two.
