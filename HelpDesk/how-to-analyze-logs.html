How-to guide

How to analyze a log.

Where to find each log type, how to read the verdict, and what to do next — about a minute.

The Log Analyzer accepts two kinds of input: browser console output from RapidReconciler in your browser, or the RR agent log file from the application server. Paste or upload either; the tool classifies every line and returns a verdict.

Part 1Where to find each log

Browser console

When something looks wrong in the RapidReconciler UI — a button doesn’t respond, a page renders blank, an error toast appears — the answer is usually in your browser’s console. Capture it from the Inventory Reconciliation page, which is where most of the agent / server interaction surfaces.

  1. Log into RapidReconciler and navigate to the Inventory Reconciliation page.
  2. Press F12 (or right-click → Inspect) to open DevTools, then click the Console tab.
  3. If the issue isn’t already visible in the console output, reproduce it or refresh the page so the relevant lines appear.
  4. Right-click anywhere in the console output → Save as… and save to a .log or .txt file. Or select all and copy to clipboard.

Paste the contents into the Browser console card (blue, on the left) on the analyzer page. Don’t edit the lines — the classifier looks at the original prefixes (base.js:80, net::ERR_…, etc.) to identify patterns.

Why the Inventory Reconciliation page? Most of RapidReconciler’s meaningful traffic — agent heartbeat, database connectivity, status checks, variance fetches — happens on this page. Capturing the console from the login screen or a navigation-only page typically gives the classifier nothing to chew on.

RR agent log

When something looks wrong on the application server — agent not connecting, database unreachable, service won’t start — check the agent’s daily log file.

The default location on the app server:

C:\Program Files\Rapid Reconciler\logs\rr-valc-agent_YYYYMMDD.out.log

Pick the file matching the day the issue happened (the timestamp is in the filename). If you’re looking at today’s, that’s the most recent .out.log. Upload it via the RR agent log card (orange, on the right) on the analyzer page — or paste the contents into the textarea.

Tip. If you’re not sure which log applies, drop whichever you have. The analyzer auto-classifies based on content shape, so a console log accidentally pasted into the agent card (or vice versa) still produces useful output.

Part 2How to read the verdict

After you drop or paste a log, the analyzer scans every line, stitches multi-line entries (stack traces, continuation lines) together, and returns one of three verdicts:

Looks OK

No action needed

Every line is either a known healthy signal or known benign noise. The log doesn’t show anything that needs attention.

Review

Worth a quick look

At least one line is unusual but not confirmed critical — a connection retry, DNS hiccup, or unexpected pattern. Open the concerning-lines section and decide whether to act.

Critical

Confirmed problem

At least one line is a real issue — an application error, full stack trace, repeated failure, certificate failure, or instance start timeout. Needs investigation now.

Two sections appear below the verdict:

  • Concerning lines — expanded by default. The classifier groups them by pattern (e.g., “DNS / name not resolved”, “TLS certificate failure”, “Uncaught JS exception”). Each group has a one-line rule note. When a matching runbook exists, a link to the Help Desk scenario sits beside the group.
  • Benign noise — collapsed by default. Lines the classifier confirmed are safe to ignore (Edge tracking-prevention, Hibernate WARNs, heartbeat noise). Expand it if you want to see what was filtered out and confirm nothing was misclassified.

Part 3What to do next

Critical or Review verdict

  1. Click the runbook link on the matching concerning-lines group. It takes you straight to the Help Desk scenario with step-by-step resolution.
  2. No runbook link? Copy the line text and paste it into the search box on the Help Desk page. The matching scenario will surface as you type.
  3. Still stuck? Save the page (Ctrl+S) or screenshot the verdict + concerning-lines section and attach it to your support ticket. Email rrsupport@getgsi.com with the classified output rather than the raw log — the analysis tells the responder exactly what was found, in seconds.

Looks OK verdict

The log itself doesn’t contain anything actionable. If you’re still experiencing an issue:

  • The cause may be elsewhere — in a different log type (try the other input card), in your network, or in the JD Edwards backend. Use the symptom-based search on the Help Desk page to find scenarios by what you’re seeing rather than what the log shows.
  • Or the log was captured at the wrong time — reproduce the issue with the browser console open or with a fresh agent log session, then re-analyze.
Don’t skip a Critical verdict. The classifier only marks a verdict Critical when at least one line matches a confirmed-problem pattern (ERROR level, stack trace, cert failure, etc.). It’s not raised for ambient noise. If you got Critical, something needs attention even if everything looks fine in the UI.

Part 4What the classifier looks for

Helpful to know, especially if you’re a GSI tech triaging multiple incidents. The classifier matches every line against four categories:

CategoryExamples
CriticalERROR level lines, full stack traces, Uncaught TypeError, ERR_CERT_DATE_INVALID, instance start timeout
Review (flag)Connection refused, ERR_NAME_NOT_RESOLVED (DNS), retry-with-backoff sequences
HealthyHeartbeat, Database connectivity: [OK], status checks, Started AgentApplication in N seconds
Benign noiseEdge tracking-prevention warnings, Hibernate WARN log-shipping noise, routine VALC client queue chatter

Two behaviors worth knowing:

  • Multi-line stitching. Stack traces ( at com.gsi…) and continuation lines are folded onto the preceding entry-start line and treated as one logical entry. You don’t need to clean up multi-line errors before pasting.
  • Auto-analyze. The classifier runs ~220 ms after you stop typing or pasting. No Analyze button to click — just drop the content and watch the verdict appear.
Found something the classifier missed? Send the raw log line and what it should have been flagged as to rrsupport@getgsi.com. The pattern rules are updated regularly based on what real logs surface.
