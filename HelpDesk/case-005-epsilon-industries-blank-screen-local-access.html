Case 005 · 2026-01-12 · Path 2 (matched card) · customer-browser-local-network-access

Blank screen, single user — browser blocking local network access.

A single user reports a blank screen on RR — the page loads but no data displays. Their colleagues are fine. Walking through the standard scope-first funnel (single-user check, cross-machine test, ping, basic browser hygiene) eliminated network and server causes one rung at a time, and the eventual fix was the matched scenario card’s answer: enable the browser’s Local Network Access permission for the RR site. This is the first case in the library where the scenario matcher already had the right card — the case study captures the diagnostic walk that leads to it.

Read the inbound. Two signals.

(1) "Blank screen" on RR — the page loads but no data displays. That's a recognisable symptom. Could be a per-user cause (stale cache, missing browser permission), could be a per-machine cause (network filtering, OS-level restriction), could be a per-environment cause (server outage). Need scoping data to narrow.

(2) "My colleagues seem to be using RR fine" — suggests single-user, but that's their guess. Worth confirming with one explicit teammate before committing to the local-machine path. (If multiple users at the same site are affected, the funnel goes a completely different way.)

Drop the email into the matcher (troubleshooting.html). Several cards score above noise: customer-no-data-hard-reload, customer-clear-browser-data, customer-browser-local-network-access. The last one is the leading candidate — the symptom matches exactly — but I shouldn't jump straight to it. Walk the funnel; confirm the scope; reach the card the right way.

Send the scoping email. Three questions, definitive answers.

The first reply does not push a fix. It collects scoping data. Three carefully-chosen questions: (1) confirm with a colleague — really single-user? (2) cross-machine test — works elsewhere? (3) ping — can their machine reach the server? The combined answers narrow the diagnosis to one of three buckets.

Subject:RE: Rapid Reconciler showing blank screen

Send the matched-card fix. With both end-user and IT-policy paths.

The reply mentions both options because customers vary: some users have admin rights on their PC and can toggle the permission themselves; others have managed environments where Edge/Chrome are locked down by group policy and only their IT can push the change. To:E. Edwards
Subject:RE: Rapid Reconciler showing blank screen
Hi, OK, that puts us at the more specific fix. The blank-screen-on-RR symptom you're describing is most often caused by a browser security feature called Local Network Access (sometimes shown in browsers as "Insecure private network requests"). This is a relatively recent permission in Edge and Chrome that defaults to blocking — it's well-meaning, but it breaks apps like RR that need the page to talk to a local agent on your computer. To enable it for the RR site specifically: Microsoft Edge: 1. Open Edge, navigate to rapidreconciler.getgsi.com 2. Click the lock icon in the address bar (left of the URL) 3. Click "Permissions for this site" 4. Find "Insecure private network requests" (or "Local network access") and set it to Allow 5. Reload the page Google Chrome: 1. Open Chrome, navigate to rapidreconciler.getgsi.com 2. Click the lock icon in the address bar 3. Click "Site settings" 4. Find "Insecure private network requests" and toggle to Allow 5. Reload the page If you don't see that permission in the list, it's probably one of two things: your browser is on an older version that doesn't expose the setting, OR your IT team has the setting locked by group policy. Either way, your IT team can push the change via managed browser settings — let them know "the rapidreconciler.getgsi.com site needs the Local Network Access permission allowed." Try that, then refresh RR and let me know how it looks. Thanks, Sam GSI Helpdesk

Section 3

Resolution & lessons — the matched-card path, and why this case is structurally different.

What resolved it

Enabling browser Local Network Access for the RR site — the matched scenario card’s prescription.

Modern Edge and Chrome have introduced a browser security feature called Local Network Access (also surfaced as "Insecure private network requests" in some versions). It defaults to blocking websites from making requests to local-network resources, which is sensible for general security but breaks apps like RR where the page needs to talk to a local agent installed on the user's computer. The fix was to allow this permission for the rapidreconciler.getgsi.com site specifically. The thread doesn’t make explicit whether Edwards toggled the setting himself or his IT team pushed it via group policy — both paths produce the same outcome and the reply template above accommodates either. The matched scenario card customer-browser-local-network-access already documents this fix in its detail pane; this case validates that the card’s prescription is the right answer for the symptom it claims to cover.

What this case taught us
  • First case in the library where a scenario card matched. Cases 001-004 all surfaced gaps in the scenario library. Case 005 is the first to validate that the library does real work — the matcher had customer-browser-local-network-access as a top candidate from the moment the email was dropped, and it was the right answer. Both kinds of walked ticket are valuable: the no-card cases show where to invest in scenario authoring; the matched-card cases show what good looks like and prove the library’s prescriptions hold up. This case starts a second pile.
  • The "scope first" funnel works as a unit. Walk it even when you "know" the answer. Even when the matcher pointed at the right card from the first email, the discipline of asking the scoping questions first — single-user check, cross-machine test, ping — eliminated three classes of cause in one round and gave certainty fast. Skipping the funnel to jump straight to the matched-card fix would have worked here, but it would also have set a habit of trusting the matcher blindly. The matcher is a guide, not an oracle.
  • "Add to trusted sites + reboot" is worth trying as basic hygiene before the more specific fix. When stale browser state or temp file corruption is the cause, this clears it cheaply — one round of email, customer can do it themselves quickly. When it doesn’t work, you've ruled out an entire class of cause and you can move to the more specific fix with confidence. Skipping this step to jump straight to the matched-card prescription is a false economy: you don’t learn anything about why the simpler thing didn’t fix it, and you’ve robbed yourself of cheap diagnostic information.
  • The Local Network Access permission will keep showing up. Edge and Chrome are progressively rolling this default-block setting out across versions. As enterprises update browsers on managed fleets, this scenario card is going to become one of the most-hit cards in the library — possibly the highest-volume Tier 1 case in 2026. Worth keeping the reply template up to date as browser UIs evolve (the click paths above were correct as of January 2026 but Microsoft and Google revise menu structure regularly).
  • The reply template should mention both end-user-toggle and IT-policy-push paths. Some customers have admin rights on their machine; others don’t. The third email above does this explicitly; it’s worth promoting to the scenario card’s detail pane as a refinement to the existing reply text, so future Tier 1 staff see it in the matcher’s output and don’t have to free-write it.
