Case 002 · 2026-04-30 · Path 2 · No card matched

Daily export fails — “restart service” is a temporary fix.

A customer reports their End-of-Day export works some days and not others. Their workaround — clicking restart service on the RR application server — resolves the failure for ~24 hours, then the issue recurs. The durable fix turned out to be a full app-server reboot. Path 2 — Tier 1 had the answer; the customer's own IT team executed the fix.

Section 2

The walkthrough — the case spans three days.

Day 1
2:18 PM
arrives

Read the email. Notice the diagnostic gold buried in the workaround.

The CC line: just A. Lopez, the customer's IT contact. No GSI engineer on the thread. Subject is RE: Rapid Reconciler export problems — the “RE:” suggests prior history but scrolling up finds none. The customer is reaching out cold.

The body itself is short and unusually well-described, and one phrase jumps out: “Our workaround so far is to click restart service ... which gets the export working again. But the issue comes back the next day.”

That sentence is the diagnostic gold. When clicking “restart service” temporarily fixes a feature and the issue recurs daily, the underlying cause is almost always service-state accumulation — a stuck worker thread, an orphaned file handle, memory pressure crossing a threshold, a connection pool that's exhausted. The Windows Service restart clears the in-process state but not the OS-level state the service depends on. A full server reboot does both.

Section 3

Resolution & lessons — what worked, what made this case different.

What worked, with a caveat about root cause

A full app-server reboot resolved the symptom; the underlying cause was not investigated.

After A. Lopez performed a full reboot of the Rapid Reconciler application server during the next maintenance window, the daily export failure did not recur. The exact root cause was not established — the customer's IT team did not investigate further once the symptom went away, and we didn't push for it. Plausible causes for this kind of presentation include a worker thread getting stuck on a long-running operation, file handle exhaustion in a long-lived service, memory pressure crossing a paging threshold, or a connection pool that fills up faster than it drains. The "restart service" button restarts a single Windows service in-process; a full server reboot also clears OS-level state — orphaned handles, stale TCP connections, paged-out memory — which is typically what makes a temporary fix become permanent.

What this case taught us
  • “Restart service is a temporary workaround” is diagnostic gold. When a customer's daily workaround is to click restart service, that pattern points to service-state accumulation rather than a config bug or a downstream outage. The first instinct should be to recommend a full server reboot — it clears OS-level state that a single-service restart leaves behind. Recognise the signal; don't waste the customer's time on configuration questions.
  • Path 2 with a customer-side action is still Path 2. This case looks at first glance like an escalation — the action happens on the customer's app server, by their IT team. But Path 4 means escalating inside GSI to Network Tech / DBA / Developer for triage. A lateral handoff to the customer's own admin to perform a routine maintenance action is Path 2 — Tier 1 had the answer; the customer's own people executed it. Keep that distinction clean in your routing.
  • Provisional resolution ≠ resolved. A reboot fixed the symptom for one day. That is not a confirmed fix — service-state issues can take 3–7 days to recur depending on traffic patterns. The case stays in “monitoring” until at least a clean week. If it recurs, the next step is escalation to RR Developer for log-level analysis: agent .out logs around the time of failure, plus a memory snapshot if possible.
  • Documentation gap surfaced. The matcher had nothing for this. Provisional new Tier 1 scenario card: “RR feature works intermittently; ‘restart service’ is a temporary daily workaround → recommend a full app-server reboot.” The export-button symptom is one expression of the pattern; the same diagnosis likely applies to other intermittent feature failures (sync stalls, agent timeouts after long uptime, etc.). The card should be phrased around the diagnostic pattern — not the specific feature.
