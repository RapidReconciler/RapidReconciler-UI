Discovery Call Internal Use Only

A guide for the GSI sales contact and tech leads running the Step 6 discovery call — the first joint working session between the Sales contact, Customer I/T, RR DBA, and RR Network Tech. The goal is to introduce the people, walk the customer through what's coming next, surface anything unusual about their environment, and collect a small set of items that aren't captured by the customer's later install-prep submission.

Internal Document

For GSI staff only. Do not distribute to customers. The customer-facing companion is the Installation Prep Document, sent at Step 7 after this call concludes.

Purpose & Scope

The discovery call is a relationship and orientation meeting, not a technical deep-dive. By Step 6 the customer has already received the Provisioning Document and has reviewed the technical requirements with their I/T, security, and JDE teams. The detailed environment data (server names, IPs, JDE connection details, decimal handling, etc.) is collected later through the Installation Prep Document, which the customer fills out after this call.

Use the discovery call to accomplish three things:

  1. Introduce the GSI tech team to Customer I/T. The RR DBA and RR Network Tech are the people who will be on the install meeting; this is their first chance to meet the customer's I/T contact and put a face to the name on the email thread.
  2. Walk the customer through what happens next. Explain the install-prep submission, how it kicks off the install, and what the install meeting itself looks like.
  3. Surface environment wrinkles early. Some customers have unusual requirements (change-control windows, security review boards, multi-domain environments, non-standard JDE deployments). Identifying these in conversation is faster than discovering them mid-install.
Key principle

The install-prep submission is comprehensive. Don't try to collect detailed technical values during this call. The customer will fill those out at their own pace using the install prep document; reading values aloud over a video call is error-prone and slower. Use the call for the seven items below and for context the document can't give you.

Participants

Four roles attend by default. The customer's JDE owner is invited but optional — their content arrives on the install-prep submission and they're rarely needed live unless the customer's JDE deployment is unusual.

GSI Side

Sales Contact Host

  • Owns the customer relationship and runs the call
  • Walks through agenda, owns introductions, owns timing
  • Carries the contract context (company count, license tier, special terms)
  • Sends the install prep doc immediately after the call
GSI Side

RR DBA

  • Owns the database side of the install (creates RapidReconciler_Prod, deploys SSIS package)
  • Answers questions about SQL Server requirements and the database server topology
  • First contact point for the customer DBA if one is named
GSI Side

RR Network Tech

  • Owns the application server / agent install side of the install
  • Answers firewall, outbound-access, and DNS questions
  • Runs the agent install on the customer's application server when the time comes
Customer Side

Customer I/T Lead

  • Primary technical contact — will fill out the install prep doc
  • Owns server provisioning, OS, SQL Server install, network and firewall config
  • Will also be on the install meeting (provides local administrator credentials)
Customer Side · Optional

Customer JDE Owner

  • Provides the JDE source server, credentials, and table qualifier on the install-prep submission
  • Confirms data dictionary decimal settings (ECST, UNCS, PQOH, TRQT)
  • Invite by default; presence on the call is welcome but rarely required
Customer Side · Optional

Customer Security / Compliance

  • Invite if the customer has flagged a security review or change-control process during sales
  • Useful for confirming firewall, antivirus, and service-account expectations
  • Flagging here often saves a multi-week delay later

Agenda

Plan for 45 minutes of agenda time and 15 minutes of buffer. Times below are guidelines; in practice the introductions and Q&A vary the most. The Sales contact owns the clock.

  • 0:00–0:05

    Introductions Sales

    Each GSI attendee introduces themselves and their role for the install. Sales then invites the customer side to do the same. Confirm the customer’s I/T lead by name — this is the person who will fill out the install prep document.

  • 0:05–0:10

    Where we are in the process Sales

    Recap the path so far: provisioning doc reviewed, contract signed, this discovery call, then install prep doc, then the install meeting itself. Set the expectation that the install meeting comes after the customer submits their install prep.

  • 0:10–0:25

    Environment review Both

    Have the customer describe their environment in their own words: which topology they’re leaning toward, which JDE source platform they have, whether the servers exist or need to be provisioned, and where they are with their security team. RR DBA and RR Network Tech listen for unusual choices and ask follow-up questions.

    This is also where you flag known pitfalls: SQL Server mixed-mode authentication is required; the customer DBA may need to involve their security team if Windows-only auth is the current standard.

  • 0:25–0:35

    What we still need from you Sales

    Walk through the seven items in What to Confirm. Most can be answered live; if any require the customer to circle back internally, get a name and an ETA.

  • 0:35–0:45

    Walk through what comes next Sales + Network Tech

    Preview the install prep doc — topology selection, source platform, server names & IPs, Windows security, the pre-install checklist, and the JDE submission & submit step. Reinforce that the doc is the customer’s working tool; submitting the email at the end is what triggers GSI to schedule the install meeting.

    Briefly preview the install meeting itself: 2–3 hours, screen-share, Customer I/T provides local administrator credentials on each server, GSI runs the database creation, SSIS package deployment, and agent install.

  • 0:45–1:00

    Q&A and next steps Both

    Open floor. Common topics are covered in the FAQ below — have it open in a tab. Close by confirming: (1) Sales will send the install prep doc within the day, (2) any open items the customer is circling back on, (3) target submission date.

What to Confirm on the Call

Seven items aren’t in the install-prep submission and are best confirmed live. Most come from the contract or a quick exchange with Customer I/T. Capture each in the customer record — the install team will reference these during VALC setup.

ItemSourceWhy It Matters
Number of companies purchased Sales contract Drives the Maximum Companies Allowed value on the customer’s VALC client record. Confirm aloud to make sure Sales and Customer I/T have the same number.
Exact JDE company numbers (5-digit) Sales contract The specific company numbers the customer is licensing (e.g., 00001, 00002). Verified during VALC licensing; confirming up front prevents licensing the wrong companies.
Maintenance window timezone & preferred install time Customer I/T contact The agent install briefly disrupts data flow. Schedule the install meeting for a low-impact window in the customer’s timezone.
Planned go-live date Sales / Customer I/T Sets the timing target. Customer I/T uses this to plan their internal communications and user training.
Additional databases beyond Production Customer I/T contact Whether Test, Dev, or Training databases are required at install time. Each needs its own backup-restore and a separate registration in VALC.
Complex password policy required at go-live? Customer I/T contact If yes, the toggle is enabled at install time rather than later. Cleaner to know up front.
Customer’s RapidReconciler administrator (email) Customer I/T contact The customer-side person who will manage their users and own day-to-day administration. Their email is needed for the initial GSI Admin user record in VALC.
Tip

Document these answers in the customer’s record (CRM, shared customer file, or wherever your team tracks install prep) so they’re available to the install team when they pick up the work.

Out of Scope for This Call

Resist the urge to collect detailed technical values during the call. The install prep document is the right place for these — the customer can fill it out at their own pace, with their reference materials in front of them, and the result is more accurate than any reading-aloud session.

  • Server names & internal IPsCaptured by the install prep doc’s topology page. Customer fills in once and the value flows into the submission email.
  • JDE connection detailsServer name/IP, database username, password, server type, table qualifier — all in the install prep doc’s JDE section.
  • Data dictionary decimalsECST, UNCS, PQOH, TRQT — the customer’s JDE owner enters them on the install prep doc.
  • Hardware specs & OS versionsAlready covered in the provisioning doc the customer has reviewed. Don’t re-walk it; address only specific questions.
  • Detailed firewall rulesRR Network Tech can answer specific questions, but the full rule set is in the provisioning + install prep docs. Don’t draft rules live.
  • Pricing renegotiationBy Step 6 the contract is signed. Pricing questions should be deflected back to a separate Sales conversation.

After the Call

The Sales contact owns wrap-up. Aim to have everything below complete within one business day of the call.

Sales Wrap-Up
What happens next

The customer works through the install prep document at their own pace and submits the email when complete. That submission lands in rrsupport@getgsi.com with the subject line RapidReconciler Pre-Install Submission — {Customer Name} and triggers the install — the RR DBA picks it up first, completes the database side, and signals the RR Network Tech to begin VALC setup and the agent install.

FAQ — Common Customer Questions

Common topics that come up on the call. Have answers ready; don’t improvise on security or licensing questions.

How long does the install meeting itself take?

Plan for 2–3 hours of working session. The RR DBA runs the database creation and SSIS deployment first; the RR Network Tech runs the agent install on the application server once the database is ready. The customer’s I/T contact needs to be on-session the whole time to provide local administrator credentials and screen-share access.

What does the agent send back to GSI?

The agent maintains an outbound-only, 256-bit encrypted connection from the customer’s application server to GSI’s VALC infrastructure. There is no inbound traffic from GSI to the customer’s network.

It transmits: the public-facing IP of the application server, the internal IP, the IP/port of the database server, the credentials the agent uses to access the RR database (so it can be re-pointed without re-prompting the customer), the database names and ports, hashed user passwords, user access security settings, and the licensed JDE company numbers and names. Most of this is for keeping VALC’s view current and supporting GSI in answering customer questions without screen-share.

Can our DBA be involved if it’s not the same person as our I/T lead?

Yes — very common. Invite both. The I/T lead typically owns the application server side and the install prep submission; the DBA owns the database server side and works directly with the RR DBA during the install meeting.

We have a strict change-control window. Can the install fit?

Yes — flag this on the call so RR DBA and RR Network Tech can plan accordingly. The install meeting is schedulable; what matters is that the customer’s I/T contact is available for the full 2–3 hour window with local administrator credentials in hand. Multi-night installs are not supported.

Our security team requires a review of any new outbound rules. How do we handle that?

Direct them to the Network Requirements section of the Provisioning Document — it lists the exact hosts, IPs, and ports the application server needs to reach. The connection is outbound only; no inbound exception is required. RR Network Tech can join a follow-up call with their security team if it accelerates approval.

We’re considering Configuration 3 (separate SSIS server). Anything special?

Configuration 3 is uncommon but supported. The main implication is that the OLE DB driver for the JDE source must be installed on the dedicated SSIS server (not the database server) and the database server’s firewall needs to scope inbound 1433 to both the application server and the SSIS server. The install prep doc filters its checklist to Configuration 3 once the customer selects it.

Do we need the JDE owner on this call?

Usually no — the JDE-side content (server name, credentials, table qualifier, decimal handling) arrives on the install prep submission and rarely needs live discussion. Invite them anyway in case Customer I/T wants the introduction. The exception is if the customer has an unusual JDE deployment (multiple environments, non-standard table qualifiers, recent JDE upgrade in flight) — in that case, their presence is valuable.

What if the customer hasn’t provisioned servers yet?

That’s fine — the install prep document includes a checklist they’ll work through, and provisioning new servers is a normal part of that process. The trigger for the install meeting is the customer’s install-prep submission email, not a fixed calendar date. Sales should set expectations that the install meeting is scheduled after the customer signals they’re ready.

Does GSI provide SQL Server licensing?

No. SQL Server is customer-provided. Defer to the licensing breakdown in the Provisioning Document for a complete table of what GSI provides versus what the customer provides. If they don’t have an existing SQL Server license, this is a budget conversation Sales should have ahead of go-live.

What about our test or dev databases?

One agent supports any number of RapidReconciler databases (Prod, Test, Dev, Train), including databases hosted on different SQL Servers. Each database is registered separately in VALC during install. Confirm during the call which non-prod environments the customer wants stood up at install time — that drives whether the install meeting includes additional database registrations.
