Customer logs in but sees no data — ladder of fixes from cheapest to most invasive.

Customer authenticates to RapidReconciler successfully but no data is displayed in the UI. The cause is rarely on the GSI side — far more often it's something on the customer's network, browser, or VPN. Walk down the list of possible solutions in order; the first three or four will resolve most cases without escalation.

Section 1

Scenario description — what the customer is reporting.

Symptom

Customer reports they can log in to RapidReconciler successfully but no data is displayed in the UI. Common phrasings include "I see no data," "the page is blank," "everything is empty," or "I logged in but nothing's there."

Symptom variants that all fold into this scenario:

  • The Reconciliation, Inventory, or other data-bearing pages are empty
  • Admin > General displays the message "No database connection. Delete Rapid Reconciler CARDEX requires a database connection."
  • The customer reports it started suddenly with no apparent change on their end
  • Multiple users at the same customer are reporting the same problem at the same time
  • A single user reports the issue while colleagues are unaffected
Section 2

Possible solutions — ordered roughly easiest to hardest.

1 Verify the customer is on VPN or the corporate network Path 2 · Respond directly
When this applies: The customer is working remotely (home, travel, off-site). RapidReconciler's backend is only reachable from the customer's network or via VPN; if they're disconnected, login succeeds but no data flows. This is the single most common cause for a remote worker.
Steps to resolve
  1. Ask the customer if they are working remotely (home / travel / off-site).
  2. If remote, ask them to confirm they are connected via VPN or their organization's approved remote-access method.
  3. If they are not connected, ask them to connect to VPN and try again.
  4. If they believe they are connected, ask them to disconnect and reconnect the VPN, then hard-refresh the browser.
  5. If the issue persists after confirmed VPN connection, proceed to Solution 2 (hard reload).
2 Hard reload the browser Path 2 · Respond directly
When this applies: The customer is confirmed on the corporate network or VPN. Stale browser cache or a partially-failed initial page load is the most common simple cause; the agent script may need to re-authenticate after a transient connectivity blip.
Steps to resolve
  1. Ask the customer to perform a hard reload of the browser using Shift + F5 (Windows) or Cmd + Shift + R (Mac).
  2. If that doesn't resolve it, ask them to fully close all browser windows and reopen, then log in again.
  3. If still no data appears, proceed to Solution 3 (browser local-network access permission).
3 Enable browser local-network access permission Path 2 · Respond directly
When this applies: The customer's browser was recently updated, or this is a new workstation. Edge and Chrome introduced a permission gate that controls whether sites can connect to local-network devices — if RapidReconciler isn't on the allowed list for that customer's machine, all data calls silently fail.
Steps to resolve
  1. Walk the customer through Edge: Settings → Privacy, search, and services → Site permissions → All permissions → Local network access.
  2. In the "Customized behaviors" section, check the "Allowed to connect to any device on your local network" list.
  3. Verify rapidreconciler.getgsi.com is present in that allowed list.
  4. If missing, click "Add site" next to that list and add rapidreconciler.getgsi.com.
  5. Chrome has an equivalent setting under its site-permissions area — the wording differs slightly but the concept is the same.
  6. After adding the site, hard-reload the page and verify data appears.
  7. If the customer cannot find or change the setting on their machine, escalate to their I/T department.
Real example
Epsilon Industries — single user, blank screen

A single user at Epsilon Industries reported a blank screen on RapidReconciler — the page loaded but no data showed, while colleagues on the same network were unaffected. Walking the standard scope-first funnel (single-user check, cross-machine test, ping, basic browser hygiene) eliminated network and server causes; the matched fix — enabling Local Network Access permission for the RR site in their browser — resolved it.

This was the first case in the library where a scenario matched the symptom on the first match attempt — validates the matcher pattern for this class of issue.

4 Clear browser data — last-resort browser cleanup Path 2 · Respond directly
When this applies: VPN confirmed, hard reload didn't help, local network access is correctly configured, and Customer I/T has confirmed no firewall changes. Stale or corrupted authentication tokens, cookies, or service-worker state can hold the page in a broken state even after the underlying issue is gone.
Steps to resolve
  1. Refer the customer to their I/T department for browser-specific instructions on clearing browser data.
  2. The general approach is to clear cookies and cached files for rapidreconciler.getgsi.com (not all sites), then close the browser fully and reopen.
  3. After clearing, have the customer log in again with a fresh session.
  4. If the issue still persists after all customer-side checks, the issue is on the GSI side — proceed to Solution 5 (DevTools console) to gather diagnostic information before escalating.
5 Check the browser DevTools console for errors Path 3 · Research / docs
When this applies: All customer-side basic checks (VPN, hard reload, local network access, browser data) come back clean and the Database tab in VALC shows healthy. Opening the customer's browser DevTools Console is the next step to see whether the page is silently failing — JS errors, blocked requests, CORS rejections, or the agent script not loading at all.
Steps to resolve
  1. Have the customer open DevTools: in Edge or Chrome press F12 (or Ctrl+Shift+I), then click the Console tab.
  2. Ask the customer to clear the console (the ∅ icon at top left), then have them perform the action that fails — usually login, then navigating to the page where data should appear.
  3. Compare against the healthy-baseline pattern. A working session shows lines like URL visited /page/login and URL visited /app/inventory/reconciliation from base.js — that confirms the agent is loaded and the customer is navigating the app.
  4. Ignore the known noise: warnings about 'popover' attribute with an invalid value are AngularJS template noise from the UI and are harmless. 404s on fontawesome-webfont.woff or Simple-Line-Icons.woff are cosmetic font fallbacks and do not affect data flow.
  5. Look for the actual signal: any red error from base.js with a stack trace, any 4xx/5xx on the application's own API calls (not the .woff files), CORS or Mixed Content warnings, net::ERR_BLOCKED_BY_CLIENT entries from a browser extension, or the complete absence of URL visited lines from base.js.
  6. Have the customer screenshot or copy the relevant errors and email them to rrsupport@getgsi.com. Right-click in the console → Save as… exports the full log if a longer trace is needed.
  7. If the console shows no base.js activity at all, the agent script never loaded — jump to the Agent services not running and Browser shows "Not Secure" scenarios.
  8. If the console shows red errors on application API calls (4xx/5xx), the agent loaded but the back end is unhappy — check SQL Agent job not running and the database scenarios.
  9. If the console shows net::ERR_BLOCKED_BY_CLIENT or CORS blocks, the customer's browser environment is interfering — ad-blocker, corporate DLP, or proxy. Have them retry in an incognito window with extensions disabled to confirm.
  10. If the console shows red net::ERR_NAME_NOT_RESOLVED errors on a per-customer hostname (e.g. rrprod-<customercode>.getgsi.com), proceed to Solution 7 (per-customer hostname DNS).
6 Investigate recent customer-side network or security changes Path 3 · Research / docs
When this applies: Sudden onset (started today, no apparent customer-side change), multiple users at the same customer affected simultaneously, OR the admin reports a "No database connection" message specifically on Admin > General. The customer's IT team has likely rolled out a firewall change, DNS filter, web content filter, secure web gateway, or endpoint security update that blocks rapidreconciler.getgsi.com.
Steps to resolve
  1. If the customer mentions Admin > General specifically — acknowledge their theory if they've offered one (e.g. "we added a new company yesterday"), then explain that the "No Database Connection" message means traffic to the database is being blocked between their workstation/server and GSI, which a RapidReconciler configuration change cannot cause. This refocuses the conversation on the network path.
  2. Ask the customer's I/T contact directly: "Has your I/T team recently turned on, updated, or changed the policy on any web filtering, DNS filtering, secure web gateway, or endpoint security product?" Common products to ask about by name so the customer can recognize them:
  3. Web / DNS filters & secure web gateways: Cisco Umbrella, Zscaler Internet Access (ZIA), Cloudflare Gateway, Forcepoint Web Security, Barracuda Web Security Gateway, iboss, DNSFilter, TitanHQ WebTitan, Symantec / Broadcom Cloud SWG, Netskope.
  4. Next-gen firewalls with URL filtering: Palo Alto Networks (incl. Prisma Access), Fortinet FortiGate / FortiGuard, SonicWall, Check Point, Sophos Firewall / Web Appliance, Cato Networks.
  5. Endpoint security / EDR with web blocking: CrowdStrike Falcon, SentinelOne, Microsoft Defender for Endpoint / Defender SmartScreen, Sophos Endpoint, Trend Micro Apex One.
  6. Also ask whether anything changed even if the customer doesn't think it's related: Windows Update / patches over the weekend, a new VPN client, a new corporate root certificate push, or a switch from split-tunnel to full-tunnel VPN.
  7. Run the cross-network triage. Ask the customer to test from a different machine on the same network and from a machine on a different network (mobile hotspot is fine) — if the issue follows the network and not the user, that further confirms a customer-network filter.
  8. Have the customer's I/T team explicitly whitelist rapidreconciler.getgsi.com (and any related GSI hostnames/IPs the network team has on file) on the product they identified, then have the customer hard-refresh the browser and re-check.
  9. If Customer I/T confirms no recent changes and no filter is in scope, escalate to Solution 7 (per-customer hostname DNS) or Solution 8 (Cloudflare propagation).
  10. If the issue persists after Customer I/T whitelisting and a hard refresh, email rrsupport@getgsi.com with the customer details, the products checked, and the steps already attempted.
Real example
Delta Co — DNS filter rule blocking *.getgsi.com

A customer admin at Delta Co reported an urgent morning outage — all users seeing no data, Admin > General showing "No database connection." The system had worked the previous afternoon. The customer had pre-emptively ruled out Citrix, JDE CNC, and certs on their side before reaching out.

The diagnostic gold was the cross-network DNS test: nslookup and ping failed from inside the customer's network but succeeded from outside, immediately pointing to a customer-side filter. Root cause was a DNS-filtering rule the customer's IT had added two days earlier that inadvertently blocked *.getgsi.com. Customer's IT removed the rule; system back online same day.

7 Per-customer hostname fails to resolve (escalate to Network Tech) Path 4 · Escalate
When this applies: DevTools Console shows red net::ERR_NAME_NOT_RESOLVED errors on requests to a per-customer RapidReconciler hostname (e.g. rrprod-<customercode>.getgsi.com on a non-standard port). The login page may load fine but data fetches fail because the front-end can't reach the customer-specific backend. The error is a DNS failure — the browser tried to look up the hostname and got nothing back.
Steps to resolve
  1. Confirm the failing URL pattern in the customer's DevTools Console. Note the exact per-customer hostname and port (e.g. rrprod-<customercode>.getgsi.com:NNNNN). Capture a screenshot of the console so the network signature is preserved if the issue is intermittent.
  2. Run the cross-network triage. Have the customer test from another machine on the SAME network, then from a DIFFERENT network (mobile hotspot or personal phone tether is fine). This is the single highest-value test — about two minutes and it tells you which side owns the problem.
  3. Interpret the test:
    • Same-network machine fails the same way + different network works → customer-side. Their corporate network is blocking or failing to resolve the hostname.
    • Same-network machine fails the same way + different network ALSO fails → GSI-side. The DNS record is broken or missing on GSI's authoritative DNS.
    • Same-network machine works + this user's machine fails → user-machine local. Stale DNS cache, hosts file, or browser profile.
  4. Check whether multiple users at the same customer are reporting identical errors right now. If yes, treat as GSI-side until proven otherwise (a customer-network issue typically wouldn't cause every user at the customer to fail simultaneously).
  5. If the verdict is customer-side: have the customer's I/T explicitly whitelist the per-customer hostname (and the port) on their DNS filter, web filter, and firewall. Ask whether they recently rolled out a DNS-filtering product (Cisco Umbrella, Zscaler, Cloudflare Gateway, DNSFilter, etc.) or changed VPN config (split- vs. full-tunnel). After whitelisting, have the customer hard-refresh and recheck the console.
  6. If the verdict is GSI-side: escalate to GSI I/T (Network Tech) to verify the A record for the per-customer hostname on GSI's authoritative DNS, and to confirm the per-customer backend host is up. Do not have the customer keep retrying — the issue is not on their end.
  7. If the verdict is user-machine local: have the user run ipconfig /flushdns (Windows) or sudo dscacheutil -flushcache (Mac), check their hosts file (C:\Windows\System32\drivers\etc\hosts on Windows; /etc/hosts on Mac/Linux) for a stale entry containing the GSI hostname, and retry. If still failing, have them try in a private/incognito window and a different browser to rule out browser-profile DNS-over-HTTPS misconfiguration.
  8. Once resolved, make a note in the ticket of which path resolved it (customer I/T whitelist, GSI DNS fix, local DNS flush). This helps the next person when the same pattern surfaces from a different customer.
8 Cloudflare update made but data not appearing (escalate to Network Tech) Path 4 · Escalate
When this applies: Cloudflare has been updated by GSI I/T (Daren) for this customer and confirmed propagated, but data still doesn't appear in the customer's RapidReconciler UI. Most common during or just after a server migration where SQL Agent / SSIS / DNS were intentionally staged.
Steps to resolve
  1. On the database server, confirm the SQL Agent job schedule is enabled.
  2. Trigger a one-time manual run of the SQL Agent job to validate it works.
  3. Check the job history for errors on the most recent run.
  4. Have the customer log out and log back in, or hard-refresh their browser.
  5. If the issue is post-migration, double-check SSIS package connection strings on the new server — they may still point at the old database server. See Migration complete but data not flowing for the full migration-cutover checklist.
