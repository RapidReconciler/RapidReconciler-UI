Post-deployment JDE import fails — rruser login error — connection passwords got most layers; SSIS package internal one missed.

Customer-side admin deployed RR to production successfully but "Import JDE data" errors out after a few minutes with an rruser authentication failure. The first email-based fix — "update your connection passwords" — is correct in principle, but the customer can update the deployment-time connections and miss the package-internal one. Resolution typically requires a brief Teams session.

Section 1

Scenario description — what the customer is reporting.

Symptom

Customer-side admin completed a successful RapidReconciler production deployment but the "Import JDE data" task errors out a few minutes in with an rruser authentication failure. The error often references the SSIS package and a login failure for rruser.

Symptom variants that fold into this scenario:

  • Customer says they "updated the rruser password" but it didn't fix it
  • Error appears post-deployment in the data import phase, not at deployment time
  • Customer-admin doing the deployment is technical and may already be in the deployment-time connection screen
Section 2

Possible solutions — ordered roughly easiest to hardest.

1 Recommend updating connection passwords (try first — cheap) Path 2 · Respond directly
When this applies: First-pass response. The customer probably needs to update their connection passwords in the deployment-time connection configuration. This is correct in principle and resolves a fraction of cases — about 24-48 hours later you'll know if it stuck.
Steps to resolve
  1. Reply to the customer recommending they update their rruser connection passwords on the production deployment.
  2. Frame it as "this is the most common cause — updating the connection passwords usually resolves it." Set expectations: if it doesn't fix it within 24 hours, schedule a Teams session for a deeper look.
  3. Diary the ticket for 24-hour follow-up.
  4. If the customer reports back that the import is still failing, proceed to Solution 2 (Teams session for the SSIS package internal connection).
2 Teams session: update the rruser Connection Manager INSIDE the SSIS package Path 3 · Research / docs
When this applies: Solution 1 didn't resolve it. The customer updated deployment-time passwords but missed the Connection Manager INSIDE the SSIS package itself — that one is hardcoded into the package on deploy and needs Visual Studio to update.
Steps to resolve
  1. Schedule a 15-minute Teams session with the customer admin who has access to the SSIS package source.
  2. On the call, have them open the SSIS package in Visual Studio.
  3. In the Solution Explorer or Connection Managers panel, locate the Connection Manager that uses the rruser credential.
  4. Right-click → Edit, re-enter the rruser password, and save.
  5. Redeploy the package to the SSIS catalog.
  6. Trigger the JDE import job manually to confirm it succeeds.
  7. After the call, document which package and which connection manager needed the update — future deployments at the same customer may need the same fix.
Real example
Gamma Holdings — the email fix wasn't enough

Gamma Holdings' admin deployed RR to production successfully but the JDE import errored out with an rruser auth failure. The first email response — "update your connection passwords" — was correct in principle but pointed at the wrong layer; Gamma fixed the deployment-time connections and missed the package-internal one.

Resolved in a 15-minute live Teams session: opened the SSIS package in Visual Studio, found the rruser Connection Manager, re-entered the password, redeployed. Future SSIS deployments at this customer may need the same internal-connection fix.
