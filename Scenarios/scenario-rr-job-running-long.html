Job won’t finish For your IT

RR nightly job has been running for hours / hasn’t completed

The nightly RapidReconciler job has been running far longer than it usually does — sometimes 4+ hours, sometimes still going at the start of business — and it hasn’t completed. There’s no error message and the System Status indicator may still show running rather than red. Today’s figures aren’t available because the refresh never finished.

This almost always means the SQL Server side got stuck on something — a long-running transaction, a blocked session, a runaway log growth — not a problem with RapidReconciler itself. The fix is for your IT to stop the running SQL job, let it roll back, and re-run it. Most of the time that clears it.

Different symptom: if the System Status indicator is red and the job has clearly failed (not just hung), see System status light is red instead — that path covers the post-failure environment checks (disk, TempDB, SSIS errors).

What you’re seeing

  • The RR nightly job started at its usual time but is still running hours later.
  • No error has been raised; the System Status indicator may show running, amber, or just “data is older than usual.”
  • The Reconciliation page shows yesterday’s data (or older); today’s figures aren’t available yet.
  • Common phrasings: “RR is running long,” “the nightly didn’t finish,” “data isn’t updating,” “the job’s been going since last night.”

Recommendations to share with your IT

Step 1
Capture the basic context

Note the following before forwarding to your IT.

  • The job’s normal completion time vs. how long it’s been running today.
  • The date the last successful refresh completed (so you know how stale today’s figures are).
  • Whether the System Status indicator is amber, red, or shows “running.”
Captured? Forward to your IT with Steps 2–3 to walk.
Step 2 For your IT
Stop the running SQL job and let it roll back

This is the standard fix and it resolves most cases. Have IT do the following on the RapidReconciler database server:

  1. Open SQL Server Management Studio → SQL Server Agent → Job Activity Monitor.
  2. Find the running RapidReconciler job (the one that hasn’t completed).
  3. Right-click → Stop Job.
  4. Wait for the job to finish rolling back. Don’t kill the SQL Server process or restart the service while a transaction is rolling back — let it complete cleanly. Rollback can take a while (sometimes as long as the original run); that’s normal.
  5. Once Job Activity Monitor shows the job is no longer running, re-start the job manually (right-click → Start Job at Step → first step).

Most of the time, the re-run completes in its usual time and today’s figures appear once it finishes.

Re-run completed cleanly? Stop here — you’re done. Job hangs again, or rollback never completes? Continue to Step 3.
Step 3 For your IT
If it hangs again — investigate what’s blocking it

If the job hangs at the same point on the re-run, something on the SQL Server side is consistently blocking it. Have IT check, in this order:

  • Activity Monitor → Active Expensive Queries and Processes while the job is running. Look for the RR session and check what it’s blocked on (look at the Blocked By column).
  • Long-running transactions or open transactions on TempDB. Run DBCC OPENTRAN on the RapidReconciler database and on TempDB to see if anything’s holding locks.
  • TempDB log file size and autogrowth. A TempDB log that’s near full will stall any large operation. Same goes for the database’s own log file (RapidReconciler_Prod_log.ldf). Free space if needed.
  • Free disk space on every drive hosting the database, the data and log directories, and TempDB. Low free space stalls the job silently.
  • Recent SQL Server changes — SQL patches, service-account changes, antivirus or backup software added, log shipping or replication enabled. Anything that changed since the last successful run is suspect.

If IT identifies and fixes the underlying issue (kill the blocking session, expand the log, free disk space), have them re-run the job from Step 2.

Issue identified and resolved by IT? Stop here. Still hangs after the checks above? Continue below to RR support.

If your IT can’t resolve it — email RapidReconciler support

Send what your IT found

If the job consistently hangs and the SQL Server-side checks don’t point at a fixable cause, email rrsupport@getgsi.com and include:

  • The job’s normal vs. actual run duration, and the date of the last successful completion.
  • What your IT saw in Activity Monitor — which step the job was on, what session it was blocked by (if anything), how long it ran before being stopped.
  • The SQL Agent job history error or warning messages, if any.
  • What your IT already checked and ruled out (disk, TempDB, log space, recent SQL Server changes).
  • Whether the issue is a one-off or recurring — e.g., “hangs every Monday but not other days” is a useful pattern.
