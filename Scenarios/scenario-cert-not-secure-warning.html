Browser shows “Not Secure” warning

Your browser is showing a security warning or “Not Secure” banner when you reach the RapidReconciler URL. This is almost always one of two things: a stale connection your browser is holding from before the certificate was renewed, or a genuine certificate problem on the GSI side. The first three steps below take a minute and resolve the typical case — you only need IT for Step 4.

What you’re seeing

  • The browser displays “Not Secure” in the address bar, or a full warning page (“Your connection isn’t private”) before the login page loads.
  • The error code below the message is usually NET::ERR_CERT_DATE_INVALID, NET::ERR_CERT_AUTHORITY_INVALID, or similar.
  • Common phrasings: “the site says it’s not secure,” “there’s a certificate warning,” or “the lock icon is missing.”

Try these steps in order

Step 1
Close Edge completely and reopen

Most cert warnings right after a renewal are caused by Edge holding an open connection from before the new cert was deployed. A clean restart of Edge resolves the typical case.

  • Close every Edge window (not just the affected tab).
  • Reopen Edge from the taskbar or Start menu.
  • Visit the RapidReconciler URL again.
Warning gone? Stop here. Otherwise, continue to Step 2.
Step 2
Hard-reload the page

Force the browser to fetch the page and the certificate fresh.

  • With the warning page open, press Shift F5 on Windows (or Cmd Shift R on Mac).
Warning gone? Stop here. Otherwise, continue to Step 3.
Step 3
Try the URL in InPrivate (or a different browser)

This is the diagnostic that tells you which side the problem is on. InPrivate uses a fresh session with no cached connections, no cookies, no extensions — and a different browser uses an entirely separate certificate trust store.

  • Press Ctrl Shift N to open an InPrivate window in Edge, or open Chrome / Firefox if you have it.
  • Visit the same URL.

While you’re here, click the red Not secure badge in the address bar to see the certificate details — note the Valid to date. You’ll include it in the support email if you need to escalate.

Read the result:

  • Warning gone in InPrivate / a different browser — it’s a stale local cache. Continue to Step 4 (or have IT do it).
  • Still warning in InPrivate / a different browser — the certificate is genuinely broken on the GSI side. Skip Step 4 and email RR support directly using the block below.
Worked in InPrivate? Continue to Step 4. Still warning everywhere? Skip to the escalation block below.
✉
Step 4 below is for your IT
If Step 3 confirmed the warning is local (works in InPrivate, fails in normal Edge), the next step is workstation-level cache clearing — that needs IT. Click to draft an email with the steps below pre-filled.
Step 4
Hand off to IT For your IT
When this applies: Steps 1–3 didn’t resolve it, but the URL works in InPrivate or a different browser. That means the cert is renewed on the server and the workstation is holding stale state.

Two paths for IT to try, in order:

  • Clear the workstation’s cached cert / SSL state — standard browser network-state flush, Windows SSL state reset via Internet Options, and if needed a sweep of the Windows cert store and a full Edge restart.
  • Verify the renewed cert is actually live — if the workstation cleanup doesn’t resolve it, the new cert may not be deployed yet, or a CDN / load balancer / reverse proxy in front of the site may still be serving the old one.
Padlock shows the new cert? Stop here. Otherwise, escalate to RR support with what was tried.
