Stale or wrong data

Data looks old and isn’t refreshing

RapidReconciler is showing yesterday’s or older figures. The system-status light in the page header tells you what’s going on:

  • Flashing amber — the refresh is running right now. Just wait.
  • Red — the refresh failed. See System status light is red instead.
  • Green — the refresh didn’t run when it should have. The steps below cover that case.

What you’re seeing

  • RapidReconciler is showing yesterday’s or older figures rather than current data.
  • You expect a scheduled refresh that hasn’t happened.
  • Other parts of RapidReconciler load and look healthy — only the data is old.

Try these steps in order

Step 1
Is the refresh actually overdue?
When this applies: the system-status light is green (not amber, not red).
  • Hover the status indicators in the page header — the tooltip shows the last refresh time. There is no separate timestamp printed on the page.
  • Compare that timestamp against your usual refresh schedule. If it’s from after the most recent expected refresh, today’s data is in.
  • Hard-reload the page (Shift F5 on Windows, Cmd Shift R on Mac) to make sure you’re not looking at a cached older version.
Refresh overdue and figures genuinely stale? Continue to Step 2. Otherwise, stop here — this is normal pre-refresh behavior.
✉
Hand off to your IT
The remaining checks need to be performed by your IT or DBA. Click to draft an email.
Step 2
Possible problem with RapidReconciler job? For your IT
When this applies: the system-status light is green and Step 1 confirmed the refresh is overdue. That means the job didn’t run when it should have — not that it failed.

The refresh runs through a SQL Agent job on your database server. The job name is typically RapidReconciler_Prod (your environment may use a custom name). Standard checks for your IT or DBA, in order:

  • Is the database server up and reachable? Most "didn’t run" cases are a server that rebooted or went offline.
  • Is the SQL Server Agent service running? Open services.msc → find SQL Server Agent → Status should be Running. Set the Startup type to Automatic so the agent comes back up after a server reboot.
  • Is the job itself enabled? In SSMS, expand SQL Server Agent → Jobs, find RapidReconciler_Prod (or your custom name), right-click → confirm it’s not disabled. Check the schedule under Properties → Schedules — was it edited, paused, or deleted?
  • What does the history show? Right-click the job → View History. When did it last run? If it never ran on the expected schedule, the schedule itself is the suspect. If a recent run is missing where you’d expect one, see the previous bullet.
  • Run it manually to confirm it works. Right-click the job → Start Job at Step. Watch it complete cleanly. Then re-confirm the schedule will fire next cycle.
  • Let the user know. Tell the user the refresh is running. Fresh figures will appear on the page once the job completes.
Once IT runs through these — job runs and data refreshes? Stop here. Job is broken in a way IT can’t resolve? See “Need help?” below.

Need help?

Most cases are customer-side

The SQL Server, the SQL Agent service, and the job schedule all live on a server your IT or DBA owns. RR support can’t reach into your environment to start a stopped service or re-enable a disabled job.

If you have a GSI services contract (GSI manages your server for you), draft an email to RR support and they’ll take it from there.
