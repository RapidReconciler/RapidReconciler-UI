Status lights red

Inventory validation light is red

The Inventory Validation indicator in the page header is red. What this light checks depends on the module you’re in:

  • Inventory and Transfers — the prior period’s ending variance must equal the current period’s carry-forward, on both the GL balance and the variance carry-forward.
  • PO Receipts — only that the GL balance rolls forward correctly.

The light is binary — red or green, never amber, and no other message accompanies the colour change. The most common cause is unposted GL batches in the prior period; most cases resolve once those batches are posted in JDE. Don’t start your month-end close until both status lights are green.

What you’re seeing

  • The Inventory Validation indicator in the page header is red.
  • Common phrasing: “the red light came on.”

Try these steps in order

Step 1
Look for unposted GL batches in the prior period
When this applies: always check this first — it resolves roughly 80% of cases.

RapidReconciler tells you whether unposted GL batches from the prior period are driving the red light. You’ll find this in the variance calculation section on the main reconciliation page:

  • Change the period selection in the dropdown to the prior period.
  • If the page surfaces an unposted batch link next to the variance calculation, click that link directly — it opens the same pop-up in one step.
  • Otherwise, hover the GL Batches label to reveal a click icon, then click it to open the pop-up.
  • Look at what’s listed.

What you find decides what happens next:

  • Entries listed — those are the unposted batches. They need to be posted in JDE. The Inventory Validation light won’t go green until the next refresh cycle has run after the posting.
  • No entries — unposted batches aren’t the cause. Continue to Step 2.
Batches posted? Wait for the next refresh cycle, then re-check the indicator. No entries, or still red after the refresh? Continue to Step 2.
Step 2
Open the Inventory Validation Report
When this applies: all prior-period batches are confirmed posted but the indicator is still red.

Click the red Inventory Validation indicator. RapidReconciler exports an Excel report of the period-by-period roll-forward calculation. Save the file — you’ll read from it directly to decide what to do next.

The GLOK column tells you whether each period’s ending GL balance rolled forward into the next period. It takes one of four values:

  • Baseline — the earliest period in the data for that account; nothing to compare against.
  • Yes — the prior period’s ending balance carried forward correctly.
  • No — the carry-forward failed; this is the row you act on.
  • End — the most current period in the data for that account.

The VarOK column applies the same Yes/No to the variance carry-forward. If GLOK is No, VarOK will also be No; the GL failure cascades.

Heads-up — GLOK and VarOK can read “Yes” on every row even when the light is red. That usually means the broken period is the most current one in the report, which shows as End rather than No. In that case, focus your inspection on the End row and the row immediately before it: the carry-forward from prior to current is where the gap sits. The fixes in Step 3 still apply — treat the End row as if it were the No row.

Got the Excel? Continue to Step 3 to read the result.
Step 3
Decide based on what the report shows

Find the row(s) where GLOK = No or VarOK = No and follow the matching path:

What the row shows What to do
GLOK = No with a non-zero amount in UnpostBatch for the prior period. Post the unposted batches in JDE. The light won’t go green until the next refresh cycle has run.
GLOK = No with a non-zero amount in UnpostBatch in a period two or more periods in the past. The batches are too old for the standard fix. Email rrsupport@getgsi.com and attach the Excel from Step 2.
GLOK = No or End with UnpostBatch zero on every row. Run the JDE Account Repost program (R099102) in proof mode for the affected account. Apply any updates the proof report shows. The light won’t go green until the next refresh cycle has run.

If the row instead shows VarOK = No while GLOK = Yes, the GL is fine but the variance carry-forward broke. The fix is for an administrator to use the Reroll by Company function in Companies configuration.
