Network & Connectivity

Customer logs in but sees no data — first-pass hard reload

Symptom: Customer reports they can log in but no data is displayed. They are confirmed on the corporate network or VPN.
Likely causes
  • Browser is showing a stale cached version of the page
  • A transient connection issue caused the initial page load to fail partially
  • Browser session needs to re-authenticate against the agent
Troubleshooting steps
  1. Ask the customer to perform a hard reload of the browser using Shift + F5 (Windows) or Cmd + Shift + R (Mac).
  2. If that doesn't resolve it, ask them to fully close all browser windows and reopen, then log in again.
  3. If still no data appears, proceed to the browser local network access check.

Network & Connectivity

Customer logs in but sees no data — working remotely without VPN

Symptom: Customer reports they can log in to RapidReconciler successfully, but no data is displayed. They are working from home, a coffee shop, or an off-site location.
Likely causes
  • Customer is not connected to their corporate network
  • Customer is not connected via VPN or another approved remote-access method
  • VPN connection dropped during the session
Troubleshooting steps
  1. Ask the customer if they are working remotely (home / travel / off-site).
  2. If remote, ask them to confirm they are connected via VPN or their organization's approved remote-access method.
  3. If they are not connected, ask them to connect to VPN and try again.
  4. If they believe they are connected, ask them to disconnect and reconnect the VPN, then hard-refresh the browser.
  5. If the issue persists after confirmed VPN connection, proceed to the next customer-side check (hard reload).

User sees no companies after admin granted access (session refresh)

Symptom: Customer admin reports that a user with access to one or more companies clicks into a tab (Reconciliation, Inventory, etc.) and sees no companies in the company selector. The user is set to Active, the customer admin has confirmed in Admin → Users that the right companies and tab columns are checked under , and they have saved the user.
Likely causes
  • Cause 1's most common flavor: the user was already logged in when the customer admin updated their permissions. Their browser has the old permission set cached on the page.
  • The widget on the page the user is looking at hasn't refreshed its filter/company list since the permission change — widgets pull their list once when the page first loads.
Troubleshooting steps
  1. If the boxes are correctly ticked and saved, the user's session is the issue. Have the user click the refresh icon in the top-right corner of the widget (filter / company-selector area) on the page they're on. This forces the widget to re-read security settings from the RR app and the missing companies should appear.
  2. If the widget refresh doesn't surface the companies, have the user log out and log back in. A fresh login pulls the current security settings from the server and bypasses anything the browser had cached for the previous session.
  3. If neither the widget refresh nor a re-login works, have the user hard-refresh the browser (Ctrl+F5 on Windows / Cmd+Shift+R on Mac) on the affected page, then log in again. This clears any cached page assets that might still be holding old permissions.

Network & Connectivity

Browser blocking local network access (Edge / Chrome permission)

Symptom: Customer logs in successfully but no data displays. VPN is confirmed connected and a hard reload didn't help. Customer's browser was recently updated, or this is a new workstation.
Likely causes
  • Recent browser updates introduced a permission controlling whether sites can connect to devices on the local network
  • rapidreconciler.getgsi.com is not in the allowed list for local network access
  • A workstation policy is preventing the user from changing the setting
Troubleshooting steps
  1. Walk the customer through Edge: Settings → Privacy, search, and services → Site permissions → All permissions → Local network access.
  2. In the "Customized behaviors" section, check the "Allowed to connect to any device on your local network" list.
  3. Verify rapidreconciler.getgsi.com is present in that allowed list.
  4. If missing, click "Add site" next to that list and add rapidreconciler.getgsi.com.
  5. Chrome has an equivalent setting under its site-permissions area — wording differs slightly but the concept is the same.
  6. After adding the site, hard-reload the page and verify data appears.
  7. If the customer cannot find or change the setting on their machine, escalate to their I/T department.

Network & Connectivity

Customer logs in but sees no data — last-resort browser data clear

Symptom: Customer logs in but no data displays. VPN is confirmed, hard reload didn't help, local network access is correctly configured, and Customer I/T has confirmed no firewall changes.
Likely causes
  • Browser cache contains stale or corrupted authentication tokens
  • Cookies for the rapidreconciler.getgsi.com domain are in an invalid state
  • Service worker or local storage holds outdated state from a prior session
Troubleshooting steps
  1. Refer the customer to their I/T department for browser-specific instructions on clearing browser data.
  2. The general approach is to clear cookies and cached files for rapidreconciler.getgsi.com (not all sites), then close the browser fully and reopen.
  3. After clearing, have the customer log in again with a fresh session.
  4. If the issue still persists after all customer-side checks, the issue is on the GSI side — proceed to the agent-side scenarios in this document and check the agent connectivity from the customer's server.
  5. If exhausted all checks and no resolution, email rrsupport@getgsi.com with the customer details and the steps already attempted.

Application & Data

Inventory Validation light is red — variance / out of balance

Symptom: Customer reports a variance, "out of balance" message, or that the Inventory Validation indicator at the top of the Reconciliation page has turned red. The Variance Calculation panel typically shows a Carry Forward amount that matches the Unreconciled Variance. Customer phrasings often include "the red light came on," "this just popped up," or "we have a $X variance for [period]."
Likely causes
  • One or more GL batches in the prior period are unposted — by far the most common cause; the product itself surfaces this as the most likely scenario in the indicator's tooltip.
Troubleshooting steps
  1. Acknowledge the variance with the customer. Note the period (e.g. April 2024) and the variance amount. Ask for a screenshot of the Reconciliation page including the Variance Calculation panel if not already attached.
  2. Read the indicator’s own tooltip back to the customer. Hovering over the red Inventory Validation indicator shows a popup that says: “This indicator is red if the roll forward from the prior period fails. This is most likely due to one or more un-posted GL batches. If all batches in the prior period have been posted, please contact us at rrsupport@getgsi.com for assistance.” Many customers don’t notice this message — pointing them at it sometimes resolves the case before any further troubleshooting.
  3. Ask the customer to verify whether all GL batches in the prior period have been posted. This is a customer-side check in their JDE/ERP, not in RapidReconciler. The product’s own tooltip says this is the most likely cause and is correct in our experience.
  4. If unposted batches are found: have the customer post the batches, then refresh the Reconciliation page and confirm the Inventory Validation indicator goes green. If green, resolved — close the loop with the customer. Roughly 80% of cases end here.
  5. If the customer confirms all prior-period batches are already posted: ask them to click the red Inventory Validation indicator. This exports an Excel report of the carry-forward calculation. Have them email the Excel to rrsupport@getgsi.com (or attach it to the existing support thread).

Application & Data

System Status light is red — job crashed

Symptom: Customer reports the System Status indicator at the top of the Reconciliation page has turned red. This signals that the underlying job has crashed. Customer phrasings often include "system status is red," "the red light is on," "the job failed," or "something crashed."
Likely causes
  • Disk space issue on the application or database server — the job ran out of room while writing intermediate or output data and aborted. Common after long-running periods without housekeeping.
  • Data processing error — the job hit a record or condition it couldn't handle (bad data shape, unexpected null, type mismatch, calculation overflow, etc.) and threw before completing.
Troubleshooting steps
  1. Acknowledge the alert with the customer. Note the date/time the indicator turned red, the company/business unit they were viewing, and ask for a screenshot of the Reconciliation page header (showing the System Status indicator) if not already attached.
  2. Ask the customer to click the red System Status indicator. This exports an Excel report containing the crash details. Have them email the Excel to rrsupport@getgsi.com (or attach it to the existing support thread).
