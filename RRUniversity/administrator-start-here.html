What You Configure

Five minutes of orientation. RapidReconciler asks an administrator to manage four configuration domains, all reached from the Admin menu in the main navigation. Knowing what they are — and how they relate — makes the procedural topics much easier to navigate.

Where They Live in the Application

From any page, the four configuration areas are reached by clicking Admin in the left-rail main navigation, which expands to show the four sub-items below:

The Four Domains

DomainWhat It CoversProcedural Topic
Companies The licensed JDE companies RapidReconciler can see. Fiscal year start dates, base currency, AAI document types, and tolerance thresholds for each licensed company. Company Management + Adding a Company License
Users Who can log in. What each user can see (which companies, which modules) and do (which functions, which tabs). The boundary between an inventory accountant and an A/P clerk is drawn here. User Management
General Application-wide configuration that doesn’t belong to a specific company or user. Common units of measure for cross-item analysis, the Roll Forward report column mapping, and cardex purge controls. General
Inventory Offsets The GL accounts RapidReconciler suggests as offsets when end-users post variance journal entries. Configure these correctly and end-users get clean Excel exports they can paste into JDE without editing. Inventory Offsets

How They Relate

The four domains are mostly independent, but a few dependencies are worth knowing up front:

  • Companies must exist before users can be assigned to them. A user’s per-module company access list (Inventory, In Transit, PO Receipts) is filtered to companies that are licensed and configured. Configure companies first.
  • Offset accounts depend on companies and on the AAI document types each company uses. If a company changes its AAI document type configuration, the offset account mappings should be reviewed.
  • General settings — particularly the Common UOM list — affect what end-users see on the As-Of page. Changes here are visible to all users immediately on the next page load.

What You Do Not Configure

Worth being explicit. RapidReconciler is read-only against JD Edwards. The configuration items below live in JDE, not in RapidReconciler — an administrator should know they exist but they aren’t managed here:

  • DMAAI table 4152. Inventory accounts come from DMAAI; managed in JDE.
  • GL chart of accounts. Owned by the JDE finance configuration.
  • JDE security or row-security. RapidReconciler trusts what JDE has authorized; it does not enforce its own row-level security on JDE data.
  • The nightly import schedule. Operationally owned by the team that runs the JDE-to-RapidReconciler import jobs.
See About Administration for a fuller orientation, including how the administrator role is assigned.
Try this now

If you have admin access to RR, log in and click through the four sub-items under the Admin menu in order: Companies → Users → General → Inventory Offsets. Don’t change anything — just look at each page and notice what’s configurable on it.

The goal is recognition, not understanding. Five minutes here will make every procedural topic in this walkthrough easier to follow because you’ll have already seen the screen each topic is describing.

Your First Setup

The recommended order to configure RapidReconciler for a first deployment. Each step links into the procedural topic that explains how to do it. Following the order matters — some steps depend on earlier ones.

Recommended Sequence

#ActionWhy This Order
1 Read About Administration. Confirms how the administrator role is assigned and gives you the vocabulary for the rest of the setup.
2 Configure each licensed company. See Company Management. Fiscal year start, base currency, AAI document types, tolerance thresholds. Everything downstream filters by company, so getting this right first prevents rework later.
3 Configure General. See General. Common UOM list and Roll Forward report column mapping. These are application-wide and worth configuring once before users start running into them.
4 Configure inventory offsets. See Inventory Offsets. Required for clean variance JE exports. Without this, end-users will see “TBD” placeholders in their Excel exports.
5 Add users and assign access. See User Management. Users are added last because per-module company access depends on companies being configured first. Adding users earlier means revisiting them after company setup.
6 Walk the Pre-launch Checklist before announcing availability to end-users. Confirms each domain is set up correctly and that a representative end-user can productively work in the application before broader rollout.

If You Are Adding a Company After Initial Deployment

The sequence above describes a first deployment. Once RapidReconciler is live and you need to add a newly-licensed JDE company:

  1. Submit the company licensing request to GSI support — see Adding a Company License.
  2. Once GSI confirms the license is in place, configure the new company per Company Management.
  3. Review and update offset account mappings if the new company introduces new AAI document types.
  4. Update existing user permissions to grant per-module access to the new company where appropriate.

If You Are Inheriting an Existing Deployment

If you are taking over administration of a RapidReconciler instance that someone else configured:

  • Walk the Pre-launch Checklist as a sanity check — treat it as a configuration audit rather than a setup procedure.
  • Pay particular attention to user access lists. Stale users (departed staff still configured) are the most common finding when an administrator changes hands.
  • Verify offset account mappings against the chart of accounts. Account numbers do change, and stale offsets produce wrong entries on user JE exports.
  • Review the Common UOM list for entries that no longer reflect current item-master conventions.
When you’re done with the sequence, walk the Pre-launch Checklist to confirm the deployment is ready for end-users.
Try this now

For new deployments: identify which step from the recommended sequence you’re currently on and what’s blocking the next one. The most common blockers are waiting for company licensing from GSI or waiting for end-user lists from the customer’s HR or operations team. Knowing where you’re stuck is the first step to unstuck.

For inherited deployments: walk the Pre-launch Checklist now — not as a setup procedure but as a configuration audit. Most inherited deployments have at least one finding (usually stale users or stale offset mappings). Catching them on a quiet day is far better than discovering them at period close.

Pre-launch Checklist

What must be configured and verified before users can productively work in the application. Use this for first deployment, as a configuration audit when inheriting an instance, and as a periodic review at major milestones (new fiscal year, new licensed company, organizational changes).

Companies

  • Each licensed company is configured
  • The fiscal year start date matches the company’s fiscal calendar in JDE.
  • The base currency is correct.
  • AAI document types are populated and match the document types actually in use in JDE.
  • Tolerance thresholds are set to your organization’s materiality standard (or to the GSI default if you have not yet established one).
  • The company appears in the period selector with reasonable historical periods available.

General

  • Application-wide configuration is set
  • The Common UOM list includes the units of measure that span multiple item master records.
  • The Roll Forward report column mapping reflects the document and order types your organization actually uses.
  • The cardex deletion policy matches your organization’s data retention requirements.

Inventory Offsets

  • Offset account mappings are complete
  • Every variance source that produces a JE export has an offset account configured.
  • The configured GL accounts exist in the chart of accounts and are open for posting.
  • A test export from the Inventory Offsets widget produces an Excel file with no “TBD” entries other than expected ones.
  • If multiple companies use different offset account schemes, each company’s mappings have been reviewed individually.

Users

  • User access is appropriate
  • Every user who needs access has an account.
  • Departed staff have been removed.
  • Each user’s authorized functions and tabs match their role (inventory accountants don’t see PO Receipts admin functions; A/P clerks don’t see inventory configuration).
  • Per-module company access is set correctly for each user.
  • At least one administrator other than yourself exists, so the system isn’t single-administrator-dependent.

Operational Readiness

  • The application is ready for end-users
  • The nightly JDE-to-RapidReconciler import has run successfully at least once.
  • Both status lights on the Reconciliation page are green for the periods end-users will work with.
  • A representative end-user from each role family (inventory, A/P) has logged in, completed their Start Here or Start Here orientation, and confirmed they can see what they need to see.
  • End-users know how to contact you (or the support inbox) when something doesn’t look right.

What to Do When the Checklist Surfaces a Problem

If a company is missing or misconfigured

Resolve through Company Management (for already-licensed companies) or Adding a Company License (if the company is not yet licensed).

If offset accounts are missing or stale

Update the affected mappings in Inventory Offsets. Re-export from the Inventory Offsets widget to verify the change took effect.

If user access is wrong

Edit the affected user(s) in User Management. For departed staff, deactivate or remove rather than just clearing module access — this keeps the active user list clean.

If the nightly import has not run or produced errors

This is operational rather than configuration. Contact the team that runs the import jobs (typically IT or the JDE administration team). RapidReconciler administrators do not own the import schedule itself.

Try this now

The single highest-value pre-launch action is to log in as a representative end-user (or have one of your end-users log in on a call with you) and verify that they can see what they should see.

Specifically: do their account filters show the companies and business units they need? Do they see the modules they need (Inventory, In Transit, PO Receipts) and not the ones they don’t? If they look at a closed period, do the GL Balance values match what JDE shows for the same accounts?

Most pre-launch surprises surface in the first five minutes of a real end-user logging in. Better to find them now than at period close.

About Administration

Orientation for the administrator role within RapidReconciler — including how the role is assigned and the four areas this manual covers.

Overview

Any RapidReconciler user can be designated as an administrator by enabling the Admin checkbox within their user rights settings. When RapidReconciler is first installed, a GSI Admin account is configured as the initial administrator in order to set up the application.

GSI will also designate the first customer user as an administrator. That customer administrator then assumes responsibility for maintaining the RapidReconciler configuration going forward.

Areas of Administrator Responsibility

This training manual covers the following four areas of RapidReconciler administration. They are accessed from the Admin menu in the main navigation panel — visible only to users who have administrator rights enabled on their account.

Common Units of Measure

Common units of measure allow quantities to be restated in a single unit of measure across all items. This is useful in scenarios where multiple items in a warehouse have different primary units of measure but each has a conversion factor set up for a common unit.

Example

If hundreds of items in a particular warehouse need to be totaled by weight, and each item has a conversion factor defined for LB (pounds), adding LB as a common unit of measure allows the total weight to be reported across all items in that single unit.

Procedure

  1. Type the 2-character UOM code into the input field.
  2. Click the Add UOM button.
  3. The added UOM will be available for use after the next data refresh cycle.

Roll Forward Settings

The roll forward settings define the columns displayed between the opening and closing balance columns on the Roll Forward page within the Inventory module. Each row in the settings represents one column in the report. Units or amounts are summarized from the item ledger table F4111 based on either the order type (DCTO) or document type (DCT) field.

Use the New Setting button in the top right corner to add a new row. The maximum limit is 12 rows or "totals columns."

Each setting contains the following fields:

FieldDescription
Column TitleA user-defined title describing the type of transactions being summarized in that column.
SortThe sequence of the column in the report output. The order can be manually adjusted.
FieldDetermines whether the total is calculated using the order type (DCTO) or document type (DCT).
Codes 1 through 6The specific order types or document types to be summarized in that column. The maximum is 6 codes per row. If additional codes are required, add a new setting.

To delete a setting, click the trash can icon at the far right of the row. To edit an existing setting, click the paper/pencil icon. When editing, key the desired values into the applicable fields. For fields with drop-down arrows, select the value from the list provided. Click Save Changes when complete.

Request a License

When the customer needs to add a new JDE company to RapidReconciler, more user seats beyond the current count, or any other expansion to their existing RR entitlement, the request goes to GSI Sales. License changes are not handled by RR Support — Sales owns the entitlement, RR Support owns the running application.

What to Include in Your Email

Send a short email to gsisales@getgsi.com with the items below. Sales will reply to confirm the change is in place and to coordinate any billing follow-up.

  • Customer / instance name — so Sales can match the request to your account.
  • What you need to add — e.g., “license for JDE company 00050” or “five additional user seats”.
  • When you need it by — especially if there’s a go-live, audit, or period-close deadline.
  • Who to copy on the response — procurement, project lead, IT, the user’s manager.
Quick mailto

Email gsisales@getgsi.com with the four items above.

After Sales Confirms

Once Sales replies that the license is in place:

  • For a new company license, follow Your First Setup → If You Are Adding a Company After Initial Deployment to configure the new company in RapidReconciler.
  • For additional user seats, follow User Management to add the users and grant module / company access.

Resetting the RapidReconciler Database

A database reset wipes RR’s stored snapshots of JDE data and the corrections an administrator has layered on top. It is a heavyweight, irreversible operation. RR Support performs the reset — administrators request it, but do not run it themselves.

Before requesting

Schedule a discovery call with RR Support to discuss whether a reset is actually the right answer. The vast majority of “the numbers look wrong” situations are not solved by a reset — they’re solved by a configuration change, posting a missing batch, or a roll-forward correction. A reset should be the last option considered, not the first.

When a Reset May Be Recommended

Resets are rarely the right tool. The situations where RR Support has historically agreed a reset was appropriate include:

  • Significant chart-of-accounts restructuring in JDE that invalidates RR’s historical inventory account mappings.
  • GL class code overhaul — widespread renumbering or remapping of GL classes that breaks the reconciliation history’s account assignments.
  • Cardex baseline change — a JDE-side cardex rebuild or recosting that makes RR’s prior cardex snapshots irrelevant.

Even in these situations, RR Support will work through the impact with you before scheduling the reset. The decision is collaborative.

Important Scope

A reset is instance-wide. It affects every company configured in RapidReconciler — companies cannot be reset in isolation. If only one company has issues, a reset is almost certainly not the right path; work with RR Support on a company-scoped correction instead.

What Happens After a Reset

After RR Support completes the reset, RapidReconciler rebuilds its snapshots from JDE on the next scheduled job. Expect:

  • Reconciliation history (period-by-period RR snapshots) populates forward from the reset date, not backfilled retroactively.
  • Administrator-applied corrections (inventory offsets, suspension and exclusion rules) need to be re-evaluated — some carry forward, some don’t, depending on what the reset cleared. RR Support will confirm specifics for your case.
  • End-users may see “no data” for prior periods until the next nightly job completes.
How to start the conversation

Email rrsupport@getgsi.com with a description of the symptoms you’re seeing and what changed in JDE. Ask for a 30-minute call to discuss whether a reset is appropriate. Do not request the reset directly — the call is the gate.

Vocabulary

RapidReconciler-specific terms relevant to the administrator role. Each entry is a one-line definition plus a link to the topic that explains it in depth. Skim once for awareness; come back when a term comes up in a configuration screen and you want to remember what it meant.

AAI document type
Per-company configuration that tells RapidReconciler which JDE document types apply to inventory transactions for that company. Drives the AAI-based account derivation in the Variance Calculation. Company Management.
Authorized functions
Per-user permissions for what actions a user can take inside a module (e.g., apply a suspension, post a JE export, edit a note). Configured in User Management alongside authorized tabs.
Authorized tabs
Per-user permissions for which views (tabs) a user sees inside a module. Used to hide irrelevant pages from users whose role doesn’t need them — e.g., an A/P clerk doesn’t need to see the Inventory Reconciliation page.
Common UOM
Application-wide list of units of measure that span multiple item master records, used to restate quantities across items for analysis on the As-Of page. Configured in General.
Company license
The mechanism that grants RapidReconciler access to a JDE company’s data. Licensed at the GSI level; configured at the customer level. Adding a previously-unlicensed company requires a request to GSI support. Adding a Company License.
Fiscal year start date
Per-company configuration that tells RapidReconciler when each company’s fiscal year begins. Drives period boundaries throughout the application. Must match what JDE has for the same company.
Offset account
The GL account configured to be used as the offset side of a variance journal entry. Configured per company per variance source. When end-users export from the Inventory Offsets widget, the configured offset is filled in automatically; missing offsets show as “TBD”. Inventory Offsets.
Per-module company access
The list of companies a user can see inside a specific module (Inventory, In Transit, PO Receipts). A user can have access to different sets of companies in different modules. Configured per-user in User Management.
Roll Forward report column mapping
Configuration that maps JDE document and order types to columns on the Roll Forward report. If end-users see a non-zero value in the Variance column at the right, a document or order type is missing from the mapping. General.
Suggested Entry popup
The end-user-facing popup that uses configured offset accounts to pre-build a variance JE for the user to review and post. Quality of this popup is directly proportional to the quality of offset account configuration.
Tolerance threshold
Per-company materiality threshold used to filter the Transactions page so end-users see only meaningful variances rather than sub-unit rounding noise. Differences below the threshold are excluded by default but still roll up into the Transactions line of the headline variance.
Tolerance Adjust line
An informational line on the Offset Account export representing the aggregate of sub-tolerance differences. Shown to end-users so they know it exists; they replace it with the correct GL account before pasting into JDE.
