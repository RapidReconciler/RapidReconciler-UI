JD Edwards Order & Work Order Status Codes — Reference

Three separate UDC tables drive every status decision in JDE: UDC 00/SS for work orders, UDC 40/AT for sales orders, and UDC 40/AT (PO variant) for purchase orders. Status codes are customer-defined per shop's Order Activity Rules, but the canonical JDE-shipped values are highly standardized. This page is the reference for both.

Overview

Every active order or work order in JDE carries two status values: last status (what it last completed) and next status (the next step allowed). The status codes themselves come from three different UDC tables depending on the order type:

Order typeUDC tableRangeHeader tableDetail tableLast-status fieldNext-status field
Work Order00/SS10–99F4801n/aWrSttsNxtSts
Sales Order40/AT500–999F4201F4211LSTSNXTR
Purchase Order40/AT200–999F4301F4311LSTSNXTR

Customer-defined values are common. All three UDC tables ship with the canonical values below, but a shop's Order Activity Rules can insert intermediate statuses (a "550 — Quality Hold" between Pick Slip and Pick Confirmed, a "65 — Material Issued" between Released and In Process, etc.) or relabel the canonical ones. When troubleshooting, always confirm the shop's actual rules in P98012 (work orders), P40203 (sales), or P40204 (purchasing) before assuming the canonical meaning.

Work Order Statuses (UDC 00/SS)

Stored on F4801 (Work Order Header). Two-digit values, typical range 10–99. Configured via P98012 Work Order Activity Rules.

StatusDescriptionMeaning
10EnteredWO created but not approved
20ApprovedReady for planning
30PlannedMaterial / capacity planning complete
40Firm PlannedFrozen for scheduling
50ReleasedAvailable to shop floor — controls when inventory can be issued
60Parts List / Routing CompletePick lists and routing ready
70In ProcessProduction started
80CompletedQuantity completed — controls completions to inventory
90ClosedAccounting complete — prevents further transactions; R30837 (WIP Revaluation) will not revalue
99CancelledWO voided / cancelled

More detailed implementations may also use intermediate values: 15 (Awaiting Approval), 25 (MRP Planned), 45 (Scheduled), 55 (Picked), 65 (Started), 85 (Production Complete), 95 (Financial Close).

R30837 (WIP Revaluation) skips Closed WOs. The Closed status (typically 90 but customer-defined) is the cutoff point for several manufacturing-accounting programs. Once a WO is closed, R30837 will not revalue it even if re-run — a standard cost change after that point requires a manual journal entry to clear. Confirm your shop's actual Closed value in P98012 before assuming the canonical 90.

Order Activity Rules

Each order type has its own program for configuring allowed status transitions. The activity rule defines, for a given order type + line type combination, the sequence of statuses a line moves through and which statuses are skippable, terminal, or editable.

Why Status Codes Matter for Reconciliation

Several reconciliation patterns key off status values directly. When a Transaction Detail report shows an unexpected variance, the line's NxtSts on the source order is often the first thing to check:

  • Standard Cost Change orphan on a closed WO (IC docs). When R30822 (Frozen Cost Update) changes the standard cost after a WO has reached its Closed status (typically 90), R30837 (WIP Revaluation) will not revalue it. The F4111 cardex captures a "Standard Cost Change" row but no F0911 counterpart fires. Resolution is a manual JE crediting AAI 3240 / 3260 — not a re-run of R30837 (which would do nothing on a closed WO).
  • Post-ship-confirm order edits (sales orders). If a line's NxtSts is at or past 540 (Shipment Confirmation in the canonical SO range) and the F4211 line qty is larger than what F4111 captured, the order was edited after ship-confirm. The cardex is locked at the confirmed qty; the post-edit qty flows through Sales Update (R42800) into the GL. The variance equals (new qty − confirmed qty) × unit cost. Resolution is process / access-control — restrict P4210 / P42101 edits past status 540.
  • Receipt vs voucher stage mismatches on PO docs. An OV (P4312 receipt) advances a PO line through the receipt statuses (typically 400–420). A PV (P4314 voucher match) advances it further to 430. Reconciling against the wrong stage of the cycle (e.g. expecting voucher-side accounting at a status that's only past receipt) is a common source of "missing GL" diagnoses that are actually "GL hasn't fired yet."

If your shop uses non-standard Closed values (e.g. WOs close at 95 instead of 90, or POs close at 980 instead of 999), the analyzer's Pattern 5.6 / 5.13 / 5.16 closed-status branches need to be configured for those values. Otherwise the analyzer will recommend the "still open, R30837 re-run viable" path on an effectively-closed WO, or fail to detect a post-confirm sales-order edit. See your shop's Order Activity Rules to confirm the actual closed value used.
